## 第1章 概述

### 1.1 总览

Adapter Core JE 是基于 **Fabric 模组 + Java 21** 的 Minecraft Java 版接入核心，共 15 个版本迭代，实现 26 个工具，由 2 名开发者并行推进。

**研发阶段**：

```
阶段一：基础建设（V1-V3）→ Fabric 模组骨架 + TCP 通信 + 工具注册（3 周）
   ↓
阶段二：执行核心（V4-V6）→ 寻路/移动 + 背包/方块 + 生存（5 周）
   ↓
阶段三：工具矩阵（V7-V9）→ 生物/战斗 + 感知 + 对话（3 周）
   ↓
阶段四：智能增强（V10-V12）→ 假人管理 + 数据持久化 + 配置接入（4 周）
   ↓
阶段五：集成发布（V13-V15）→ 任务对接 + 性能优化 + 打包文档（5 周）

总时长：约 20 周（5 个月），与 BE 版本节奏同步
```

### 1.2 版本节奏

| 阶段 | 版本 | 周期 | 核心交付 |
|:----:|:----:|:----:|----------|
| 基础建设 | V1-V3 | 各 1 周 | 模组骨架 → TCP 通信 → 工具注册闭环 |
| 执行核心 | V4-V6 | 各 1-2 周 | 寻路移动 → 背包操作 → 生存+方块 |
| 工具矩阵 | V7-V9 | 各 1 周 | 战斗+生物 → 感知 → 对话 |
| 智能增强 | V10-V12 | 各 1-2 周 | 假人管理 → 数据持久化 → 配置接入 |
| 集成发布 | V13-V15 | 各 1-2 周 | 任务对接 → 性能优化 → 打包发布 |

### 1.3 集成节点

| 版本 | 集成内容 | 参与模块 |
|:----:|----------|----------|
| V3 | TCP 通信 + 工具注册闭环 | AC + JE |
| V6 | 执行AI 主体 + LLM 调度 | AC + JE |
| V9 | 全部 26 工具就绪 | AC + JE |
| V13 | 全部功能模块完成 | AC + JE |
| V15 | 最终集成发布 | AC + BE + JE 三模块 |

### 2.2 公共模块分工

### 2.3 协作规则

1. **每日同步**：每日站会 15min，沟通进度和阻塞项
2. **代码审查**：每个版本末交叉 review，重点审查工具实现正确性和性能
3. **接口对齐**：V3/V6/V9/V13 集成节点前，与 BE 核对工具接口定义是否一致
4. **版本 Tag**：每个版本完成时打 Git Tag `je-v<version>`（如 `je-v1`、`je-v2`）

## 第5章 与 BE 同步策略

### 5.1 接口定义同步

| 同步内容 | 同步时机 | 同步方式 |
|----------|:--------:|----------|
| 工具名称和参数定义 | V3 工具注册前 | 双方核对工具清单 JSON |
| JSON-RPC 消息格式 | V2 TCP 通信前 | 基于 Shared 库的 D1-D4 定义 |
| 状态上报数据字段 | V3 状态上报前 | 对齐 status_report 消息格式 |
| 事件通知类型 | V14 事件完善前 | 对齐 event_type 枚举 |
| 数据库表结构 | V11 数据持久化前 | 对齐 SQLite 表 DDL |
| 错误码定义 | V2 握手协议 | 对齐 ErrorCode 枚举 |

### 5.2 代码同步策略

1. **不共享实现代码**：BE/JE 代码完全独立，不共用代码库
2. **共享接口定义**：工具名称、参数 Schema、消息格式在 Shared 库（TypeScript）中定义，JE 参照定义用 Java 实现
3. **同步集成测试**：V3/V6/V9/V13/V15 集成节点，BE 和 JE 分别与 Agent Core 联调

### 5.3 版本同步

| 版本 | BE 里程碑 | JE 里程碑 | 对齐检查点 |
|:----:|:---------:|:---------:|------------|
| V3 | 工具注册 + 状态上报 | 工具注册 + 状态上报 | 工具清单 JSON Schema 一致 |
| V6 | 执行AI 主体完成 | 执行AI 主体完成 | 工具执行结果格式一致 |
| V9 | 全部 26 工具就绪 | 全部 26 工具就绪 | 逐个工具参数/输出对齐 |
| V13 | 任务工具对接 | 任务工具对接 | 任务系统工具调用一致 |
| V15 | E2E 测试 + 打包 | E2E 测试 + 打包 | 全链路测试通过 |

### 5.4 差异处理

| 差异项 | 处理方式 |
|--------|----------|
| Java 类型体系 vs TypeScript | JE 用注解 + 反射生成 Schema，保证输出 JSON 格式一致 |
| 游戏 API 不同 | 各自实现相同功能，输出格式一致 |
| 序列化方式不同 | JE 用 Gson，BE 用原生 JSON，消息格式一致 |
| 假人实现不同 | JE 客户端实体 vs BE 服务器实体，功能对等但实现不同 |
