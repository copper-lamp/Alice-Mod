```
AliceModAdapter.onInitialize()
    → ToolScanner.scan("io.alice.mod.adapter.tool.module")
```

### 1.3 需完善的关键缺口

1. **ToolResult 格式**：当前 `ToolResult` 仅含 `{success, message, data}`，协议要求返回 `ResultEnvelope` 含 `{success, error:{code,message}, data, meta:{duration,cost}}` — 需增强
2. **AI 执行层缺失**：每个工具方法内部需要调用 Minecraft API 执行真实操作，当前完全不存在
3. **假人基础设施缺失**：工具操作需要假人（Bot）实例作为执行主体，当前 `bot` 包为空

## 第2章 架构设计

### 2.1 三层架构

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 1: 工具契约层 (tool/)                                │
│  职责：定义工具接口、注册表、结果信封、注解                   │
│  文件：AliceTool.java, ToolResult.java, ToolRegistry.java   │
│  注解：@ToolModule, @ToolMethod, @ToolParam                 │
├─────────────────────────────────────────────────────────────┤
│  Layer 2: 工具实现层 (tool/*/)                              │
│  职责：每个 @ToolModule 类实现一类工具的 @ToolMethod         │
│  原则：薄层代理，只做参数校验 + 委托 AI 层                   │
│  模式：@ToolMethod 方法接收 Map args → 委托 ai.* 层 → 返回 ToolResult │
├─────────────────────────────────────────────────────────────┤
│  Layer 3: 执行AI 层 (ai/*/)                                 │
│  职责：实际的 Minecraft 游戏操作，直接调用 MC API            │
│  原则：厚层实现，含所有游戏交互逻辑                           │
│  不依赖：不依赖 tool 层的任何注解，纯逻辑层                   │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 工具调用链路

```
Agent Core
    │ tool_call("move_to", {x, z})
    ▼
TCP 客户端 → ToolRegistry.resolve("move_to") → AliceTool.invoke({x, z})
    │
    ▼ (注解工具方法)
MoveToTool.teleport(Map args)  ← @ToolMethod
    │ 参数校验 + 目标类型推断（坐标/实体/升降）
    ▼
ai.movement.MovementController.moveTo(x, y, z, allowBreak)
    │ 寻路 + 路径跟随 + 避障
    ▼
Carpet Mod EntityPlayerActionPack / ServerPlayer 原生 API
    │
    ▼
ToolResult {success, data, meta} → JSON → TCP → Agent Core
```

### 2.3 与 numen 架构映射

| Alice Mod 概念 | numen-api 对应 | 复用策略 |
|---------------|---------------|----------|
| `AliceTool` 接口 | `NumenTool` 接口 | 参考接口设计模式，但 `invoke` 保持 `Map args` → `ToolResult` 同步模式 |
| `ToolResult` | `TaskResult` | 参考工厂方法命名（ok/fail），增加 `meta` 字段 |
| `ToolRegistry` | `ToolRegistry` | 当前实现已对齐，无需改动 |
| `@ToolModule` 类 | `minecraft-numen` 工具类 | 参考其工具拆分粒度（每类一个文件） |
| `ai.movement.MovementController` | `MoveToTool` 内部逻辑 | 参考其 ServerPlayer API 调用模式 |
| `bot.BotManager` | `CompanionFactory` | V10 实现，当前先通过 Carpet API 获取假人 |

## 第3章 代码复用策略（针对 libs/numen）

### 3.1 可直接参考的模式

| 模式 | numen 位置 | 参考方式 |
|------|-----------|----------|
| **ServerPlayer API 调用** | `minecraft-numen` 各工具 `runOnServer()` | 参考 MC API 调用模式（`PlayerList`, `ServerLevel`, `BlockPos` 等） |
| **工具参数解析** | `ToolArgs.requireInt()`, `optionalPos()` | 参考参数提取的异常处理模式 |
| **安全装备切换** | `NumenPlayer.holdInHand(int slot)` | 参考槽位边界检查 + 背包交换逻辑 |
| **物品查找** | `NumenPlayer.ensureInInventory(Predicate)` | 参考背包扫描的谓词模式 |
| **工具命名约定** | `MoveToTool`, `EatItemTool` 等 | 参考 `{Verb}{Target}Tool` 命名风格 |

### 3.2 不可直接复用的部分

| 部分 | 原因 | Alice Mod 做法 |
|------|------|---------------|
| `NumenTool.invoke(ToolCall)` | numen 使用异步 `complete()`，Alice 用同步 `invoke()` | 保持当前同步模式 |
| `NumenActuator` acquire/invoke/release | 面向 LLM 在客户端的架构 | Agent Core 天然是外部大脑 |
| `EntityAgentLoop` | LLM 调度循环，Alice 无此概念 | 不需要，Agent Core 负责调度 |
| `CompanionFactory.placeNewPlayer` | V10 假人管理时才需要 | 当前用 Carpet API `EntityPlayerMPFake` |

### 3.3 适配注意事项（来自 JE-05 §7.2）

1. **`PlayerList.load()` 版本差异**：1.21.4 返回 `CompoundTag`（可为 null），1.21.5+ 返回 `Optional<CompoundTag>` — 当前项目用 1.21.4
2. **死亡处理**：假人在 `tick()` 中检测死亡并触发 `onDeath()` — Alice Mod 同样需要
3. **FakeConnection**：不需要 NeoForge 兼容，仅需 Fabric

## 第4章 ToolResult 增强

### 4.1 当前状态 vs 目标格式

```java
// 当前 ToolResult (V3 已实现)
public record ToolResult(boolean success, String message, Map<String, Object> data) {
    public static ToolResult ok(String message) { ... }
    public static ToolResult ok(String message, Map<String, Object> data) { ... }
    public static ToolResult fail(String message) { ... }
    public String toJson();
}
```

```json
// 当前输出格式
{
    "success": true,
    "message": "移动成功",
    "data": { "x": 128, "y": 64, "z": -256 }
}
```

```json
// 目标格式（符合 00-工具协议规范.md §3 ResultEnvelope）
{
    "success": true,
    "data": { "x": 128.5, "y": 64.0, "z": -256.2 },
    "meta": { "duration": 4500, "cost": { "hunger": 0.5 } }
}
```

### 4.2 改造方案

保持 `ToolResult` 的 record 签名不变（向后兼容），通过工厂方法增强：

```
ToolResult.ok(message)                    → success=true, data={message}, meta={duration}
ToolResult.ok(message, data)              → success=true, data+message, meta={duration}
ToolResult.ok(message, data, meta)        → success=true, data+message, meta
ToolResult.fail(code, message)            → success=false, error:{code,message}, meta={duration}
ToolResult.fail(code, message, details)   → success=false, error:{code,message,details}, meta={duration}
```

序列化时输出匹配 `ResultEnvelope` 结构：
- `data` 字段内嵌 `message`（LLM 可读的自然语言）
- 新增 `meta.duration`（自动计时）
- 新增 `meta.cost`（资源消耗）
- 失败时输出 `error` 对象

### 6.2 控制器公共依赖

所有控制器通过 `AliceBotManager`（暂用 Carpet API 直接调用）获取当前 Bot 实例：

```java
// 临时方案：所有操作走服务端玩家（V10 之前通过 Carpet API）
// MinecraftServer 通过 Fabric API 的 ServerLifecycleEvents 获取
public class BotAccess {
    private static MinecraftServer server;
    
    public static void init() {
        ServerLifecycleEvents.SERVER_STARTED.register(s -> server = s);
    }
    
    // 获取第一个假人（单 bot 模式）
    public static ServerPlayer getBot() {
        // V10 前：使用 Carpet 的 EntityPlayerMPFake 或找名字含 "alice" 的玩家
        return server.getPlayerList().getPlayers().stream()
            .filter(p -> p.getScoreboardName().startsWith("alice_"))
            .findFirst()
            .orElse(null);
    }
}
```

> **V10 适配说明**：V10 实现 `BotManager` 后，所有控制器改为通过 `BotManager.getBot()` 获取，接口不变。

## 第8章 开发优先级与建议

### 8.1 阶段划分（按依赖关系）

```
阶段一：基础设施增强（2 天）
  ├── ToolResult 增强（meta + error 支持）
  ├── BotAccess 基础（获取 ServerPlayer）
  └── 2-3 个最简 AI 控制器（PerceptionController、ChatController）

阶段二：感知 + 对话（3 天）—— 无寻路依赖，可独立开发
  ├── PerceptionTools + PerceptionController
  ├── ChatTools + ChatController
  └── 验证完整调用链路（TCP → 工具注册 → 工具调用 → ToolResult）

阶段三：背包 + 方块（4 天）—— 需 ScreenHandler 交互
  ├── InventoryTools + InventoryController + BlockController
  ├── BlockTools（mine/place/use）
  └── 验证容器交互和方块放置

阶段四：移动 + 寻路（5 天）—— 最复杂
  ├── AStarPathfinder + NavMeshBuilder
  ├── MovementController
  ├── MoveToTools
  └── ride/dismount

阶段五：生物 + 生存（3 天）—— 依赖移动和背包
  ├── EntityInteractionTools + EntityInteractionController + CombatController
  ├── SurvivalTools + SurvivalController
  └── area_operation 的 vein 模式（依赖 VeinScanner）

总计：约 17 天（含联调）
```

### 8.3 与 V10 假人管理的衔接

当前方案使用临时 `BotAccess.getBot()` 获取假人实例（通过 Carpet Mod API）。V10 实施后，所有 `ai/*/Controller` 中的 `getBot()` 调用替换为 `BotManager.getBot()`。

- 工具系统开发和假人管理开发可以**完全并行**
- 工具系统无需等待 V10 完成即可启动
- V10 完成后只需修改 AI 控制器的 getBot 方式，工具层 0 改动

### 9.2 AI 控制器标准模板

```java
package io.alice.mod.adapter.ai.perception;

import net.minecraft.server.level.ServerLevel;
import net.minecraft.server.level.ServerPlayer;
import net.minecraft.world.entity.LivingEntity;
import net.minecraft.world.entity.Mob;
import net.minecraft.world.entity.player.Player;
import net.minecraft.world.phys.AABB;
import net.minecraft.world.phys.Vec3;

import java.util.*;

public class PerceptionController {

    public static ScanResult scanNearby(int radius, Map<String, Object> filter) {
        ServerPlayer bot = BotAccess.getBot();
        ServerLevel level = bot.serverLevel();
        Vec3 pos = bot.position();
        AABB area = AABB.ofSize(pos, radius * 2, radius * 2, radius * 2);

        List<EntityInfo> entities = new ArrayList<>();
        boolean filterHostile = getBool(filter, "hostile");
        boolean filterPassive = getBool(filter, "passive");
        boolean filterPlayer = getBool(filter, "player");
        boolean filterItem = getBool(filter, "item");
        boolean noEntityFilter = !filterHostile && !filterPassive && !filterPlayer && !filterItem;

        for (var entity : level.getEntitiesOfClass(LivingEntity.class, area)) {
            if (!noEntityFilter) {
                if (filterHostile && !isHostile(entity)) continue;
                if (filterPassive && isHostile(entity)) continue;
                if (filterPlayer && !(entity instanceof Player)) continue;
            }
            entities.add(new EntityInfo(
                entity.getStringUUID(),
                entity.getType().getDescriptionId(),
                entity.getName().getString(),
                entity.distanceTo(bot),
                entity.position(),
                (float) entity.getHealth(),
                (float) entity.getMaxHealth(),
                isHostile(entity),
                entity instanceof Player,
                entity instanceof net.minecraft.world.entity.item.ItemEntity
            ));
        }

        String message = formatNearbyMessage(entities, radius);
        Map<String, Object> data = Map.of(
            "entities", entities,
            "total", entities.size()
        );

        return new ScanResult(message, data);
    }

    public record ScanResult(String message, Map<String, Object> data) {}

    public record EntityInfo(
        String id, String type, String name,
        double distance, Vec3 position,
        float health, float maxHealth,
        boolean hostile, boolean isPlayer, boolean isItem
    ) {}

    private static boolean isHostile(LivingEntity entity) {
        return entity instanceof Mob mob && mob.isAggressive();
    }

    private static boolean getBool(Map<String, Object> map, String key) {
        return map != null && Boolean.TRUE.equals(map.get(key));
    }

    private static String formatNearbyMessage(List<EntityInfo> entities, int radius) {
        if (entities.isEmpty()) return "附近没有实体";
        StringBuilder sb = new StringBuilder();
        sb.append("附近环境 (").append(radius).append("格内, 共").append(entities.size()).append("个):\n");
        for (var e : entities) {
            sb.append("  - ").append(e.name());
            if (e.hostile()) sb.append(" (敌对)");
            if (e.isPlayer()) sb.append(" (玩家)");
            sb.append(" ").append(String.format("%.1f", e.distance())).append("格");
            sb.append(" 血量 ").append((int)e.health()).append("/").append((int)e.maxHealth());
            sb.append("\n");
        }
        return sb.toString();
    }
}
```

### 9.3 move_to 参数推断模式

```java
// MoveToTools 中的核心推断逻辑
@ToolMethod(name = "move_to", description = "移动到目标位置")
public ToolResult moveTo(..., Map<String, Object> allArgs) {
    Double x = (Double) allArgs.get("x");
    Double y = (Double) allArgs.get("y");
    Double z = (Double) allArgs.get("z");
    String entity = (String) allArgs.get("entity");
    boolean allowBreak = Boolean.TRUE.equals(allArgs.get("break"));
    int distance = allArgs.containsKey("distance") ? ((Number)allArgs.get("distance")).intValue() : 2;

    if (entity != null) {
        // 追踪实体
        return movementController.followEntity(entity, distance, allowBreak);
    } else if (x != null && z != null) {
        // 坐标目标
        return movementController.moveTo(x, y, z, allowBreak);
    } else if (y != null && x == null && z == null) {
        // 升降高度
        return movementController.moveToHeight(y, allowBreak);
    } else {
        return ToolResult.fail("INVALID_PARAMS", "需要提供 x+z（去位置）、x+y+z（精确方块）、entity（实体）或 y（升降高度）");
    }
}
```
