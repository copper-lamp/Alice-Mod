## 第1章 概述

### 1.1 背景

Alice Mod JE 目前实现为 `ModInitializer`（Fabric 通用模组入口），仅在 Minecraft 客户端集成服务器（单人模式）下工作。但实际使用场景涉及多种部署模式：

| 场景 | 描述 | 主机行为 | 客场行为 |
|------|------|:--------:|:--------:|
| 单人游戏 | 客户端内置服务器，加载单个存档 | ✅ 完整运行 | — |
| 客户端切存档 | 单人模式退出世界→加载另一个存档 | ✅ WorldContext 切换 | — |
| 连接远程服务器 | 客户端连接多人游戏服务器 | ✅ 服务端运行（如有 Mod） | ❌ 客户端自动停用 |
| 纯服务端 | 无 GUI 的 Dedicated Server | ✅ 完整运行 | — |
| 局域网开放 | 主机开放到 LAN | ✅ 主机运行 | ❌ 游客自动停用 |

本档设计 Alice Mod JE 在上述所有场景下的运行架构，重点是：**运行环境感知切换** 和 **多存档下的状态隔离与生命周期管理**。

### 1.2 术语表

| 术语 | 说明 |
|------|------|
| **WorldContext** | 世界上下文。每个 Minecraft 世界（存档/服务器）对应一个上下文实例 |
| **世界切换** | 客户端从当前世界退出并加载另一个世界（或连接另一台服务器） |
| **ServerLifecycle** | Minecraft Server 的生命周期事件（started / stopping / stopped） |
| **Integrated Server** | 客户端内置的本地服务器（单人模式） |
| **Dedicated Server** | 独立的 Minecraft 服务端程序 |

### 1.3 Fabric 环境模型

```
┌──────────────────────────────────────────────┐
│              Fabric Loader 启动                │
│                                                │
│  ModInitializer.onInitialize()                 │
│    ├── 通用初始化（工具扫描、API 注册等）            │
│    └── 注册 ServerLifecycleEvents 监听          │
│                                                │
│  ClientModInitializer（如有）                    │
│    └── 客户端专属初始化（渲染、按键等）              │
│                                                │
│  DedicatedServerModInitializer（如有）           │
│    └── 服务端专属初始化                           │
└──────────────────────────────────────────────┘
         │
         ▼ (用户选择世界/连接服务器)
┌──────────────────────────────────────────────┐
│          Minecraft Server 启动（每个世界）        │
│                                                │
│  ServerLifecycleEvents.SERVER_STARTING          │
│  ServerLifecycleEvents.SERVER_STARTED           │
│     └── 当前世界上下文激活                        │
│                                                │
│  ServerLifecycleEvents.SERVER_STOPPING          │
│  ServerLifecycleEvents.SERVER_STOPPED           │
│     └── 当前世界上下文清理                        │
└──────────────────────────────────────────────┘
```

## 第2章 运行环境感知

### 2.1 Fabric 入口点分裂

目前只实现了 `ModInitializer`。为实现完整的客户端/服务端支持，需要分裂为三个入口点：

```java
// ── 模组主入口（通用初始化） ──
public class AliceModAdapter implements ModInitializer {
    @Override
    public void onInitialize() {
        // 只做模组级初始化（一次性的）：
        // 1. 工具扫描与注册
        // 2. Fabric API 事件注册
        // 3. 配置加载
        // 4. 插件发现
        // ❌ 不启动 TCP
        // ❌ 不初始化 BotManager
    }
}

// ── 服务端专属入口（Dedicated Server 或 Integrated Server 都会调用） ──
public class AliceModServer implements DedicatedServerModInitializer {
    @Override
    public void onInitializeServer() {
        // 服务端初始化：
        // 1. 注册 ServerLifecycleEvents
        // 2. 初始化 WorldContextManager
        // 3. 启动 TCP 连接（当服务器就绪后）
    }
}

// ── 客户端专属入口（仅在客户端调用） ──
public class AliceModClient implements ClientModInitializer {
    @Override
    public void onInitializeClient() {
        // 客户端初始化：
        // 1. 注册 ClientTickEvents
        // 2. 注册世界切换监听（进入/离开世界）
        // 3. GUI/渲染相关（如有）
    }
}
```

### 2.2 fabric.mod.json 配置

```json
{
  "entrypoints": {
    "main": [
      "io.alice.mod.adapter.AliceModAdapter"
    ],
    "server": [
      "io.alice.mod.adapter.AliceModServer"
    ],
    "client": [
      "io.alice.mod.adapter.AliceModClient"
    ]
  },
  "environment": "*"
}
```

**生命周期对比：**

```
单人游戏（Integrated Server）
  ModInitializer.onInitialize()
  ClientModInitializer.onInitializeClient()
  DedicatedServerModInitializer.onInitializeServer()  ← 集成服务器也会触发此入口
  ServerLifecycleEvents.SERVER_STARTED
  ← 玩家游戏 →
  ServerLifecycleEvents.SERVER_STOPPED

多人游戏（客户端连接远程服务器）
  ModInitializer.onInitialize()
  ClientModInitializer.onInitializeClient()
  ← 连接远程服务器 →
  ← 工具操作在客户端侧受限 →
  ← 服务端侧的 Alice Mod 通过 Server 入口初始化 →

专用服务器（Dedicated Server）
  ModInitializer.onInitialize()
  DedicatedServerModInitializer.onInitializeServer()
  ServerLifecycleEvents.SERVER_STARTED
  ← 服务器运行 →
  ServerLifecycleEvents.SERVER_STOPPED
```

### 2.3 仅主机模式（Host-Only）

Alice Mod 的设计原则是：**只有主机运行 Alice Mod，客场自动停用**。

#### 2.3.1 检测逻辑

核心检测规则非常简单——是否有本地 MinecraftServer 实例：

```
isHost = (MinecraftServer 本地实例 != null)
```

Fabric 场景下，这个判断等价于：

| 场景 | 是否有本地 Server | 是否为主机 | Alice Mod 行为 |
|------|:-----------------:|:----------:|:--------------:|
| 单人游戏（加载存档） | ✅ 集成服务器 | ✅ 是主机 | ✅ **激活** |
| 局域网开放者 | ✅ 集成服务器 | ✅ 是主机 | ✅ **激活** |
| 局域网游客 | ❌ 无本地 Server | ❌ 是客场 | ❌ **停用** |
| 连接远程服务器 | ❌ 无本地 Server | ❌ 是客场 | ❌ **停用** |
| 专用服务器 | ✅ DedicatedServer | ✅ 是主机 | ✅ **激活** |

#### 2.3.2 客户端客场自动停用

当玩家加入别人的世界时（局域网游客或连接远程服务器），客户端侧不会有 `SERVER_STARTED` 事件触发。因此 `WorldContextManager.activate()` 永远不会被调用，整个 Alice Mod 在客户端侧保持休眠状态：

```
客户端主菜单（无世界上下文）
    ↓ 连接远程服务器
客户端侧：ServerLifecycleEvents.SERVER_STARTED 不会触发
    → WorldContextManager.activate() 不会被调用
    → TcpClient 不启动
    → BotManager 不初始化
    → 工具注册表存在但不活跃
    → 整个 Mod 在客户端侧零开销

服务端侧（如果安装了 Alice Mod）：
    → 独立激活 WorldContext，正常运作
```

**为什么不需要额外检测？**
- Fabric 的 `ServerLifecycleEvents.SERVER_STARTED` 只在**本地有 Server 进程**时触发
- 作为客户端连接远程服务器时，本地没有 Server 进程，该事件根本不会发生
- 因此 `WorldContextManager.activate()` 天然只在主机场景被调用
- 不需要额外的配置开关或命令行参数

#### 2.3.3 客户端在客场的作用

作为客场的客户端，Alice Mod 仍然处于已加载状态（`ModInitializer` 已执行），但没有激活的世界上下文。此时：

- **工具注册表**：在内存中存在，但不活跃（无 TCP 连接，无世界上下文）
- **附属模组插件**：已加载，但 `registerTools()` 中的 `AliceServiceAccess` 的 `botService()` 等方法会返回空结果
- **配置文件**：可读取，不做修改
- **事件监听**：已注册但不会收到世界事件

> 这意味着客场玩家的 Alice Mod 不会影响游戏体验——不占 CPU（无 tick）、不占网络（无 TCP）、不影响存档。

#### 2.3.4 服务端部署场景

```
远程服务器（play.example.com）
├── 服务端安装了 Alice Mod
│   ├── DedicatedServerModInitializer → 注册 ServerLifecycleEvents
│   ├── WorldContextManager.activate() ✅ → 完整运行
│   ├── TCP → Agent Core
│   └── 所有假人在服务端生存
│
└── 玩家 A（客户端安装 Alice Mod）      玩家 B（客户端未安装）
    └── 客户端 Alice Mod 自动停用        └── 无影响
```

如果远程服务器**没有**安装 Alice Mod，则没有任何一方激活 WorldContext，完全透明。

### 2.4 核心变化：从"模组级"到"世界级"

当前设计的一个重要问题是：**所有状态都是模组级（静态单例）的**。

| 组件 | 当前范围 | 目标范围 |
|------|:--------:|:--------:|
| `TcpClient` | 模组级单例 | 世界级实例 |
| `BotManager` | 静态单例 | 世界级实例 |
| `BotRepository`（SavedData） | 世界级（已正确） | 世界级（不变） |
| `StatusCollector` | 模组级单例 | 世界级实例 |
| `EventDispatcher` | 模组级单例 | 世界级实例 |
| `instance_id` | 模组级 UUID | 世界级 UUID |
| `ToolRegistry` | 模组级单例 | 模组级单例（工具集不变） |

## 第3章 世界上下文（WorldContext）

### 3.1 概念

`WorldContext` 是 Alice Mod 在**一个世界会话**内的运行上下文。每个 Minecraft 世界（存档）对应一个上下文实例，包含该世界专属的组件实例。

```
WorldContext ── 每个世界一个
├── contextId: String        ← 世界唯一标识
├── identity: WorldIdentity  ← 实例身份（instanceId + authToken）
├── tcpClient: TcpClient     ← 专属 TCP 连接
├── botManager: BotManager   ← 专属假人管理器
├── statusCollector          ← 专属状态采集器
└── eventDispatcher          ← 专属事件分发器
```

### 3.2 WorldIdentity

每个世界有自己的实例身份，存储在世界的配置目录中：

```java
public record WorldIdentity(
    String instanceId,    // 世界级唯一 ID（第一次加载该世界时生成）
    String authToken,     // 认证令牌
    String worldName      // 世界名（存档文件夹名或服务器地址）
) {
    // instance_id 生成规则：基于世界唯一特征
    // 单人存档：UUID.nameUUIDFromBytes(("world_" + saveDirName).getBytes())
    // 远程服务器：UUID.nameUUIDFromBytes(("server_" + host + ":" + port).getBytes())
}
```

**持久化路径：**

| 场景 | 身份文件路径 |
|------|-------------|
| 单人存档 save1 | `saves/save1/config/mcagent/world_identity.json` |
| 单人存档 save2 | `saves/save2/config/mcagent/world_identity.json` |
| 专用服务器 | `./config/mcagent/world_identity.json` |
| 客户端连接远程服务器 | `config/mcagent/servers/play.example.com_config/mcagent/world_identity.json` |

### 3.3 上下文管理器

```java
/**
 * 世界上下文管理器。
 * <p>
 * 管理当前活跃世界上下文，处理世界切换时的生命周期。
 * 线程安全，可在任意线程访问。
 */
public final class WorldContextManager {

    private static final Logger LOG = ...;
    private static final WorldContextManager INSTANCE = new WorldContextManager();

    /** 当前活跃的世界上下文。 */
    private volatile WorldContext activeContext;

    private WorldContextManager() {}

    /** 获取当前活跃上下文。 */
    public static WorldContext getActive() { return activeContext; }

    /**
     * 创建并激活一个新世界上下文。
     * 在 ServerLifecycleEvents.SERVER_STARTED 中调用。
     */
    public static WorldContext activate(MinecraftServer server) {
        // 1. 关闭前一个上下文（如果有）
        deactivate();

        // 2. 构建身份
        WorldIdentity identity = WorldIdentity.forServer(server);

        // 3. 创建新上下文
        WorldContext ctx = new WorldContext(server, identity);
        ctx.initialize();

        // 4. 设为活跃
        activeContext = ctx;
        LOG.info("WorldContext activated: {}", ctx);
        return ctx;
    }

    /**
     * 停用当前世界上下文。
     * 在 ServerLifecycleEvents.SERVER_STOPPING 中调用。
     */
    public static void deactivate() {
        WorldContext ctx = activeContext;
        if (ctx != null) {
            ctx.shutdown();
            activeContext = null;
            LOG.info("WorldContext deactivated: {}", ctx);
        }
    }

    /** 是否有活跃上下文。 */
    public static boolean isActive() { return activeContext != null; }
}
```

### 3.4 WorldContext 完整设计

```java
public class WorldContext implements AutoCloseable {

    /** MinecraftServer 引用（此上下文对应的服务器实例）。 */
    private final MinecraftServer server;

    /** 世界身份。 */
    private final WorldIdentity identity;

    /** TCP 客户端（与 Agent Core 通信）。 */
    private final TcpClient tcpClient;

    /** 假人管理器。 */
    private final BotManager botManager;

    /** 状态采集器。 */
    private final StatusCollector statusCollector;

    /** 事件分发器。 */
    private final EventDispatcher eventDispatcher;

    /** 活跃的假人生成索引（BotManager 内部维护）。 */
    /** 上下文启动时间戳。 */
    private final long startTime;

    // ── 构造与初始化 ──

    public WorldContext(MinecraftServer server, WorldIdentity identity) {
        this.server = server;
        this.identity = identity;
        this.botManager = new BotManager();
        this.eventDispatcher = new EventDispatcher();
        this.statusCollector = new StatusCollector(...);
        this.tcpClient = new TcpClient(...);
        this.startTime = System.currentTimeMillis();
    }

    /** 初始化上下文（启动 TCP 连接、恢复假人注册表等）。 */
    public void initialize() {
        // 1. BotManager 初始化（从 SavedData 恢复假人注册表）
        botManager.init(server);

        // 2. 启动 TCP 客户端（连接 Agent Core）
        tcpClient.connect(identity.host(), identity.port());

        // 3. 注册服务端 tick 事件
        ServerTickEvents.END_SERVER_TICK.register(s -> botManager.tick());

        LOG.info("WorldContext initialized: world={}, id={}",
                identity.worldName(), identity.instanceId());
    }

    /** 关闭上下文（断开 TCP、停止采集、清理假人）。 */
    @Override
    public void shutdown() {
        // 1. 停止状态上报
        statusCollector.stop();

        // 2. 断开 TCP（主动通知 Agent Core 当前世界下线）
        tcpClient.disconnect();

        // 3. 下线所有假人（保存存档）
        botManager.shutdown();

        LOG.info("WorldContext shut down: world={}", identity.worldName());
    }

    // ── 访问器 ──

    public MinecraftServer getServer() { return server; }
    public WorldIdentity getIdentity() { return identity; }
    public TcpClient getTcpClient() { return tcpClient; }
    public BotManager getBotManager() { return botManager; }
    public StatusCollector getStatusCollector() { return statusCollector; }
    public EventDispatcher getEventDispatcher() { return eventDispatcher; }
    public long getUptimeMs() { return System.currentTimeMillis() - startTime; }
}
```

## 第4章 多存档场景流程

### 4.1 单人模式：正常启动→游戏→退出

```
时间线
│
├── Fabric Loader 启动
│   └── ModInitializer.onInitialize()
│       ├── 工具扫描注册（一次）
│       ├── 插件发现（一次）
│       ├── 注册 ServerLifecycleEvents
│       └── 注册 ClientTickEvents（世界切换检测）
│
├── 用户选择世界 "New World" 并进入
│   └── ServerLifecycleEvents.SERVER_STARTED
│       └── WorldContextManager.activate(server)
│           ├── 创建 WorldIdentity（基于存档名 "New World"）
│           ├── 初始化 BotManager（恢复 SavedData）
│           ├── 启动 TCP 客户端 → 连接 Agent Core
│           ├── 注册工具 → register_tools
│           └── 启动状态上报
│
├── 玩家在 "New World" 中游戏
│   ├── TCP 连接正常 → 接收 tool_call → 执行工具
│   ├── 状态每 2s 上报
│   └── 假人管理正常工作
│
├── 玩家保存并退出世界
│   └── ServerLifecycleEvents.SERVER_STOPPING
│       └── WorldContextManager.deactivate()
│           ├── 停止状态上报
│           ├── 断开 TCP → 通知 Agent Core 世界下线
│           │   └── 发送 notification("world_offline", {world_name, instance_id})
│           ├── BotManager 下线所有假人（保存存档）
│           └── 清理上下文（供 GC）
│
└── 游戏回到主菜单（等待下一个世界选择）
```

### 4.2 单人模式：切存档（不退出游戏）

```
同 4.1 的前半部分...（世界 "New World" 启动）

├── 玩家保存 "New World" → 返回主菜单
│   └── ServerLifecycleEvents.SERVER_STOPPING
│       └── WorldContextManager.deactivate()  ← 世界 A 下线
│
├── 玩家选择世界 "Another World" 并进入
│   └── ServerLifecycleEvents.SERVER_STARTED
│       └── WorldContextManager.activate(server)  ← 世界 B 上线
│           ├── 新的 WorldIdentity（基于 "Another World"）
│           ├── 新的 TcpClient 连接
│           ├── 新的 BotManager 恢复
│           └── ...（与 4.1 同）
│
└── 每个世界完全隔离，互不影响
```

### 4.3 连接远程服务器

```
├── Fabric Loader 启动（同 4.1）
│
├── 用户连接 "play.example.com:25565"
│
├── 客户端侧（安装 Alice Mod 但作为客场）：
│   └── ServerLifecycleEvents.SERVER_STARTED 未触发（无本地 Server）

│
│   └── 服务端侧（play.example.com）：
│       ├── [情况 A] 服务端安装了 Alice Mod
│       │   └── ServerLifecycleEvents.SERVER_STARTED ✅

│       │           ├── WorldIdentity：{server_play.example.com:25565, ...}
│       │           ├── TCP → Agent Core
│       │           └── 假人在服务端创建与操作
│       │
│       └── [情况 B] 服务端未安装 Alice Mod
│           └── 无任何激活，完全透明
│
└── 用户断开连接
    └── [情况 A] 服务端 ServerLifecycleEvents.SERVER_STOPPING

## 第5章 世界身份（WorldIdentity）设计

### 5.1 身份生成规则

```java
public static WorldIdentity forServer(MinecraftServer server) {
    if (server.isDedicatedServer()) {
        // 专用服务器
        return loadOrCreate(
            server.getServerDirectory().resolve("config/mcagent/world_identity.json"),
            "server_dedicated"
        );
    } else {
        // 集成服务器（单人模式）
        String worldDirName = server.getWorldData().getLevelName();
        return loadOrCreate(
            server.getServerDirectory()
                .resolve("saves")
                .resolve(worldDirName)
                .resolve("config/mcagent/world_identity.json"),
            worldDirName
        );
    }
}

private static WorldIdentity loadOrCreate(Path identityFile, String worldTag) {
    if (Files.exists(identityFile)) {
        // 读取已有身份
        return loadFromFile(identityFile);
    }
    // 创建新身份
    String instanceId = UUID.nameUUIDFromBytes(
        ("alice_world_" + worldTag + "_" + System.currentTimeMillis()).getBytes()
    ).toString();
    String authToken = "mct_" + UUID.randomUUID().toString()
        .replace("-", "").substring(0, 24);
    WorldIdentity identity = new WorldIdentity(instanceId, authToken, worldTag);
    saveToFile(identityFile, identity);
    return identity;
}
```

## 第6章 BotManager 世界化

### 6.1 从静态单例到实例化

当前 `BotManager` 是全局静态单例。多存档场景下需要改为可实例化的对象：

**当前（静态单例）：**
```java
public final class BotManager {
    private static final Map<UUID, EntityPlayerMPFake> bots = new ConcurrentHashMap<>();
    private static MinecraftServer server;
    
    public static EntityPlayerMPFake spawn(MinecraftServer server, String name, ...) { ... }
    public static void onServerStarted(MinecraftServer server) { ... }
    public static void onServerStopped() { ... }
}
```

**目标（实例化）：**
```java
public final class BotManager {
    private final Map<UUID, EntityPlayerMPFake> bots = new ConcurrentHashMap<>();
    private final MinecraftServer server;
    
    public BotManager(MinecraftServer server) {
        this.server = server;
    }
    
    public EntityPlayerMPFake spawn(String name, ServerLevel level, Vec3 pos) { ... }
    // 不再需要 static，不再需要 server 参数（通过构造传入）
}
```

**调用链变化：**

```
当前：
  AliceModAdapter.onInitialize()
    → BotManager.onServerStarted(server)     ← 设置静态 server 引用
    → BotManager.tick()                       ← 访问静态 server
    → BotManager.spawn(server, name, level, pos)  ← 每次传参

目标：
  WorldContext ctx = new WorldContext(server, ...);
    → ctx.getBotManager()                     ← 获取上下文中的管理器
    → ctx.getBotManager().spawn(name, level, pos)  ← 无需传 server
    → ctx.getBotManager().tick()              ← 由 WorldContext 驱动
```

### 6.2 兼容性过渡方案

为了平滑过渡且不破坏现有代码，分两阶段实施：

**阶段一（过渡期）：**
- `BotManager` 保留静态方法，内部委托到当前 `WorldContext` 的实例
- 增加 `BotManager.setCurrentContext(WorldContext)` 设置活跃上下文
- 现有工具代码无需改动

```java
// 过渡期 BotManager
public final class BotManager {
    // 保留静态方法，委托给活跃上下文
    private static volatile BotManager instance;
    
    public static void setInstance(BotManager mgr) {
        instance = mgr;
    }
    
    // 现有工具调用 BotManager.spawn(server, name, level, pos)
    // 变更为委托到 instance
    public static EntityPlayerMPFake spawn(MinecraftServer server, String name, 
                                            ServerLevel level, Vec3 pos) {
        if (instance != null && instance.server == server) {
            return instance.spawnInternal(name, level, pos);
        }
        throw new IllegalStateException("No active BotManager for this server");
    }
}
```

**阶段二（完全实例化）：**
- 所有使用 `BotManager.spawn()` / `BotManager.findByName()` 等静态方法的代码改为通过 `WorldContext` 调用
- 静态方法标记为 `@Deprecated`
- 移除 `BotManager` 静态委托

## 第9章 Agent Core 侧变更

### 9.1 概念变化：从"Mod 会话"到"World 会话"

当前 Agent Core 将每个 TCP 连接视为一个**Mod 会话（Session）**——一个实例 ID 对应一个 Bot 集合。引入 WorldContext 后，需要将视角升级为**World 会话**：

```
当前模型：
  TCP 连接 ←→ Mod 实例 (instance_id)
                 └── 一个 Bot 集合（跨存档混淆）

目标模型：
  TCP 连接 ←→ World 会话 (instance_id_world)
                 │
                 ├── 单人存档 A：instance_id_A, world_name="New World"
                 ├── 单人存档 B：instance_id_B, world_name="Another World"
                 ├── 专用服务器：instance_id_C, world_name="server"
                 └── 切存档时：
                     旧连接 world_offline → 新连接 handshake (不同 instance_id)
```

### 9.2 协议变更

#### 9.2.1 Handshake 消息扩展（v2）

握手阶段需要携带世界身份信息，让 AC 区分不同世界：

```json
// 当前 handshake (v1)
{
  "jsonrpc": "2.0",
  "method": "handshake",
  "params": {
    "instance_id": "xxx",
    "auth_token": "mct_xxx",
    "version": "1.0.0",
    "mod_name": "alice-mod"
  }
}

// 目标 handshake (v2)
{
  "jsonrpc": "2.0",
  "method": "handshake",
  "params": {
    "instance_id": "d4e5f6a7-...",
    "auth_token": "mct_a1b2c3d4...",
    "version": "1.0.0",
    "mod_name": "alice-mod",
    // ---- 新增字段 ----
    "world_name": "New World",
    "world_online": true,
    "edition": "java",
    "game_version": "1.21.4"
  }
}
```

#### 9.2.2 新增 world_offline 通知

世界关闭/切换时，Alice Mod 主动通知 AC 当前世界下线：

```json
{
  "jsonrpc": "2.0",
  "method": "world_offline",
  "params": {
    "instance_id": "d4e5f6a7-...",
    "world_name": "New World",
    "uptime_seconds": 5400,
    "bot_count": 3,
    "reason": "world_switch"   // world_switch | server_stop
  }
}
```

#### 9.2.3 新增 world_online 通知

世界加载完成后补充发送（与 handshake 配合），适用于同一 TCP 连接内发生了世界切换但 instance_id 未变的情况：

```json
{
  "jsonrpc": "2.0",
  "method": "world_online",
  "params": {
    "instance_id": "d4e5f6a7-...",
    "world_name": "Another World",
    "bot_count": 0
  }
}
```

### 9.3 AC 侧的会话映射

AC 需要维护一个双层映射：

```
AC Session Manager
│
├── connection_id (TCP 连接唯一标识)
│   └── current_world: WorldSession
│       ├── instance_id
│       ├── world_name
│       ├── auth_token
│       ├── connected_at
│       └── bot_context: { tool_registry, bot_states }
│
├── world_history: [WorldSession]     ← 该连接的历史世界记录
│
└── instance_index: Map<instance_id, WorldSession>  ← 用于跨连接查询
```

关键变化：

| 维度 | 当前 | 目标 |
|------|------|------|
| 会话标识 | 仅 instance_id | instance_id + world_name |
| 断线重连 | 恢复上次会话 | 同一个 world_name 可恢复，不同的视为新会话 |
| store/list 范围 | 模组级别 | 世界级别（每个 world 独立 context） |

### 9.4 世界切换时的 AC 行为

```
时间线                                          Agent Core
──────                                          ──────────

世界 A 运行中
Client → handshake {instance_id_A, "New World"}
                                                 创建 WorldSession A
                                                 工具调用 → Alice Mod A

世界 A 关闭
Client → world_offline {instance_id_A, "New World"}
                                                 标记 WorldSession A 为 OFFLINE
                                                 保留 context 供后续恢复

世界 B 启动
Client → handshake {instance_id_B, "Another World"}
                                                 创建 WorldSession B（与 A 隔离）
                                                 开始接收 B 的工具调用

世界 A 再次启动
Client → handshake {instance_id_A, "New World"}
                                                 查找已有 instance_id_A
                                                 恢复 WorldSession A 的 context
                                                 通知 LLM "世界 A 重新上线"
```

### 9.5 数据隔离要求

AC 侧存储的数据需要按 world 隔离：

```
Agent Core Storage
└── instances/
    └── <instance_id>/
        ├── world_<world_name>/
        │   ├── tool_call_history.jsonl    ← 该世界的工具调用历史
        │   ├── bot_config.json            ← 该世界的假人配置
        │   ├── task_queue.json            ← 该世界的任务队列
        │   └── status_history/            ← 该世界的状态上报历史
        │
        └── meta.json                     ← 所有世界的索引
                                            {worlds: ["New World", ...]}
```

### 9.6 对外 API 变更（AC → Agent 层）

Agent Core 对外暴露的 API 需要增加 world 参数：

| API | 当前 | 目标 |
|-----|------|------|
| 获取实例列表 | `GET /instances` | `GET /instances?world=<world_name>`（可选过滤） |
| 获取工具列表 | `GET /instances/:id/tools` | 不变（工具集不随世界变化） |
| 调用工具 | `POST /instances/:id/tool_call` | `POST /instances/:id/world/:world/tool_call` |
| 获取状态 | `GET /instances/:id/status` | `GET /instances/:id/world/:world/status` |
| 事件流 | `GET /instances/:id/events` | `GET /instances/:id/world/:world/events` |

### 9.7 多连接场景（客户端 + 服务端）

当客户端和服务端都安装了 Alice Mod 时：

```
Agent Core
│
├── TCP Connection C1 (客户端, 客场休眠)
│   └── handshake → instance_id_client
│       └── world_online=false（客场）
│       └── AC 标记为"客户端侧，仅出口"
│
├── TCP Connection C2 (服务端, play.example.com)
│   └── handshake → instance_id_server
│       └── world_online=true
│       └── AC 标记为"服务端侧，可执行工具"
│
└── 两连接互不干扰，AC 分别管理
```

AC 可以通过 `world_online` 字段判断该连接是否为活跃世界，只向活跃世界下发 tool_call。

## 附录B：容错与边界情况

| 场景 | 行为 |
|------|------|
| Agent Core 未启动，TCP 连接失败 | TcpClient 内部重连（指数退避），WorldContext 保持活跃标记 |
| 世界初始化时 TCP 连接失败 | 记录日志，WorldContext 仍正常激活，工具调用排队等待 TCP 恢复 |
| 游戏中 TCP 断开 | 状态上报暂停，工具调用返回 "connection lost" 错误，WorldContext 存活 |
| 世界保存时崩溃 | `.dat` 文件由 Minecraft 自动备份保护，BotRepository 由 SavedData 保护 |
| 快速切世界（< 1 秒） | deactivate() → activate() 顺序执行，前一个 TcpClient 完全断开后才创建新连接 |
| 同一个世界连续加载/退出 | 每次都是新 WorldContext，但 instance_id 相同（从 world_identity.json 读取） |
