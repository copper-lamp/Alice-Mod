## 第1章 概述

### 1.1 模块定位

Alice Mod JE 是 McAgent 系统中运行在 **Minecraft Java Edition 客户端** 上的 **Fabric 模组**，基于 **Carpet Mod** 提供的假人基础设施构建。它是系统的"身体"，承担两个核心角色：

| 角色 | 说明 |
|------|------|
| **AI 执行器** | 通过 TCP 客户端与 Agent Core 通信，接收并执行 LLM 的低级操作指令 |
| **游戏内工具** | 通过 Minecraft Java API + Carpet Mod 实现假人控制、环境感知、状态上报等 26 个工具 |

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **站在巨人的肩膀上** | 优先利用 Fabric 生态已有的成熟方案，不自研轮子 |
| **Carpet 优先** | 假人生命周期、基础动作控制依赖 Carpet Mod |
| **接口解耦** | 工具契约不绑定 Minecraft 概念（参考业界成熟的 NumenTool 接口模式） |
| **版本兼容** | 版本敏感 API 统一走 compat 适配层，升级 MC 版本时只需改适配层 |
| **BE 独立** | JE 版与 BE 版各自独立实现，仅协议层面对齐 |

### 1.3 技术栈

| 技术 | 版本 | 用途 |
|------|------|------|
| **Java** | 21 (LTS) | 开发语言 |
| **Fabric Loader** | 0.16.x | 模组加载器 |
| **Fabric API** | 0.105.x+ | 模组 API，提供游戏交互接口 |
| **Carpet Mod** | 1.4.161+ | 假人基础设施（EntityPlayerMPFake、EntityPlayerActionPack） |
| **Minecraft** | 1.21.x | 目标游戏版本 |
| **Gradle** | 8.x | 构建工具（Fabric Loom 插件） |
| **Java Socket** | JDK 内置 | TCP 客户端网络通信 |
| **Gson** | 最新版 | JSON 序列化/反序列化 |
| **SQLite JDBC** | 最新版 | 数据持久化 |
| **SLF4J** | Fabric 内置 | 日志门面 |
| **JUnit 5 (Jupiter)** | 最新版 | 单元测试 |

### 1.4 术语表

| 术语 | 说明 |
|------|------|
| **Agent Core** | 外部 LLM 调度中心，通过 TCP 发送工具调用指令 |
| **Carpet Mod** | Fabric 生态模组，提供假人（Fake Player）能力 |
| **EntityPlayerMPFake** | Carpet Mod 中的假人实体类，继承 ServerPlayer |
| **EntityPlayerActionPack** | Carpet Mod 中的假人动作控制器 |
| **compat 层** | 版本适配层，隔离 MC 版本差异 |
| **numen-api** | 社区项目，其公开接口的设计模式被本模组参考 |

## 第3章 模块设计与代码复用策略

### 3.1 模块总览

Alice Mod JE 共 8 个模块（C1-C8），与需求文档的模块划分一致。实现上充分利用 Fabric 生态和社区项目的成熟设计。

| 模块 | 编号 | 实现策略 |
|------|:----:|----------|
| TCP 客户端 | C1 | 自研：Java Socket + BufferedReader/PrintWriter |
| 工具注册 | C2 | 参考业界通用的工具接口模式设计，自研注解+注册表 |
| 假人管理 | C3 | 基于 Carpet Mod 的 `EntityPlayerMPFake` + 自研管理封装 |
| 状态上报 | C4 | 自研：周期性采集 + 事件监听 |
| 执行AI 引擎 | C5 | 自研 7 个子模块，寻路算法参考业界的 A* 实现思路 |
| 数据持久化 | C6 | 自研：SQLite JDBC + Repository 模式 |
| JSON 入口 | C7 | 自研：首次启动生成 `mcagent_instance.json` |
| 配置接入 | C8 | 基于 Fabric Command API 自研 |

### 3.2 各模块详细设计

#### C1: TCP 客户端模块

**职责**：与 Agent Core 建立 TCP 长连接，消息编解码，心跳保活，断线重连。

**设计**：
- `TcpClient`：封装 Java Socket，提供 connect/disconnect/send/receive
- `JsonRpcCodec`：JSON-RPC 2.0 消息编解码（基于 Gson）
- `MessageFrameCodec`：`\n` 分隔的粘包处理
- `HeartbeatManager`：响应 Agent Core 的 ping，回复 pong
- `ReconnectManager`：指数退避重连（1s→2s→4s→8s→16s，最多 5 次）

**外部依赖**：无（仅使用 JDK 内置的 java.net.Socket）

#### C2: 工具注册模块

**职责**：工具注解定义、运行时扫描注册、JSON Schema 生成、工具调用调度。

**设计**：
- 工具接口参考业界成熟的 `NumenTool` 接口模式设计（纯 Java 接口，不绑定 Minecraft 概念）
- 使用 Java 内置的注解 + 反射机制实现扫描，不依赖外部库
- 工具结果信封 `ToolResult` 参考社区通用的 `TaskResult` 设计模式

**工具接口定义**：
```java
public interface AliceTool {
    String name();
    String description();
    Map<String, Object> parameterSchema();
    ToolResult invoke(Map<String, Object> args);
}
```

**工具结果信封**：
```java
public record ToolResult(
    boolean success,
    String message,
    Map<String, Object> data
) { }
```

**外部依赖**：无（仅使用 Java 反射 + Gson）

#### C3: 假人管理模块（基于 Carpet Mod）

**职责**：假人的创建、销毁、生命周期管理、动作控制。

**技术基础**：
- **Carpet Mod** 的 `EntityPlayerMPFake` 提供了完整的假人实体（继承 `ServerPlayer`）
- 假人的连接处理参考社区项目的 `FakeConnection` 思路实现（继承 `Connection`，丢弃出站包）
- 假人注册表使用 Minecraft 原生的 `SavedData` 机制持久化

**关键设计**：
- 假人是服务端第一等公民（`ServerPlayer`），所有游戏机制天然兼容
- 通过 Carpet 的 `EntityPlayerActionPack` 控制基础动作（移动/攻击/使用）
- 自定义 NBT 存储玩家 UUID，实现跨维度的主人识别

**外部依赖**：Carpet Mod (>=1.4.161)

#### C5: 执行AI 引擎

**职责**：7 个子模块，实现 26 个游戏内工具。

**子模块划分**：

| 编号 | 子模块 | 包含工具 | 自研重点 |
|:----:|--------|----------|----------|
| C5.1 | 寻路系统 | — | A* 3D 算法，基于 `BlockState.getCollisionShape()` 构建导航网格 |
| C5.2 | 移动系统 | `move_to`/`ride`/`dismount` | 路径跟随控制器，视角平滑旋转 |
| C5.3 | 背包操作 | `drop_item`/`take_from_container`/`put_to_container`/`equip_item` | ScreenHandler 容器 GUI 交互 |
| C5.4 | 方块操作 | `mine_block`/`place_block`/`use_block`/`area_operation` | 自动选工具，区域 BFS |
| C5.5 | 战斗系统 | `set_combat_mode`/`stop_combat` | 5 种战斗模式状态机 |
| C5.6 | 生物交互 | `interact_entity`/`lead_entity` | 动物交互、村民交易 GUI |
| C5.7 | 生存操作 | `eat`/`sleep`/`use_item` | 食物选择、找床算法 |

**寻路算法**：自研 A* 3D 寻路，参考业界的 A* 实现思路进行设计。基于 Minecraft Java API 的方块碰撞数据构建导航网格，支持跳跃/攀爬/游泳路径点。

**外部依赖**：无（仅使用 Minecraft 原生的 World/Block API）

## 第5章 版本适配策略

### 5.1 compat 适配层

所有版本敏感的 API 调用统一封装在 `compat` 包中，升级 Minecraft 版本时只需修改该包下的文件。

```
compat/
├── PlayerLoadHelper.java    // PlayerList.load() — 1.21.4 vs 1.21.5+ 返回值差异
├── ServerPlayerHelper.java  // ServerPlayer 构造参数差异
└── Version.java             // 当前 MC 版本检测
```

### 5.2 已知版本差异

| API | 1.21.4 | 1.21.5+ (26.x) | 适配方式 |
|-----|--------|----------------|----------|
| `PlayerList.load()` | 返回 `CompoundTag` | 返回 `Optional<CompoundTag>` | compat 层适配 |
| `ServerPlayer` 构造 | 4 参数 | 可能有变化 | compat 层适配 |
| `Connection` 构造 | `PacketFlow` 参数 | 同左 | 兼容 |
| `CommonListenerCookie` | `createInitial` | 同左 | 兼容 |
| 方块状态 API | `BlockState` | 同左 | 兼容 |
| NBT 系统 | `CompoundTag` | 同左 | 兼容 |

### 5.3 版本升级流程

```
1. 更新 gradle.properties 中的 MC 版本号
2. 编译 → 观察编译错误（集中在 compat 层）
3. 调整 compat 层中对应的适配代码
4. 运行 Fabric GameTest 验证核心功能
5. 全量回归测试
```

## 第7章 与原始计划的差异说明

### 7.1 基于 Carpet Mod 带来的变化

| 维度 | 原始计划（JE-02） | 新计划（基于 Carpet） | 变化 |
|------|-------------------|---------------------|:----:|
| 假人实现 | 自研客户端实体 | 基于 Carpet 的 `EntityPlayerMPFake` | 省去假人底层实现 |
| 假人动作 | 自研动作系统 | Carpet 的 `EntityPlayerActionPack` | 基础动作直接可用 |
| V10 工时 | 25h | 22h | 减少 ~2 天 |
| 整体工时 | ~380h | ~340h | 减少 ~10% |

### 7.2 工具接口设计参考

工具系统采用的接口模式与社区中成熟的工具契约设计思路一致（如 numen-api 项目的 NumenTool 接口）。这种接口模式在 Minecraft AI 社区中被广泛采用，其核心特征为：
- 工具名称、描述、参数 Schema 的声明方式
- 工具结果的标准化信封格式
- 注册表的静态全局模式

### 7.3 假人连接处理

假人的网络连接处理（`FakeConnection`）采用社区中处理假人连接的标准做法：继承 `Connection` 类并丢弃出站数据包，使用 `EmbeddedChannel` 满足 Minecraft 的网络管道配置需求。这是 Fabric 假人实现中的常见模式，在 Carpet Mod 的 `EntityPlayerMPFake` 实现中也采用了类似的思路。
