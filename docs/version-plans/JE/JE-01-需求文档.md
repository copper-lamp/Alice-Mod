## 第1章 概述

### 1.1 模块定位

Adapter Core JE（Java Edition）是 McAgent 系统中运行在 **Minecraft Java Edition 客户端** 上的 **Fabric 模组**，使用 Java 语言实现。它是系统的"身体"，承担两个核心角色：

| 角色 | 说明 |
|------|------|
| **AI 执行器** | 通过 TCP 客户端与 Agent Core 通信，接收并执行 LLM 的低级操作指令 |
| **游戏内工具** | 通过 Minecraft Java API 实现假人控制、环境感知、状态上报等 26 个工具 |

**核心职责**：
- 建立与 Agent Core 的 TCP 长连接，遵循 JSON-RPC 2.0 协议通信
- 实现 26 个游戏内工具（感知x5 + 移动x3 + 背包x4 + 生物x4 + 生存x3 + 方块x4 + 对话x3）
- 自动扫描并注册工具到 Agent Core，供 LLM 调用
- 周期性上报假人状态（每 2s），主动推送事件通知
- 提供 A* 3D 寻路能力，支持假人自主移动
- 管理假人生命周期（创建/销毁）
- 数据持久化（SQLite JDBC 存储配置、日志、状态）
- 支持 Fabric 指令 `/mcagent config` 进行游戏内配置

### 1.2 技术栈

| 技术 | 版本 | 用途 |
|------|------|------|

### 1.3 团队

| 角色 | 人数 | 职责 |
|------|:----:|------|
| **JE 开发者 A** | 1 | 执行AI 引擎（寻路/移动/背包/方块）+ 感知工具 + 对话工具 |
| **JE 开发者 B** | 1 | 战斗系统 + 生物交互 + 生存工具 + 假人管理 + 配置模块 |

两人并行开发，同一版本的 BE/JE 任务结构相同但实现语言不同。

### 1.4 BE/JE 差异概述

| 维度 | JE（Java 版） | BE（基岩版） |
|------|--------------|-------------|
| **运行环境** | Minecraft Java 客户端 | BDS 服务器 |
| **模组框架** | Fabric Loader + Fabric API | LeviLamina 插件框架 |
| **开发语言** | Java 21 | TypeScript / Node.js |
| **TCP 客户端** | Java Socket 自实现 | Node.js net 模块 |
| **JSON 序列化** | Gson | 原生 JSON |
| **数据持久化** | SQLite JDBC | better-sqlite3 |
| **构建工具** | Gradle (Fabric Loom) | npm / pnpm |
| **假人实现** | 客户端实体（依赖客户端进入世界） | 服务器实体（更灵活） |
| **运行模式** | 需 Minecraft 客户端图形界面 | 无头 BDS 后台运行 |
| **工具注解** | `@ToolModule` / `@ToolMethod` Java 注解 | TypeScript 装饰器 |

## 第4章 非功能需求

### 4.1 性能指标

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| 状态上报周期 | ≤ 2s | 从游戏状态采集到通知发送到 Agent Core 的全链路耗时 |
| 工具超时时间 | < 30s（默认） | 单个工具从接收到返回结果的执行时间 |
| 寻路计算耗时 | < 50ms（单次 A* 搜索，100 节点内） | A* 算法从输入到输出路径的耗时 |
| TPS 影响 | < 5% | 模组加载前后 Minecraft TPS 对比 |
| Gson 序列化耗时 | < 5ms | 单个 JSON-RPC 消息的序列化/反序列化 |
| GC 暂停间隔 | < 200ms | JVM GC 日志分析 |
| 内存占用（模组自身） | < 256MB | 排除 Minecraft 基础内存后的增量 |
| 连续运行稳定性 | ≥ 72h 无崩溃 | 超时时间、无内存泄漏 |

### 4.2 可用性指标

| 指标 | 目标值 |
|------|--------|
| 断线重连成功率 | ≥ 99%（网络恢复后 30s 内） |
| 握手成功率 | ≥ 99.9% |
| 工具调用成功率 | ≥ 99%（排除游戏本身异常） |
| 并发假人数量 | ≤ 3 个 |
| 启动时间 | < 5s（模组加载到 TCP 连接建立） |

### 4.3 安全性指标

| 指标 | 要求 |
|------|------|
| TCP 认证 | 必须有 auth_token 握手认证 |
| 配置安全 | auth_token 以环境变量或加密文件存储 |
| 日志审计 | 所有工具调用记录持久化到 logs 表 |
| 假人隔离 | 多假人状态独立，互不干扰 |

## 第7章 与 BE 差异说明

### 7.1 相同点

| 维度 | 相同内容 |
|------|----------|
| **模块划分** | C1-C8 模块划分完全一致 |
| **工具定义** | 26 个工具的名称、输入参数、输出格式完全一致 |
| **通信协议** | JSON-RPC 2.0 over TCP，消息格式、心跳机制、重连策略一致 |
| **JSON 入口文件** | mcagent_instance.json 的 Schema 一致 |
| **数据库表结构** | config/memory_meta/logs 表结构一致 |
| **状态上报格式** | 状态数据字段结构一致 |
| **事件通知类型** | 事件类型枚举一致 |
| **版本迭代计划** | V1-V15 版本划分一致 |

### 7.2 差异点

| 维度 | JE（Java 版） | BE（基岩版） | 影响 |
|------|--------------|-------------|------|
| **类型体系** | Java 强类型（int/double/String/Enum） | TypeScript 类型 | 参数校验方式不同，JE 用注解定义 Schema |
| **注解扫描** | `@ToolModule`/`@ToolMethod` 运行时注解扫描 | TypeScript 装饰器 + 文件扫描 | JE 需自实现注解扫描器 |
| **游戏 API** | Fabric API / Minecraft Java API（`World.getBlockState()` 等） | BDS API（LeviLamina） | 所有工具实现逻辑不同 |
| **JSON 序列化** | Gson `new GsonBuilder().create()` | 原生 JSON `JSON.stringify()` | JE 需额外配置 Gson 类型适配器 |
| **数据持久化** | SQLite JDBC（java.sql.Connection/PreparedStatement） | better-sqlite3（Node.js） | API 完全不同 |
| **TCP 客户端** | Java Socket（`Socket`/`BufferedReader`/`PrintWriter`） | Node.js net 模块 | 事件模型不同，JE 使用阻塞 I/O + 工作线程 |
| **日志** | SLF4J（`LoggerFactory.getLogger()`） | winston 或 console | 配置方式不同 |
| **测试框架** | JUnit 5 + Fabric GameTest | Vitest + Mocha | 测试编写方式不同 |
| **构建工具** | Gradle（build.gradle） | npm/pnpm（package.json） | 构建流程不同 |
| **配置指令** | Fabric 指令系统 | LeviLamina 指令系统 | 注册方式不同 |
| **假人类型** | 客户端实体（ClientEntity） | 服务器实体 | 生命周期和操作方式不同 |

### 7.3 关键差异处理策略

1. **工具 Schema 生成**：JE 通过 `@ToolMethod` 注解的元数据在运行时反射生成 JSON Schema，BE 通过 TypeScript 类型定义静态生成。两者输出的 Schema 格式一致。
2. **类型适配**：JE 的 `ToolParam` 注解中 `type` 字段使用字符串（"number"/"string"/"boolean"），与 BE 的 JSON Schema `type` 字段保持一致。
3. **状态采集**：JE 通过 Minecraft Java API 的 `PlayerEntity.getHealth()`/`getHungerManager()` 等方法采集状态，BE 通过 BDS API。输出格式一致。
4. **A* 寻路**：JE 使用 Minecraft Java 的 `BlockState.isFullCube()` / `BlockCollisionShape` 判断方块可通过性，BE 使用 BDS 的方块数据。算法逻辑相同，数据来源不同。
