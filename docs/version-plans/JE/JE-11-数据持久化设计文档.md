## 第1章 概述

### 1.1 背景

Alice Mod JE 目前已有三种持久化方式：

| 持久化方式 | 用途 | 位置 |
|-----------|------|------|
| Minecraft SavedData (`BotRepository`) | 假人注册表索引（名称、UUID、位置） | 世界存档 `data/` 目录 |
| JSON 文件 (`WorldIdentity`) | 世界身份信息（instance_id、auth_token） | 各世界 `config/mcagent/` |
| JSON 文件 (`InstanceFileGenerator`) | 模组入口信息（TCP 配置、认证） | 全局 `config/mcagent/` |

但这些方式各有局限：

- **SavedData** 耦合于 Minecraft 的存档保存机制，不适合存储与存档无关的运行时数据（如工具执行日志、配置项）
- **JSON 文件** 缺乏结构化查询能力，多条记录需要全量读写，无法按条件查询
- **缺少统一的数据管理入口**，配置、日志、记忆数据分散在各模块中

V11 引入 SQLite 作为统一的结构化持久化层，解决上述问题。

### 1.2 设计目标

| 目标 | 说明 |
|------|------|
| **结构化存储** | 提供表结构化的配置、日志、记忆存储能力 |
| **世界隔离** | 每个世界独立数据库，切换存档时数据隔离 |
| **统一入口** | 通过 `DatabaseManager` 统一管理连接生命周期 |
| **轻量嵌入** | 使用 SQLite 嵌入模式，无需独立数据库服务 |
| **WAL 模式** | 启用 WAL 模式提升并发读写性能 |
| **Repository 模式** | 每个表对应一个 Repository 类，封装 CRUD 操作 |

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| **按世界分库** | 遵循 JE-10 设计，每个世界拥有独立的 SQLite 数据库文件 |
| **Repository 封装** | 对外暴露 Repository 接口，隐藏 SQL 细节 |
| **延迟初始化** | 数据库连接在首次使用时创建，不在模组加载时初始化 |
| **线程安全** | 单连接 + `synchronized` 或连接池，确保线程安全 |
| **最小化依赖** | 仅依赖 `sqlite-jdbc`，不引入 ORM 框架 |

### 1.4 术语表

| 术语 | 说明 |
|------|------|
| **DatabaseManager** | 数据库管理器，管理连接生命周期和 Repository 注册 |
| **Repository** | 数据仓库，封装单个表的 CRUD 操作 |
| **WAL 模式** | SQLite Write-Ahead Logging 模式，提升并发性能 |
| **Per-World Database** | 每个世界独立的 SQLite 数据库文件 |

## 第2章 当前文件存储架构分析

### 2.1 现有持久化机制

```
config/mcagent/
├── mcagent_instance.json       ← 模组入口文件（全局，JSON）
│
├── worlds/                     ← 各世界独立数据目录（按需创建）
│   ├── New_World/
│   │   ├── world_identity.json ← 世界身份（JSON）
│   │   └── mcagent.db          ← 世界数据（SQLite，V11 新增）
│   │
│   └── Another_World/
│       ├── world_identity.json
│       └── mcagent.db
│
└── mcagent.db                  ← 全局数据（SQLite，V11 新增，可选）
```

### 2.2 各模块现有存储方式

| 模块 | 存储方式 | 存储内容 | 优缺点 |
|------|---------|---------|--------|
| `BotRepository` | Minecraft SavedData (NBT) | 假人索引：名称、UUID、维度、位置、创建时间 | ✅ 原生存档机制，自动保存；❌ 仅限假人注册表，查询能力弱 |
| `WorldIdentity` | JSON 文件 (Gson) | 世界身份：instance_id、auth_token | ✅ 简单可靠；❌ 无查询能力，仅适合单条记录 |
| `InstanceFileGenerator` | JSON 文件 (Gson) | 模组入口：TCP 配置、认证、数据库路径 | ✅ 外部程序可读；❌ 冗余信息，仅作入口 |
| `persistence` 包 | 空（仅 `package-info.java`） | — | 待实现 |

### 2.3 需要新增的持久化内容

| 数据类别 | 存储方式 | 说明 |
|---------|---------|------|
| **配置项** | SQLite `config` 表 | 键值对配置，替代 JSON 配置文件 |
| **工具执行日志** | SQLite `tool_logs` 表 | 每次工具调用的记录，用于调试和回溯 |
| **记忆元数据** | SQLite `memory_meta` 表 | 地图标记、区域标记、生物群系等 LLM 记忆数据 |
| **事件记录** | SQLite `event_logs` 表 | 游戏事件（假人创建/销毁、死亡、世界切换） |

## 第3章 数据库架构设计

### 3.1 数据库文件布局

遵循 JE-10 的 SQLite 分库策略，数据库文件按世界隔离：

```
<game_dir>/config/mcagent/
├── mcagent.db                  ← [可选] 全局数据库，存储跨世界共享数据
│
└── worlds/
    └── <WorldName>/
        ├── world_identity.json ← 世界身份（已有）
        └── mcagent.db          ← 世界数据库（V11 新增）
```

**数据库路径获取规则：**

| 场景 | 数据库路径 |
|------|-----------|
| 单人存档 `New World` | `<game_dir>/saves/New_World/config/mcagent/worlds/New_World/mcagent.db` |
| 专用服务器 | `<game_dir>/config/mcagent/worlds/dedicated/mcagent.db` |
| 全局数据库 | `<game_dir>/config/mcagent/mcagent.db` |

> **注意**：`WorldContext` 在 `initialize()` 时确定数据库路径，与 `WorldIdentity` 的持久化路径逻辑一致。

### 3.2 表结构设计

#### 3.2.1 `config` 表 — 配置存储

存储模组配置项，替代 JSON 配置文件。

```sql
CREATE TABLE IF NOT EXISTS config (
    key         TEXT PRIMARY KEY,          -- 配置键，如 "agent_core.host"
    value       TEXT NOT NULL,             -- 配置值（JSON 字符串，支持复杂类型）
    description TEXT DEFAULT '',           -- 配置描述
    category    TEXT DEFAULT 'general',    -- 配置分类：general / tcp / bot / ai
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 按分类查询配置
CREATE INDEX IF NOT EXISTS idx_config_category ON config(category);
```

**示例数据：**

| key | value | category |
|-----|-------|----------|
| `agent_core.host` | `"127.0.0.1"` | tcp |
| `agent_core.port` | `27541` | tcp |
| `bot.max_count` | `10` | bot |
| `bot.default_game_mode` | `"survival"` | bot |
| `status_report.interval` | `2` | general |
| `ai.pathfinding.max_nodes` | `5000` | ai |

#### 3.2.2 `tool_logs` 表 — 工具执行日志

记录每次工具调用的完整信息，用于调试、回溯和性能分析。

```sql
CREATE TABLE IF NOT EXISTS tool_logs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    tool_name   TEXT NOT NULL,             -- 工具名称，如 "move_to"
    args        TEXT NOT NULL DEFAULT '{}', -- 工具参数（JSON）
    success     INTEGER NOT NULL,          -- 是否成功：1=成功，0=失败
    message     TEXT DEFAULT '',           -- 结果消息
    duration_ms INTEGER NOT NULL,          -- 执行耗时（毫秒）
    world_name  TEXT NOT NULL,             -- 世界名
    instance_id TEXT NOT NULL,             -- 实例 ID
    bot_name    TEXT DEFAULT '',           -- 执行工具的假人名称
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 按工具名查询
CREATE INDEX IF NOT EXISTS idx_tool_logs_tool_name ON tool_logs(tool_name);
-- 按时间范围查询
CREATE INDEX IF NOT EXISTS idx_tool_logs_created_at ON tool_logs(created_at);
-- 按世界名查询
CREATE INDEX IF NOT EXISTS idx_tool_logs_world_name ON tool_logs(world_name);
```

#### 3.2.3 `memory_meta` 表 — 记忆元数据

存储 LLM 记忆系统中的地图标记、区域标记、生物群系等空间索引数据。与全局地图索引的 Chunk-aligned grid (16×16) 空间索引策略对齐。

```sql
CREATE TABLE IF NOT EXISTS memory_meta (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    type        TEXT NOT NULL,             -- 记忆类型：map_point / map_region / map_biome / custom
    name        TEXT DEFAULT '',           -- 名称（如 "基地"、"矿洞入口"）
    dimension   TEXT NOT NULL,             -- 维度：minecraft:overworld / minecraft:the_nether / minecraft:the_end
    chunk_x     INTEGER NOT NULL,          -- Chunk 坐标 X（chunk_x * 16 = 方块坐标）
    chunk_z     INTEGER NOT NULL,          -- Chunk 坐标 Z
    data        TEXT NOT NULL DEFAULT '{}', -- 附加数据（JSON，如方块状态、生物群系 ID）
    importance  INTEGER DEFAULT 1,         -- 重要性：1-5，越高越重要
    tags        TEXT DEFAULT '',           -- 标签（逗号分隔，如 "base,home,chest"）
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 按类型查询
CREATE INDEX IF NOT EXISTS idx_memory_meta_type ON memory_meta(type);
-- 按 Chunk 坐标查询（空间索引）
CREATE INDEX IF NOT EXISTS idx_memory_meta_chunk ON memory_meta(dimension, chunk_x, chunk_z);
-- 按重要性查询
CREATE INDEX IF NOT EXISTS idx_memory_meta_importance ON memory_meta(importance);
```

**空间索引策略**（与全局地图索引对齐）：

- 以 Chunk (16×16) 为空间索引单位
- `chunk_x = floor(block_x / 16)`，`chunk_z = floor(block_z / 16)`
- 查询时：`WHERE chunk_x BETWEEN ? AND ? AND chunk_z BETWEEN ? AND ? AND dimension = ?`

#### 3.2.4 `event_logs` 表 — 事件记录

记录游戏事件，供 LLM 回溯和分析。

```sql
CREATE TABLE IF NOT EXISTS event_logs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    event_type  TEXT NOT NULL,             -- 事件类型：bot_spawn / bot_death / bot_despawn / world_switch / connection / error
    source      TEXT DEFAULT '',           -- 事件来源：bot_name / "system" / "world_context"
    detail      TEXT DEFAULT '',           -- 事件详情描述
    data        TEXT DEFAULT '{}',         -- 附加数据（JSON）
    world_name  TEXT NOT NULL,             -- 世界名
    instance_id TEXT NOT NULL,             -- 实例 ID
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 按事件类型查询
CREATE INDEX IF NOT EXISTS idx_event_logs_type ON event_logs(event_type);
-- 按时间范围查询
CREATE INDEX IF NOT EXISTS idx_event_logs_created_at ON event_logs(created_at);
-- 按来源查询
CREATE INDEX IF NOT EXISTS idx_event_logs_source ON event_logs(source);
```

### 3.3 数据库版本管理

为了支持未来表结构变更，设计一个简单的版本表：

```sql
CREATE TABLE IF NOT EXISTS schema_version (
    version     INTEGER PRIMARY KEY,       -- 版本号，从 1 开始
    applied_at  TEXT NOT NULL DEFAULT (datetime('now')),
    description TEXT DEFAULT ''
);
```

**版本迁移策略：**

- `DatabaseManager` 启动时检查 `schema_version` 表
- 如果表不存在，创建所有表并将版本设为当前版本
- 如果版本低于当前版本，按顺序执行迁移脚本
- 初始版本：v1（包含上述 4 个表）

### 4.2 DatabaseManager

**职责**：管理 SQLite 连接生命周期，执行数据库迁移，提供 Repository 实例。

```java
package io.alice.mod.adapter.persistence;

/**
 * 数据库管理器 — 管理 SQLite 连接生命周期。
 * <p>
 * 每个世界对应一个独立的 DatabaseManager 实例，由 WorldContext 创建。
 * 全局数据库使用独立的 DatabaseManager 实例（可选）。
 * <p>
 * 启用 WAL 模式以提升并发读写性能。
 * 使用 Repository 模式封装表操作，对外隐藏 SQL 细节。
 */
public final class DatabaseManager implements AutoCloseable {

    /** SQLite JDBC URL 前缀。 */
    private static final String JDBC_PREFIX = "jdbc:sqlite:";

    /** 当前 schema 版本。 */
    private static final int SCHEMA_VERSION = 1;

    private final Path dbPath;
    private final String worldName;
    private final String instanceId;
    private Connection connection;

    // Repository 缓存
    private ConfigRepository configRepository;
    private ToolLogRepository toolLogRepository;
    private MemoryMetaRepository memoryMetaRepository;
    private EventLogRepository eventLogRepository;

    public DatabaseManager(Path dbPath, String worldName, String instanceId) { ... }

    /** 初始化数据库连接并执行迁移。 */
    public synchronized void initialize() throws SQLException { ... }

    /** 获取 Repository 实例。 */
    public ConfigRepository configs() { ... }
    public ToolLogRepository toolLogs() { ... }
    public MemoryMetaRepository memoryMeta() { ... }
    public EventLogRepository eventLogs() { ... }

    /** 关闭数据库连接。 */
    @Override
    public synchronized void close() { ... }

    /** 检查数据库是否已初始化。 */
    public boolean isInitialized() { ... }

    // ── 内部方法 ──

    private void openConnection() throws SQLException { ... }
    private void enableWALMode() throws SQLException { ... }
    private void migrate() throws SQLException { ... }
    private void createTables() throws SQLException { ... }
}
```

**关键设计点：**

1. **WAL 模式**：`PRAGMA journal_mode=WAL;` 允许并发读取，提升性能
2. **同步模式**：`PRAGMA synchronous=NORMAL;` 在性能和安全之间取得平衡
3. **外键**：`PRAGMA foreign_keys=ON;` 开启外键约束
4. **忙等待**：`PRAGMA busy_timeout=5000;` 设置忙等待超时 5 秒
5. **延迟初始化**：`initialize()` 在首次使用时调用，不在构造函数中打开连接

## 第5章 与 WorldContext 的集成

### 5.1 生命周期集成

`DatabaseManager` 的生命周期与 `WorldContext` 绑定，由 `WorldContext` 创建和管理。

```
WorldContext.initialize()
    │
    ├── 1. 确定数据库路径
    │       └── DatabaseManager.resolvePath(server, identity)
    │
    ├── 2. 创建 DatabaseManager
    │       └── new DatabaseManager(dbPath, worldName, instanceId)
    │
    ├── 3. 初始化数据库
    │       └── databaseManager.initialize()
    │
    └── 4. 后续使用
            └── databaseManager.configs().get("agent_core.host")
            └── databaseManager.toolLogs().insert(entry)
            └── databaseManager.eventLogs().insert(entry)

WorldContext.shutdown()
    │
    └── databaseManager.close()
```

### 5.2 WorldContext 的变更

需要在 `WorldContext` 中增加 `DatabaseManager` 字段：

```java
public class WorldContext implements AutoCloseable {
    // ... 现有字段 ...

    /** 数据库管理器（V11 新增）。 */
    private final DatabaseManager databaseManager;

    public WorldContext(MinecraftServer server, WorldIdentity identity) {
        // ... 现有初始化 ...
        this.databaseManager = createDatabaseManager(server, identity);
    }

    public void initialize() {
        // ... 现有初始化 ...
        try {
            databaseManager.initialize();
        } catch (SQLException e) {
            LOG.error("Failed to initialize database for world '{}'",
                    identity.worldName(), e);
        }
    }

    public void shutdown() {
        // ... 现有 shutdown ...
        databaseManager.close();
    }

    private DatabaseManager createDatabaseManager(MinecraftServer server,
                                                   WorldIdentity identity) {
```

```java
private static Path resolveDbPath(MinecraftServer server, WorldIdentity identity) {
    Path baseDir;

    if (server.isDedicatedServer()) {
        // 专用服务器
        baseDir = server.getServerDirectory()
                .toPath()
                .resolve("config/mcagent/worlds/dedicated");
    } else {
        // 集成服务器（单人模式）
        String worldDirName = server.getWorldData().getLevelName();
        baseDir = server.getServerDirectory()
                .toPath()
                .resolve("saves")
                .resolve(sanitizeWorldName(worldDirName))
                .resolve("config/mcagent/worlds")
                .resolve(sanitizeWorldName(worldDirName));
    }

    return baseDir.resolve("mcagent.db");
}
```

### 6.1 PRAGMA 配置

```sql
-- 在 initialize() 中执行

PRAGMA synchronous = NORMAL;         -- 平衡性能与安全性
PRAGMA foreign_keys = ON;            -- 启用外键约束
PRAGMA busy_timeout = 5000;          -- 忙等待超时 5 秒
PRAGMA cache_size = -8000;           -- 缓存 8MB
PRAGMA temp_store = MEMORY;          -- 临时表存储在内存
PRAGMA mmap_size = 268435456;        -- 内存映射大小 256MB
```

### 6.2 性能说明

| 维度 | 说明 |
|------|------|
| **写入性能** | WAL 模式下，单次 INSERT 约 0.1-0.5ms，能满足每 2s 状态上报 + 工具调用的写入频率 |
| **读取性能** | 索引覆盖的查询 < 1ms |
| **数据库大小** | 预计每月每世界 < 10MB（含日志和记忆数据） |
| **并发** | 单连接串行化访问，通过 `synchronized` 保证线程安全 |

### 6.3 日志清理策略

## 附录B：DatabaseManager 关键实现细节

### 初始化流程

```java
public synchronized void initialize() throws SQLException {
    if (connection != null && !connection.isClosed()) {
        return; // 已初始化
    }

    // 1. 确保目录存在
    Files.createDirectories(dbPath.getParent());

    // 2. 打开连接
    this.connection = DriverManager.getConnection(JDBC_PREFIX + dbPath.toAbsolutePath());

    // 3. 配置 PRAGMA
    try (Statement stmt = connection.createStatement()) {
        stmt.execute("PRAGMA journal_mode = WAL");
        stmt.execute("PRAGMA synchronous = NORMAL");
        stmt.execute("PRAGMA foreign_keys = ON");
        stmt.execute("PRAGMA busy_timeout = 5000");
        stmt.execute("PRAGMA cache_size = -8000");
        stmt.execute("PRAGMA temp_store = MEMORY");
        stmt.execute("PRAGMA mmap_size = 268435456");
    }

    // 4. 执行迁移
    migrate();

    LOG.info("Database initialized: {} (world='{}')", dbPath, worldName);
}
```

### 版本迁移

```java
private void migrate() throws SQLException {
    int currentVersion = getCurrentVersion();

    if (currentVersion == 0) {
        // 全新数据库，创建所有表
        createTables();
        try (PreparedStatement ps = connection.prepareStatement(
                "INSERT INTO schema_version (version, description) VALUES (?, ?)")) {
            ps.setInt(1, SCHEMA_VERSION);
            ps.setString(2, "Initial schema: config, tool_logs, memory_meta, event_logs");
            ps.execute();
        }
        LOG.info("Database schema initialized to version {}", SCHEMA_VERSION);
    } else if (currentVersion < SCHEMA_VERSION) {
        // 执行增量迁移（未来扩展）
        for (int v = currentVersion + 1; v <= SCHEMA_VERSION; v++) {
            migrateTo(v);
        }
    }
}
```
