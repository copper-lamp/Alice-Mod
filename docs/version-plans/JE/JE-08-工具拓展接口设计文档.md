## 第1章 概述

### 1.1 背景

Alice Mod JE 目前内置了 26 个工具，覆盖移动、背包、方块、战斗、生物交互、感知、对话等类别。随着生态发展，社区开发者需要以**附属模组（Addon Mod）**的方式向 Alice Mod 添加自定义工具，且这些工具应能获得与内建工具同等的运行时能力——访问假人、游戏世界、寻路系统、TCP 通信等基础设施。

### 1.2 术语表

| 术语 | 说明 |
|------|------|
| **附属模组（Addon Mod）** | 依赖 Alice Mod 的 Fabric 模组，通过工具拓展接口注册自定义工具 |
| **Alice Tool Plugin** | 附属模组实现的入口点接口，注册工具的契约 |
| **Alice Service Access** | 服务访问层，向附属模组暴露 Alice Mod 内部能力 |
| **Fabric Entrypoint** | Fabric Mod 提供的自定义入口点机制，用于模组间发现与协作 |

## 第2章 设计目标与原则

### 2.1 目标

| 目标 | 说明 |
|------|------|
| **对等能力** | 附属模组注册的工具与内建工具享有完全一致的运行时能力 |
| **零侵入** | Alice Mod 内核无需为某个特定附属模组做适配 |
| **声明式 & 编程式双模式** | 既支持通过注解声明工具（与内建方式一致），也支持编程式注册 |
| **版本兼容** | 接口版本化，保证 API 向后兼容 |
| **热友好** | 工具注册在模组初始化阶段完成，工具调用在运行时动态派发 |

### 2.2 设计原则

| 原则 | 说明 |
|------|------|
| **最小依赖** | API 接口包仅依赖 Java 标准库 + Gson，不绑定 Minecraft / Fabric 概念 |
| **接口隔离** | 服务访问按领域拆分（BotService / WorldService / ...），接口粒度以工具的实际需求为基准 |
| **契约明确** | 每个服务接口的方法签名、异常、线程安全语义必须文档化 |
| **渐进暴露** | 服务访问层的暴露范围由 Alice Mod 控制，不盲目暴露内部实现细节 |

## 第4章 核心接口设计

### 4.1 AliceTool 接口

沿用现有 `AliceTool` 接口，保持契约一致：

```java
package io.alice.mod.adapter.api;

import java.util.Map;

/**
 * Alice Mod 工具接口 —— 所有工具的通用契约。
 * 接口刻意不绑定任何 Minecraft 概念，保持通用性。
 */
public interface AliceTool {

    /**
     * 工具名称，snake_case，全局唯一。
     * 例如：move_to、mine_block、look_around
     */
    String name();

    /**
     * 工具描述，供 LLM 理解工具用途。
     */
    String description();

    /**
     * JSON Schema 格式的参数定义。
     * 遵循 OpenAI function calling parameters 方言。
     */
    Map<String, Object> parameterSchema();

    /**
     * 执行工具。
     *
     * @param args 参数键值对（已校验）
     * @return 工具执行结果
     */
    ToolResult invoke(Map<String, Object> args);
}
```

**设计要点**：
- 保持纯 Java 接口，不引用 Minecraft / Fabric 或 Alice Mod 内部类
- 附属模组可以在编译期依赖 `AliceTool`，运行时由 Alice Mod 提供

### 4.2 ToolResult

沿用现有 `ToolResult` 记录类，保持协议兼容：

```java
package io.alice.mod.adapter.api;

import java.util.Map;

/**
 * 工具执行结果信封。
 * 格式符合 00-工具协议规范 ResultEnvelope 标准。
 */
public record ToolResult(
    boolean success,
    String message,
    Map<String, Object> data,
    Map<String, Object> meta,
    String errorCode,
    String errorMessage,
    Map<String, Object> errorDetails
) {
    // 工厂方法：ok() / fail() — 与现有代码一致
}
```

### 4.3 ToolRegistrar

注册器——附属模组通过它注册工具，它只暴露注册操作，封装注册表内部实现：

```java
package io.alice.mod.adapter.api;

/**
 * 工具注册器。
 * 附属模组在初始化阶段通过此接口向 Alice Mod 注册自定义工具。
 * 注册器由 Alice Mod 实现并注入给 {@link AliceToolPlugin}。
 */
public interface ToolRegistrar {

    /**
     * 注册一个工具。
     *
     * @param tool 工具实例
     * @throws IllegalArgumentException 如果工具名为 null 或空
     * @throws IllegalStateException    如果同名工具已存在
     */
    void register(AliceTool tool);

    /**
     * 批量注册工具（原子操作：要么全部注册成功，要么全部不回滚）。
     *
     * @param tools 工具实例列表
     * @throws IllegalArgumentException 如果列表中有 null 或无效参数
     * @throws IllegalStateException    如果存在同名工具冲突
     */
    void registerAll(Collection<AliceTool> tools);
}
```

**为什么不用 `ToolRegistry` 本身暴露给附属模组？**
- `ToolRegistry` 是全局静态单例，直接暴露会鼓励附属模组在运行时任意调用注册/清除方法
- `ToolRegistrar` 是一个受限视图，只在插件初始化阶段可用，之后即失效
- `ToolRegistrar` 可以在内部做更多校验（名称格式、Schema 合法性等）

### 4.4 AliceToolPlugin —— 插件入口点

```java
package io.alice.mod.adapter.api;

/**
 * Alice Mod 工具插件入口点。
 * <p>
 * 附属模组通过实现此接口，在 Fabric 自定义 entrypoint "alice-mod:plugin"
 * 中被 Alice Mod 发现并调用。
 * <p>
 * 实现要求：
 * <ul>
 *   <li>必须有一个无参构造方法</li>
 *   <li>不应在构造方法中执行耗时操作</li>
 *   <li>所有工具注册行为应在 {@link #registerTools} 中完成</li>
 * </ul>
 */
@FunctionalInterface
public interface AliceToolPlugin {

    /**
     * 在 Alice Mod 初始化时被调用，用于注册自定义工具。
     * <p>
     * 此阶段 Minecraft Server **尚未启动**，工具不可在此方法中
     * 访问游戏世界或假人——只能进行注册。
     *
     * @param registrar 注册器
     * @param services  服务访问入口（可用于后续获取服务实例）
     */
    void registerTools(ToolRegistrar registrar, AliceServiceAccess services);
}
```

**生命周期**：
```
Fabric Loader 启动
    │
    ├── 加载 Alice Mod
    │   │
    │   ├── AliceModAdapter.onInitialize()
    │   │   ├── 初始化 BotAccess
    │   │   ├── 扫描内建工具 → 注册到 ToolRegistry
    │   │   ├── 发现附属模组插件 → 调用 AliceToolPlugin.registerTools()
    │   │   ├── (此时 ToolRegistry 包含内建 + 附属的全部工具)
    │   │   └── 启动 TCP 客户端 → 发送 register_tools（含全部工具）
    │   │
    │   └── Alice Mod 初始化完成
    │
    ├── 加载附属模组 A
    │   └── 附属模组 A 的 ModInitializer.onInitialize()
    │       └── （可选：附属模组可以自定义配置、注册自己的 Fabric 内容等）
    │
    └── Minecraft Server 启动
        └── （此时 Alice Mod 是唯一注册工具的场所，工具已全部就绪）
```

## 第5章 服务访问层

### 5.1 设计思路

工具在执行时通常需要：
- 获取假人实例 → `BotService`
- 查询/操作世界状态 → `WorldService`
- 寻路 → `PathfindingService`
- 订阅/发布事件 → `EventService`
- 发送数据到 Agent Core → `TcpService`
- 查询玩家状态 → `PlayerService`
- 操作背包 → `InventoryService`
- 放置/挖掘方块 → `BlockService`
- 与实体交互 → `EntityService`

这些能力通过 `AliceServiceAccess` 统一暴露给附属模组。

### 5.2 AliceServiceAccess

```java
package io.alice.mod.adapter.api.service;

/**
 * Alice Mod 服务访问入口。
 * <p>
 * 附属模组通过此接口获取 Alice Mod 的各类服务。
 * 每个服务接口的方法都必须是线程安全的（Alice Mod 保证）。
 */
public interface AliceServiceAccess {

    /** 假人管理服务（假人创建/销毁/查询）。 */
    BotService botService();

    /** 游戏世界服务（方块查询/设置、实体查询）。 */
    WorldService worldService();

    /** 寻路服务（路径计算）。 */
    PathfindingService pathfindingService();

    /** 玩家状态服务（血量/饥饿/经验等）。 */
    PlayerService playerService();

    /** 背包操作服务。 */
    InventoryService inventoryService();

    /** 事件订阅服务。 */
    EventService eventService();

    /** 方块操作服务。 */
    BlockService blockService();

    /** 实体交互服务。 */
    EntityService entityService();

    /** TCP 通信服务。 */
    TcpService tcpService();
}
```

### 5.3 各服务接口

#### BotService

```java
package io.alice.mod.adapter.api.service;

import java.util.List;
import java.util.Optional;
import java.util.UUID;

/**
 * 假人管理服务。
 * 提供假人的生命周期管理和状态查询。
 */
public interface BotService {

    /** 创建并生成假人。同名假人幂等——已在线返回在线实例，已注册但离线则唤醒。 */
    BotHandle spawn(String name, Vec3 position, String dimension);

    /** 休眠假人（保存存档后移除，不删除注册信息）。 */
    boolean despawn(String nameOrUuid);

    /** 永久销毁假人（下线并删除注册信息）。 */
    boolean dismiss(String nameOrUuid);

    /** 根据 UUID 获取在线假人。 */
    Optional<BotHandle> get(UUID uuid);

    /** 根据名称获取在线假人。 */
    Optional<BotHandle> findByName(String name);

    /** 获取所有在线假人。 */
    List<BotHandle> getAllOnline();

    /** 获取所有已注册假人信息（在线 + 离线）。 */
    List<BotInfo> listAll();

    /** 是否为假人。 */
    boolean isBot(UUID uuid);
}

/** 假人操作句柄。 */
public interface BotHandle {
    UUID uuid();
    String name();
    Vec3 position();
    ResourceLocation dimension();
    float health();
    float maxHealth();
    int foodLevel();
    int experienceLevel();
    void teleport(double x, double y, double z, String dimension);
    /** 获取原生的 Minecraft ServerPlayer 实例（需要了解 Minecraft API 的开发者使用）。 */
    <T> T getNativePlayer();
}
```

**设计要点**：
- `BotHandle` 是假人的抽象句柄，不暴露 `EntityPlayerMPFake` 类型，避免附属模组依赖 Carpet Mod
- `getNativePlayer()` 提供逃生舱（escape hatch），允许高阶开发者直接操作 Minecraft 原生类型
- `Vec3` / `ResourceLocation` 是 API 包中定义的简化值类型，不绑定 Minecraft 类

### 5.4 服务获取方式

附属模组在 `AliceToolPlugin.registerTools()` 阶段获得 `AliceServiceAccess` 实例，可以：

1. 在插件初始化阶段**保存引用**，供后续工具执行时使用
2. 在工具 `invoke()` 方法中通过保存的引用获取具体服务

```java
public class MyToolPlugin implements AliceToolPlugin {
    
    private AliceServiceAccess services;

    @Override
    public void registerTools(ToolRegistrar registrar, AliceServiceAccess services) {
        this.services = services;
        registrar.register(new MyCustomTool(services));
    }
}
```

## 第6章 服务接口实现

### 6.1 实现方式

`AliceServiceAccess` 的实现类 `ServiceAccessImpl` 位于 `io.alice.mod.adapter.tool` 包中，它**封装 Alice Mod 内部模块的调用**：

```java
// 内部实现，不对外暴露
package io.alice.mod.adapter.tool;

public class ServiceAccessImpl implements AliceServiceAccess {
    
    // 封装内部模块引用
    private final BotService botService = new BotServiceImpl();
    private final WorldService worldService = new WorldServiceImpl();
    private final PathfindingService pathfindingService = new PathfindingServiceImpl();
    // ...

    @Override
    public BotService botService() { return botService; }

    @Override
    public WorldService worldService() { return worldService; }
    // ...
}

// BotServiceImpl 封装 BotManager 的调用
class BotServiceImpl implements BotService {
    @Override
```

### 6.2 版本适配

所有版本敏感的 API 调用在服务实现层通过 `compat` 适配层处理，与内建工具使用相同的适配策略：

```
附属模组的工具
    ↓ 调用服务接口（版本无关）
ServiceAccessImpl
    ↓ 调用功能模块（版本相关）
BotManager / AStarPathfinder / ... 
    ↓ 版本敏感调用
compat 适配层（版本隔离）
    ↓
Minecraft 原生 API
```

## 第7章 插件发现与注册流程

### 7.1 发现机制

使用 Fabric 的入口点（Entrypoint）API 发现附属模组：

```java
// ToolPluginDiscoverer.java（内部实现）
public final class ToolPluginDiscoverer {

    public static List<AliceToolPlugin> discover() {
        List<AliceToolPlugin> plugins = new ArrayList<>();
        
        // 使用 Fabric Loader API 获取所有 "alice-mod:plugin" 入口点
        for (AliceToolPlugin plugin : FabricLoader.getInstance()
                .getEntrypoints("alice-mod:plugin", AliceToolPlugin.class)) {
            plugins.add(plugin);
        }
        
        LOG.info("Discovered {} Alice Tool Plugin(s)", plugins.size());
        return plugins;
    }
}
```

### 7.2 完整注册流程

```
AliceModAdapter.onInitialize()
    │
    ├── 1. 初始化基础设施（BotAccess、事件分发器等）
    │
    ├── 2. 扫描并注册内建工具
    │   └── ToolScanner.scanAndRegister("io.alice.mod.adapter.tool")
    │
    ├── 3. 发现附属模组插件
    │   └── List<AliceToolPlugin> plugins = ToolPluginDiscoverer.discover()
    │
    ├── 4. 为每个插件调用 registerTools()
    │   ├── registrar = new ToolRegistrarImpl(ToolRegistry.getInstance())
    │   ├── services = new ServiceAccessImpl()
    │   └── plugin.registerTools(registrar, services)
    │
    ├── 5. TCP 连接建立后，发送完整的工具列表
    │   └── 内建工具 + 插件注册工具 → 统一 register_tools 消息
    │
    └── 6. 进入待命状态，等待 tool_call
```

### 7.3 注册阶段限制

在 `AliceToolPlugin.registerTools()` 阶段：

| 可做 | 不可做 |
|------|--------|
| 注册工具 | 访问游戏世界 |
| 保存 `AliceServiceAccess` 引用 | 操作假人 |
| 读取配置 | 发送 TCP 消息 |
| 初始化工具所需的数据结构 | 依赖 Minecraft Server（此时可能未启动） |

> 之所以限制此时不能访问游戏世界，是因为 Fabric 模组初始化发生在 Server 启动之前——这与内建工具的限制一致。工具可以**在 invoke() 方法中使用服务**，届时 Server 一定已启动。

## 第8章 附属模组开发指南

### 8.1 最小依赖

在 `build.gradle` 中：

### 8.2 最小实现

**方式一：编程式注册**（推荐，更灵活）

```java
package com.example;

import io.alice.mod.adapter.api.*;
import io.alice.mod.adapter.api.service.*;

import java.util.Map;

public class MyToolPlugin implements AliceToolPlugin {

    @Override
    public void registerTools(ToolRegistrar registrar, AliceServiceAccess services) {
        registrar.register(new MyCustomTool(services));
    }
}

class MyCustomTool implements AliceTool {

    private final BotService botService;

    MyCustomTool(AliceServiceAccess services) {
        this.botService = services.botService();
    }

    @Override
    public String name() { return "my_custom_action"; }

    @Override
    public String description() { return "执行自定义操作"; }

    @Override
    public Map<String, Object> parameterSchema() {
        return Map.of(
            "type", "object",
            "properties", Map.of("target", Map.of(
                "type", "string",
                "description", "目标名称"
            )),
            "required", new String[]{"target"}
        );
    }

    @Override
    public ToolResult invoke(Map<String, Object> args) {
        String target = (String) args.get("target");
        // 使用 botService 操作假人...
        return ToolResult.ok("操作完成");
    }
}
```

**方式二：注解声明式注册**

如果附属模组将 Alice Mod API jar 作为编译期依赖，也可以使用与内建工具相同的注解方式——但需要将 `@ToolModule` / `@ToolMethod` 注解迁移到 API 包中：

```java
package com.example;

import io.alice.mod.adapter.api.*;
import io.alice.mod.adapter.api.annotation.*;  // 注解移动到 API 包

@ToolModule(category = "custom", description = "自定义工具模块")
public class CustomTools {

    @ToolMethod(
        name = "custom_greet",
        description = "打招呼",
        parameters = {
            @ToolParam(name = "name", type = "string", description = "名字")
        }
    )
    public ToolResult greet(Map<String, Object> args) {
        String name = (String) args.get("name");
        return ToolResult.ok("Hello, " + name + "!");
    }
}
```

**附属模组的 `fabric.mod.json`**：

```json
{
  "schemaVersion": 1,
  "id": "my-custom-tools",
  "version": "1.0.0",
  "name": "My Custom Tools",
  "description": "Alice Mod 的自定义工具附属模组",
  "entrypoints": {
    "alice-mod:plugin": ["com.example.MyToolPlugin"],
    "main": ["com.example.MyAddonInitializer"]
  },
  "depends": {
    "alice-mod": ">=1.0.0",
    "fabricloader": ">=0.16.0",
    "minecraft": ">=1.21",
    "java": ">=21"
  }
}
```

### 8.3 工具命名规范

为避免附属模组之间的工具名冲突，建议：

| 规则 | 示例 |
|------|------|
| 前缀化：`<mod_id>_<tool_name>` | `my_mod_fly`、`my_mod_scan` |
| 在 description 中注明模组来源 | `"(来自 My Mod)"` |

Alice Mod 的 `ToolRegistrar` 不会修改工具名，但会在日志中记录注册来源。

## 第10章 安全与隔离

### 10.1 注册阶段安全

- `ToolRegistrar` 实例仅对每个插件提供一次，在 `registerTools()` 调用结束后失效
- 同一工具名多次注册 → `IllegalStateException`（Alice Mod 内建工具名优先保留，冲突时警告日志+拒绝覆盖）
- 插件注册工具时抛出未捕获异常 → 该插件注册中断，不影响其他插件

### 10.2 运行时安全

- 每个工具执行在 `invoke()` 的 try-catch 中包裹（与内建工具一致）
- 服务接口的所有方法都要求是线程安全的
- 服务接口实现内部的异常不会传播到附属模组之外

### 10.3 名称冲突策略

```
冲突检测顺序：
1. Alice Mod 内建工具先注册
2. 附属模组插件按 Fabric Loader 发现顺序依次注册
3. 如果插件尝试注册与已存在同名的工具 → 跳过并打印警告
4. Agent Core 收到的工具列表以最终注册状态为准
```
