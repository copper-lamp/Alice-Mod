## 第1章 总体架构

### 1.1 测试拓扑

本期仅关注 **AC ↔ JE** 两进程，测试套件包含 **2 个测试层次 + 1 个回归层**：

```
┌────────────────────────────────────────────────────────────────────────┐
│                  IT — 集成测试套件（JE 专属）                            │
│                                                                        │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │  回归层：B1/B2/B3/B4 Bug 回归测试（vitest unit）                  │  │
│  │  ├─ B1: AC ToolDispatcher.callTool method = 'tool_call'          │  │
│  │  ├─ B2: JE WorldContext.registerTools 用 sendNotification        │  │
│  │  ├─ B3: JE WorldContext.handleToolCall 正确解析参数               │  │
│  │  └─ B4: AC normalizeToolSchema 兼容 input_schema                 │  │
│  │  → 跑 vitest，与 PR 门禁挂钩，必须全绿                            │  │
│  └──────────────────────────────────────────────────────────────────┘  │
│                              ▼                                          │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │  Level 1: 协议契约测试（mock JE Client + 真实 AC，< 5s）         │  │

│  └──────────────────────────────────────────────────────────────────┘  │

│  │  ├─ 真实 AC（in-process 模式）                                    │  │

└────────────────────────────────────────────────────────────────────────┘
```

**为什么需要回归层？**

4 个 bug 都在主代码路径中，且都是"如果不主动写测试，未来重构极易再次回归"。回归层作为**契约基线**：

- 任何对 `ToolDispatcher` / `WorldContext` / `SchemaGenerator` / `normalizeToolSchema` 的修改都必须先让对应回归测试失败再通过
- 跑得极快（毫秒级），可作为 PR 门禁

```typescript
describe('B1: ToolDispatcher.callTool 必须发 method=tool_call', () => {
```

### 3.2 组件

#### 3.2.1 MockJeClient

- 使用 `packages/agent-core/src/main/tcp/json-rpc.ts` 的真实 JsonRpcCodec
- 启动 net.Socket 连接到 AC 的 TcpServer
- 模拟：握手 → register_tools Notification → 接收 tool_call Request → 应答
- 与真实 JE 唯一的差异是：**不**读 `entity_player_mp_fake`，仅做协议层

```typescript
// packages/agent-core/__tests__/it/fixtures/mock-je-client.ts
import { createConnection } from 'node:net'
import { JsonRpcCodec } from '../../../../src/main/tcp/json-rpc'

export class MockJeClient {
  private socket: any
  private buffer = ''
  private onRequest: ((req: any) => Promise<any>) | null = null
  public sentRequests: any[] = []
  public sentNotifications: any[] = []
  public receivedRequests: any[] = []

  constructor(private host: string, private port: number) {}

  async connect(): Promise<void> {
    return new Promise((resolve, reject) => {
      this.socket = createConnection(this.port, this.host, () => resolve())
      this.socket.on('data', (d: Buffer) => this.handleData(d))
      this.socket.on('error', reject)
    })
  }

  async handshake(params: any): Promise<any> {
    const resp = await this.sendRequest('handshake', params)
    return resp
  }

  sendRegisterTools(tools: any[]): void {
    this.sendNotification('register_tools', { tools })
  }

  private sendRequest(method: string, params: any): Promise<any> {
    const id = ++this.requestId
    const req = { jsonrpc: '2.0', id, method, params }
    this.sentRequests.push(req)
    return new Promise((resolve) => {
      this.pendingRequests.set(id, resolve)
      this.socket.write(JsonRpcCodec.encodeRequest(req) + '\n')
    })
  }

  private sendNotification(method: string, params: any): void {
    const notif = { jsonrpc: '2.0', method, params }
    this.sentNotifications.push(notif)
    this.socket.write(JsonRpcCodec.encodeNotification(notif) + '\n')
  }

  private handleData(data: Buffer): void {
    this.buffer += data.toString()
    const lines = this.buffer.split('\n')
    this.buffer = lines.pop()!
    for (const line of lines) {
      if (!line.trim()) continue
      const msg = JSON.parse(line)
      if (msg.id && msg.method) {
        // 收到 Request（AC → JE）
        this.receivedRequests.push(msg)
        this.handleIncomingRequest(msg)
      } else if (msg.id && (msg.result !== undefined || msg.error !== undefined)) {
        // 收到 Response
        const resolve = this.pendingRequests.get(msg.id)
        resolve?.(msg)
        this.pendingRequests.delete(msg.id)
      }
    }
  }

  private async handleIncomingRequest(req: any): Promise<void> {
    if (this.onRequest) {
      const response = await this.onRequest(req)
      this.socket.write(JsonRpcCodec.encodeResponse({ id: req.id, result: response }) + '\n')
    }
  }

  setOnRequestHandler(handler: (req: any) => Promise<any>): void {
    this.onRequest = handler
  }

  disconnect(): void {
    this.socket?.destroy()
  }
}
```

## 第6章 关键技术决策

### 6.1 回归层独立于 IT

- 原因：回归测试是契约基线，应在 PR 门禁时跑（每次提交），不能依赖 MC 服务器
- 实现：放在 `__tests__/pipeline/`、`__tests__/tcp/` 路径下，与单元测试同 runner
- 与 IT 的关系：回归测试必须先于 IT 通过（PR 阶段）；IT 验证完整业务流（nightly / release）

### 6.2 归一化放在 AC 端而非 JE 端

- 原因：协议消费方应做容错，发送方应该被强制遵循协议（但允许 B4 旧代码继续运行）
- 折中：JE 端 SchemaGenerator 暂不改动（避免大范围修改），但新增 warning 日志引导未来修复
- 实际：B4 修复采用"消费方归一化"策略

### 6.3 MockJeClient 而非 Mock AC

- 原因：测试 AC 处理 JE 端消息的正确性，AC 是被测对象
- 实现：MockJeClient 是 TcpClient 协议层的等价物，但**不**读 `entity_player_mp_fake`
- 优势：可验证归一化逻辑、提示词组装

### 6.4 AC In-Process 启动

- 原因：避免子进程管理与 IPC 复杂度
- 实现：通过 `vitest globalSetup` 调用 `initializeServices()` 但跳过 `createWindow()`
- 风险：与 Electron 主进程略有差异（如 `app.getPath('userData')`）
- 缓解：在 IT 模式下使用 `process.env.IT_USER_DATA` 注入临时目录

### 6.5 MC 服务器预生成 World

- 原因：减少世界生成时间从 30s 到 5s
- 实现：将最小化世界（5×5 平台 + 1 个假人预设坐标）打包到 `packages/adapter-java/src/test/resources/world-preset/`

### 6.6 测试隔离

- 端口隔离：AC 启动在 `27541`，MC 在 `25575`，IT 全局用 `IT_*` 前缀
- 数据库隔离：使用 `process.env.IT_USER_DATA/alice-mod.db`
- 工作目录隔离：MC 服务器拷贝到 `os.tmpdir()/it-je-{uuid}/`

### 6.7 时间预算

| 阶段 | 时间 |
|------|------|
| 回归层（vitest unit） | < 5s |
| L1 协议契约 | < 5s |
| L2 MC 启动 | 30-60s |
| L2 用例执行 | 30-60s |
| L2 收尾 | 5-10s |
| **总时长** | **1-3 分钟** |

## 第8章 风险与技术债

| 项 | 描述 | 缓解 |
|----|------|------|
| TechDebt-1 | MC 服务器启动慢导致 CI 反馈周期长 | 回归层 + L1 < 10s 跑完，PR 门禁只跑这两层；L2 nightly |
| TechDebt-2 | WorldPreset 与 Carpet 版本强绑定 | 版本固定在 `gradle.properties`，升级需重新生成 preset |
| TechDebt-3 | In-process AC 模式与 Electron 主进程存在差异 | 主要差异在 `app.getPath`，IT 模式用 `process.env.IT_USER_DATA` 替代 |
| TechDebt-4 | JE 端无成熟 JUnit + AssertJ 习惯，断言可读性差 | 使用 AssertJ 的 `assertThat(...).as("...")` 提供清晰失败信息 |
| TechDebt-5 | 失败时 MC 服务器日志与 AC 日志对齐困难 | 统一时间戳为相对毫秒（`ac_t0` / `mc_t0`），日志都加前缀 |
| TechDebt-6 | normalizeToolSchema 掩盖 B4 协议不一致 | 归一化时记录 warning 日志，引导 JE 端未来修复 |

## 第9章 总结

集成测试架构的核心是 **Bug 修复 + 两层真实度**：

- **回归层**（< 5s）：固化 4 个 Bug 修复，PR 门禁基线
- **L1 协议契约**（< 5s）：mock JE 客户端 + 真实 AC，验证消息结构
- **L2 端到端**（1-3 分钟）：真实 JE 模组 + 真实 MC 服务器 + 真实假人

三层覆盖 **契约 → 协议 → 执行** 完整链路，BE 端不在范围，专注 AC↔JE。
