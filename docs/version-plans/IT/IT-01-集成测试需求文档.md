## 第1章 概述

### 1.1 背景

Alice Mod 由两个相互独立部署的进程组成（**不再涉及 BE**）：

| 进程 | 技术栈 | 角色 | 单元测试覆盖 |
|------|--------|------|------------|
| **Agent Core（AC）** | Electron + TypeScript | LLM 编排、提示词组装、工作区管理、TCP 服务端 | 已覆盖（57 个测试文件） |
| **Adapter Java Edition（JE）** | Java + Fabric | Minecraft Java 版模组、工具执行、假人管理 | 仅覆盖参数校验等纯逻辑 |

两进程通过 TCP + JSON-RPC 2.0 协议协作，但**目前缺少真实端到端集成测试**，且在静态代码审计中发现 **4 个**会阻断正常业务闭环的关键 bug：

| Bug | 影响 | 严重度 |
|-----|------|--------|
| **B1** AC `ToolDispatcher.callTool` 使用 `method: 'call_tool'` | JE 端 `METHOD_TOOL_CALL = "tool_call"`，所有 AC→JE 工具调用返回 `Method Not Found` | **阻断** |
| **B2** JE `WorldContext.registerTools` 用 `sendRequest` 发注册 | AC 端只处理 `Notification`，register_tools 永远不触发 `registerTools` | **阻断** |
| **B3** JE `WorldContext.handleToolCall` 三元表达式两支相同 | 所有工具调用参数被丢弃为 `Map.of()`，仅 `null` 时正常 | **阻断** |
| **B4** JE `SchemaGenerator` 发 `input_schema`，AC `ToolSchema` 期望 `parameters` | 工具 Schema 字段丢失，prompt 中无参数定义 | **高** |

### 1.2 目标

设计一套**端到端集成测试套件（IT）**，专注于 **AC ↔ JE** 链路：

1. 修复上述 4 个 bug 并通过单元测试固化修复点
2. 验证 JE → AC 工具注册（`register_tools` Notification 路径）
3. 验证 AC 正确组装提示词（包含完整工具说明区）
4. 验证 AC → JE 工具调用闭环（`tool_call` Request 路径 + 参数正确透传）
5. 验证工具在**真实 Minecraft Java 版服务器**中由**真实假人**执行
6. 协议契约测试：在不启动 MC 的前提下用 mock 客户端验证 JE 消息格式

### 1.3 术语表

| 术语 | 含义 |
|------|------|
| **真实服务器** | 启动真实的 Minecraft Java 版服务器（带 Carpet Mod 与 Alice Mod） |
| **真实假人** | 通过 Carpet Mod 的 `EntityPlayerMPFake` 实际生成的 `ServerPlayer` 实体 |
| **Mock JE Client** | TypeScript 实现的轻量 TcpClient，模拟 JE 端握手、注册、调用协议 |
| **In-process AC** | 不启动 Electron 窗口，仅初始化 `WorkspaceManager` / `TcpServer` / `PromptBuilder` / `ToolDispatcher` |
| **IT** | Integration Test，本文档主题 |

## 第2章 已修复 Bug 清单

### 2.1 B1 — AC `ToolDispatcher.callTool` 方法名错

| 项 | 内容 |
|----|------|
| 文件 | [packages/agent-core/src/main/pipeline/tool-dispatcher.ts:179](file:///D:/McAgent/packages/agent-core/src/main/pipeline/tool-dispatcher.ts#L175-L187) |
| 原因 | 历史误用 action 类型名 `call_tool` 作为 JSON-RPC method |
| 影响 | 任何 `ToolDispatcher.callTool()` 调用都返回 `Method Not Found (-32601)` |
| 修复 | 改为 `'tool_call'`，与 JE 端 `METHOD_TOOL_CALL` 和 `batch-scheduler.buildBatchCalls` 对齐 |
| 回归测试 | `__tests__/pipeline/tool-dispatcher-method-name.test.ts` |

### 2.2 B2 — JE `WorldContext.registerTools` 用错消息类型

| 项 | 内容 |
|----|------|
| 文件 | [packages/adapter-java/.../world/WorldContext.java:242-258](file:///D:/McAgent/packages/adapter-java/src/main/java/io/alice/mod/adapter/world/WorldContext.java#L242-L258) |
| 原因 | `register_tools` 协议定义是 Notification（无 id 无需响应），但代码用 `sendRequest` |
| 影响 | AC 端不会进入 `handleTcpNotification` 的 `register_tools` 分支，`workspace.toolRegistry` 始终为空 |
| 修复 | 改为 `tcpClient.sendNotification("register_tools", payload)` |
| 回归测试 | `BotToolsProtocolE2ETest.testRegisterToolsIsNotification()` |

### 2.3 B3 — JE `WorldContext.handleToolCall` 参数解析逻辑错

| 项 | 内容 |
|----|------|
| 文件 | [WorldContext.java:265-278](file:///D:/McAgent/packages/adapter-java/src/main/java/io/alice/mod/adapter/world/WorldContext.java#L265-L278) |
| 原因 | 三元表达式两支都返回 `Map.of()`，原意为"有参数则解析"但解析逻辑缺失 |
| 影响 | 所有工具调用接收到的 `args` 都是空 Map，工具只能读到 null 字段 |
| 修复 | 新增 `parseJsonArgs(JsonElement)` 工具方法，用 Gson 反序列化为 `Map<String, Object>` |
| 回归测试 | `BotToolsProtocolE2ETest.testHandleToolCallParsesParameters()` |

### 2.4 B4 — JE SchemaGenerator 字段名与 AC ToolSchema 契约不一致

| 项 | 内容 |
|----|------|
| 文件 | [SchemaGenerator.java:32-57](file:///D:/McAgent/packages/adapter-java/src/main/java/io/alice/mod/adapter/tool/SchemaGenerator.java#L32-L57) |
| 原因 | JE 输出 `{name, description, category, input_schema, output_schema, execution}`；AC `ToolSchema` 契约要求 `{name, description, parameters, category, enabled}` |
| 影响 | `ToolPromptAssembler.schemaToPromptDef` 读到 `schema.parameters = undefined`，prompt 中**无参数说明** |
| 修复 | 在 AC 端 `handleTcpNotification` 接收 `register_tools` 时做归一化：识别 `input_schema` / `parameters` / `function.parameters` 三种来源，转换为标准 `ToolSchema` |
| 回归测试 | `__tests__/tcp/register-tools-normalize.test.ts` |

### 2.5 归一化协议兼容层

新增 [`normalizeToolSchema`](file:///D:/McAgent/packages/agent-core/src/main/index.ts#L557-L591) 函数作为协议消费端的容错层：

- **输入**：JE 端发 `{name, description, category, input_schema, ...}` 或 BE 端发 `{name, description, parameters, ...}`
- **输出**：统一 `ToolSchema = {name, description, parameters, category, enabled}`
- **附加**：从工具名前缀推断 category（`bot_*` → entity，`move_*` → movement 等）

## 第3章 需求清单

### 3.1 功能性需求

| 编号 | 需求 | 优先级 | 对应 Bug |
|------|------|--------|----------|
| **FN-1** | AC 端 `ToolDispatcher.callTool` 发出 method 为 `tool_call` 的 JSON-RPC Request | P0 | B1 |
| **FN-2** | JE 端在 5s 内响应 `tool_call` Request，返回 `success=true` | P0 | B1 |
| **FN-3** | JE 端在握手成功后通过 `register_tools` **Notification** 通知 AC | P0 | B2 |
| **FN-4** | AC 端在 1s 内完成 `workspaceManager.registerTools`，且工具数量 = JE 注册数 | P0 | B2 |
| **FN-5** | JE 端 `handleToolCall` 正确解析 `params.parameters` 为 `Map<String, Object>` | P0 | B3 |
| **FN-6** | AC 端接收的 ToolSchema 包含 `parameters` 字段（非空），prompt 中每个工具有参数说明 | P0 | B4 |
| **FN-7** | `bot_spawn` 工具在真实 MC 中生成假人，假人可被 `bot_info` 查询 | P0 | — |
| **FN-8** | `bot_move_to` 工具使假人位置变化 | P0 | — |
| **FN-9** | `look_around` 工具返回周围环境信息 | P1 | — |
| **FN-10** | 错误路径：参数缺失 / 假人不存在 / 坐标越界时 AC 收到正确错误码 | P1 | — |

### 3.2 非功能性需求

| 编号 | 需求 | 说明 |
|------|------|------|
| **NF-1** | JE↔AC 集成测试单次运行时间 < 5 分钟 | MC 服务器启动占主要耗时；通过 WorldPreset + JVM `-noverify` 优化 |
| **NF-2** | 协议契约单元测试时间 < 5s | Mock JE Client 即可 |
| **NF-3** | 失败用例产出诊断报告 | 包含：握手日志、注册消息、提示词快照、TCP 消息序列、MC 服务器日志 |
| **NF-4** | 测试用例之间不共享全局状态 | 通过 fresh `WorkspaceManager` + fresh MC world 隔离 |
| **NF-5** | 协议契约测试不依赖 Minecraft | 纯进程内，可 PR 门禁 |
| **NF-6** | 修复 Bug 的回归测试作为契约基线 | 任何协议字段调整必须先失败对应测试 |

### 3.3 不在本期范围

- LLM 真实推理（mock provider，已在 V20 主链路测试覆盖）
- BE 端集成测试（不在本期范围）
- 持久化层（SQLite、Chroma）端到端测试
- QQ 机器人集成测试

## 第7章 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| Minecraft 服务器启动耗时 > 60s | CI 不可行 | 预生成 WorldPreset + `-noverify` + `-XX:+UseG1GC` |
| Carpet Mod 假人 API 版本差异 | 测试不稳定 | 锁定 Carpet Mod ≥ 1.4.161 |
| 修复 Bug 1 后既有 V20 集成测试可能受影响 | 红色 build | 同步修复 tool-dispatcher.test.ts 中的 mock expectation |
| 协议字段在 AC 归一化层的容错可能掩盖未来 Bug | 静默错误 | 归一化时记录 warning 日志：检测到非标准字段时输出 "Protocol drift detected" |
| 真实 MC 服务器中工具调用异常难以调试 | 排障时间 | 诊断包统一捕获 server.log + IT 日志 |
| 协议契约测试使用 mock 客户端，与真实 JE 行为有差异 | 漏测 | 协议契约测试仅验证消息结构；完整行为验证在 e2e |
