## 第1章 概述

本文档定义 Agent Core 模块在开发过程中必须遵守的技术规范，以及每个版本交付时的验收标准。目标是确保：

| 目标 | 说明 |
|------|------|
| **代码质量** | 所有代码符合项目编码规范，通过自动化检查 |
| **接口一致** | 对外接口（JSON-RPC / LLM Provider / Memory / Task）严格遵循接口规范 |
| **可验收** | 每个版本的验收标准可测量、可验证，不接受模糊描述 |
| **可测试** | 每个功能模块有对应的测试覆盖 |

## 第2章 技术规范

### 2.1 TypeScript 编码规范

参考 [01-代码规范与风格指南.md](../standards/01-代码规范与风格指南.md)，Agent Core 特有规范如下：

#### 2.1.1 严格模式

`tsconfig.json` 必须开启严格模式：

```jsonc
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true,
    "exactOptionalPropertyTypes": false,
    "forceConsistentCasingInFileNames": true
  }
}
```

#### 2.1.2 模块命名约定

| 层级 | 命名规则 | 示例 |
|------|----------|------|
| 文件名 | kebab-case | `tcp-server.ts`, `memory-manager.ts` |
| 类名 | PascalCase | `class MemoryManager`, `class TaskScheduler` |
| 接口名 | PascalCase（无 I 前缀） | `interface LLMProvider`, `interface TaskRunner` |
| 方法名 | camelCase | `async store()`, `async queryNearby()` |
| 常量 | UPPER_SNAKE_CASE | `DEFAULT_TCP_PORT`, `MAX_RECONNECT_ATTEMPTS` |
| 类型别名 | PascalCase | `type TaskStatus`, `type MemoryType` |
| 枚举名 | PascalCase，成员 UPPER_SNAKE | `enum ErrorCode { AUTH_FAILED = -32001 }` |

#### 2.1.3 文件结构规范

每个模块文件必须遵循以下结构（按顺序）：

```typescript
// 1. import 语句（先外部依赖，再内部模块，按字母序）
import net from 'node:net';
import { EventEmitter } from 'node:events';
import { JsonRpcRequest } from '../shared/types';

// 2. 类型/接口定义
export interface WorkspaceConfig {
  // ...
}

// 3. 类定义
export class WorkspaceManager {
  // ...
}

// 4. 工具函数（如有）
function validateConfig(config: WorkspaceConfig): void {
  // ...
}
```

#### 2.1.4 错误处理规范

- 所有 async 函数必须 try-catch 顶层错误
- 自定义错误类继承 `Error`，使用 `ErrorCode` 枚举
- 禁止在 catch 块中吞掉错误（必须 log 或 rethrow）

```typescript
export class TcpServerError extends Error {
  constructor(
    public readonly code: ErrorCode,
    message: string,
  ) {
    super(message);
    this.name = 'TcpServerError';
  }
}
```

#### 2.1.5 注释规范

- **禁止添加非必要的注释**，代码应自解释
- 仅在以下情况添加注释：
  - 复杂的算法逻辑（如依赖分析拓扑排序）
  - 对外暴露的 API（JSDoc 格式）
  - 已知限制或 TODO（格式：`// TODO: 某某原因，后续优化`）

```typescript
/**
 * 解析依赖关系，构建执行层级图
 *
 * @param calls - 工具调用列表
 * @returns 拓扑排序后的执行层级，每层可并行执行
 */
analyze(calls: ToolCallContent[]): ExecutionLayer[]
```

### 2.2 JSON-RPC 2.0 接口规范

参考 [01-通信协议规范.md](../protocols/01-通信协议规范.md)，Agent Core 作为 TCP 服务端必须遵守：

#### 2.2.1 消息格式

```typescript
// 请求
interface JsonRpcRequest {
  jsonrpc: '2.0';
  id: number | string;
  method: string;
  params?: any;
}

// 响应
interface JsonRpcResponse {
  jsonrpc: '2.0';
  id: number | string;
  result?: { success: boolean; data?: any; duration_ms?: number };
  error?: { code: number; message: string; data?: any };
}

// 通知
interface JsonRpcNotification {
  jsonrpc: '2.0';
  method: string;
  params?: any;
}
```

#### 2.2.2 必须实现的方法

| 方法 | 方向 | 说明 |
|------|------|------|
| `handshake` | Adapter → AC | 握手认证，参数：`{ instance_id, auth_token, version, edition }` |
| `register_tools` | Adapter → AC（通知） | 注册工具列表，参数：`{ tools: ToolDefinition[] }` |
| `pong` | Adapter → AC | 心跳响应 |
| `status_update` | Adapter → AC（通知） | 状态上报，参数：`{ health, hunger, position, equipment, inventory }` |
| `event` | Adapter → AC（通知） | 事件通知，参数：`{ type: EventType, data: any }` |
| `call_tool` | AC → Adapter（请求） | 调用工具，参数：`{ tool_name, parameters, timeout_ms }` |
| `batch_call` | AC → Adapter（请求） | 批量调用，参数：`{ calls: Array<{ id, method, params, dependsOn }> }` |

#### 2.2.3 错误码

| 错误码 | 含义 | 触发条件 |
|--------|------|----------|
| `-32000` | 工具执行失败 | 工具返回 success=false |
| `-32001` | 认证失败 | handshake 校验不通过 |
| `-32002` | 未认证 | 未 handshake 就发送其他消息 |
| `-32003` | 版本不匹配 | AC 和 Adapter 协议版本不一致 |
| `-32004` | 工具未找到 | 调用的工具未注册 |
| `-32005` | 工具超时 | 工具执行超过 timeout |
| `-32006` | 实例忙 | 实例正在执行其他命令 |
| `-32009` | 连接数超限 | 超过最大连接数（10） |

#### 2.2.4 粘包处理

使用 `\n`（换行符）作为消息帧分隔符：
- 发送端：每条 JSON 消息末尾附加 `\n`
- 接收端：累积缓冲区数据，按 `\n` 分割提取完整消息
- 不完整消息暂存缓冲区，等待后续数据到达

#### 2.2.5 心跳协议

| 参数 | 值 |
|------|-----|
| 默认间隔 | 10s |
| 超时时间 | 30s |
| 失败重试 | 最多 5 次，指数退避 |
| 方向 | AC 作为服务端发送 Ping，Adapter 回复 Pong |

### 2.3 LLM Provider 接口规范

参考 [01-LLM抽象层接口规范.md](../api/01-LLM抽象层接口规范.md)，每个 Provider 实现必须：

#### 2.3.1 实现接口

```typescript
interface LLMProvider {
  readonly metadata: ProviderMetadata;
  chat(messages: Message[], tools?: ToolDefinition[], options?: ChatOptions): Promise<LLMResponse>;
  chatStream(messages: Message[], tools?: ToolDefinition[], options?: ChatOptions): AsyncIterable<LLMChunk>;
  embed?(text: string): Promise<number[]>;
  healthCheck(): Promise<HealthCheckResult>;
}
```

#### 2.3.2 必须支持的模型

| Provider | 最低支持的模型 | 必须支持的功能 |
|----------|---------------|----------------|
| OpenAI | gpt-4o, gpt-4o-mini | streaming + function calling |
| Claude | claude-3-5-sonnet-20241022 | streaming + function calling |
| Gemini | gemini-2.0-flash | streaming + function calling |
| Ollama | qwen2.5:7b | streaming + function calling |

#### 2.3.3 错误处理

| 错误类型 | 处理方式 |
|----------|----------|
| 网络错误（连接超时/DNS 解析失败） | 重试 3 次，仍失败触发 ModelRouter 降级 |
| 认证错误（401 Unauthorized） | 立即返回降级，不重试 |
| 限流错误（429 Rate Limit） | 指数退避重试，最大 5 次 |
| API 返回格式异常 | 重试 3 次，仍失败返回 LLM_002 错误码 |
| 上下文超限 | 自动截断后重试，返回 LLM_005 错误码 |

### 2.4 记忆系统接口规范

#### 2.4.1 双模存储策略

| 重要度范围 | 含义 | 清理策略 |
|:--------:|------|----------|
| 1-3 | 低重要度 | 30 天无访问自动清理 |
| 4-6 | 中重要度 | 90 天无访问自动清理 |
| 7-9 | 高重要度 | 永不过期 |

- 全量内存加载：启动时 < 500ms（10 万条特征以内）

### 2.5 任务系统接口规范

参考 [04-任务系统接口规范.md](../api/04-任务系统接口规范.md)，关键规范：

#### 2.5.1 任务状态机

```
pending → running → completed
pending → running → failed → pending（重试）
pending → paused → pending（恢复）
pending → cancelled
running → paused → pending（恢复）
running → cancelled
running → failed → pending（重试）或 failed（重试耗尽）
```

必须严格执行状态转换，禁止非法转换（如 `pending → completed`）。

#### 2.5.2 并发控制

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 最大并发数 | 3 | 同时最多执行 3 个任务 |
| 轮询间隔 | 1000ms | 调度循环每 1s 检查一次队列 |
| 优先级排序 | critical > high > normal > low | 同优先级 FIFO |

#### 2.5.3 重试策略

| 参数 | 默认值 |
|------|--------|
| 最大重试次数 | 3 |
| 初始重试间隔 | 30s |
| 退避乘数 | 2.0 |
| 重试优先级 | 临时提升一级（避免饿死） |

### 2.6 数据库表设计规范

#### 2.6.1 通用规范

- 所有表必须有 `created_at` 和 `updated_at` 字段（`INTEGER NOT NULL DEFAULT (unixepoch())`）
- 所有外键使用 `ON DELETE CASCADE` 或 `ON DELETE SET NULL`
- 所有 JSON 字段存储为 `TEXT` 类型，应用层做序列化/反序列化
- 所有索引遵循 `idx_表名_字段名` 命名规范

#### 2.6.2 Config 表规范

```sql
CREATE TABLE config (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL,
  value_type TEXT NOT NULL CHECK(value_type IN ('string', 'number', 'boolean', 'json')),
  description TEXT,
  updated_at INTEGER NOT NULL DEFAULT (unixepoch())
);
```

配置 key 使用前缀命名空间：
- `tcp_*`：TCP 服务端配置
- `llm_*`：LLM 配置
- `memory_*`：记忆系统配置
- `task_*`：任务系统配置
- `qq_*`：QQ 机器人配置
- `log_*`：日志配置

#### 2.6.3 迁移规范

- 新建表：在对应版本的 schema.sql 中添加 DDL
- 修改表：使用 `ALTER TABLE` + 版本号注释，禁止直接修改已有表的 DDL
- 删除表：标记 `deprecated` 字段或移入 `_deprecated` 表，至少保留 1 个版本后再物理删除
