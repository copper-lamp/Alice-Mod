## 第1章 概述

### 1.1 模块定位

Agent Core 是 McAgent 系统的**智能体核心**，基于 Electron + TypeScript/Node.js + React 构建的桌面应用。它是整个系统的"大脑"，承担三个核心角色：

| 角色 | 说明 |
|------|------|
| **LLM 大脑** | 统一调度大语言模型，组装提示词、解析 Function Calling 结果、管理对话上下文 |
| **调度中心** | 管理工作区生命周期，通过 TCP JSON-RPC 协议与多个 Adapter Core 实例通信 |
| **记忆存储** | 基于 SQLite + Chroma 实现结构化和向量化双模存储 |

Agent Core **不依赖游戏运行**，可独立启动。它通过 TCP 连接与 Adapter Core（运行在游戏服务器内的模组/插件）通信，一个 Agent Core 可同时管理多个 Adapter Core 实例（跨服、跨版本）。

### 1.2 技术栈

| 技术 | 用途 |
|------|------|
| Electron | 桌面应用框架，主进程 + 渲染进程架构 |
| TypeScript | 主进程开发语言（Node.js 20+） |
| React | 渲染进程 UI 框架 |
| Node.js net | TCP 服务端 |
| better-sqlite3 + Chroma | 记忆系统存储 |
| Vitest | 单元测试 |
| electron-builder | 打包发布 |

### 1.3 团队规模

Agent Core 模块由 **2 名开发者**完成全部 15 个版本的迭代开发：

| 开发者 | 负责方向 | 核心模块 |
|--------|----------|----------|
| 开发者 A | UI + LLM + 提示词 | A1 UI 界面、A2 LLM 调度、A3 提示词系统、配置面板 |
| 开发者 B | 工作区 + TCP + 记忆 + 任务 | A4 工作区管理、A5 TCP 服务端、A6 记忆系统、A7 QQ 机器人、A8 事件触发器、A9 任务规划、A10 日志系统 |

| 指标 | 目标 |
|------|------|
| 最大 Adapter Core 连接数 | 10 个 |

## 第6章 数据持久化设计

### 6.1 数据持久化总览

#### 6.1.1 存储策略矩阵

Agent Core 采用**多级存储策略**，根据数据特性选择不同的存储后端：

| 存储后端 | 用途 | 数据特征 | 文件位置 |
|----------|------|----------|----------|
| **SQLite**（`mcagent.db`） | 结构化数据持久化 | 关系型、可索引、需查询 | `{userData}/mcagent.db` |
| **Chroma** | 向量语义检索 | 高维向量、相似度搜索 | 独立 Chroma 服务 / `{userData}/chroma/` |
| **JSON 文件** | 导入/导出数据 | 一次性导入、不需要频繁查询 | `{userData}/instances.json` |
| **文件系统** | 日志文件 | 追加写入、轮转管理 | `{userData}/logs/` |
| **内存** | 运行时缓存 | 临时数据、可重建 | — |

> `{userData}` 为 Electron 的 `app.getPath('userData')` 目录。

#### 6.1.2 数据库文件架构

所有结构化数据统一存储在 `mcagent.db`（单个 SQLite 文件），WAL 模式运行：

```
mcagent.db
├── 系统表（1 张）
│   └── schema_version        — Schema 版本控制
├── 全局配置（1 张）
│   └── config                — Key-Value 配置存储
├── 日志与调试（2 张）
│   ├── logs                  — 分级日志记录
│   └── tool_call_records     — 工具调用记录
├── 工作区与实例（2 张）
│   ├── workspace_meta        — 工作区持久化
│   └── instances             — 实例配置管理
├── LLM 调度（2 张）
│   ├── llm_call_records      — LLM 调用观测记录
│   └── llm_provider_config   — 模型 Provider 配置
├── 记忆系统（3 张）
│   ├── memory_meta           — 记忆元数据
│   ├── memory_tags           — 标签索引
│   └── memory_access_log     — 访问日志
├── 地图索引（3 张）
│   ├── map_features          — 地图特征
│   ├── map_spatial_grid      — 空间网格索引
│   └── map_regions           — 命名区域
├── 任务系统（3 张）
│   ├── task_meta             — 任务元数据
│   ├── task_deps             — 任务依赖关系
│   └── task_schedule         — 任务调度信息
├── QQ 机器人（2 张）
│   ├── qq_bot_config         — QQ 机器人持久化配置
│   └── qq_msg_history        — QQ 消息历史
└── 预留扩展（2 张，V15+）
    ├── knowledge_base        — 知识库（future）
    └── skill_registry        — 技能注册表（future）
```

#### 6.1.3 数据库初始化流程

```
应用启动
  │
  ├─ Step 1: 打开/创建 mcagent.db（WAL 模式）
  │
  ├─ Step 2: DatabaseManager.init()
  │   ├── 检查 schema_version 表
  │   ├── 读取当前版本号
  │   ├── 按版本顺序执行迁移脚本
  │   └── 更新 schema_version
  │
  ├─ Step 3: 各模块初始化
  │   ├── MemoryManager  → SQLiteStore + MapIndex 全量加载
  │   ├── TaskManager    → 恢复未完成任务状态
  │   ├── WorkspaceManager → 恢复工作区列表
  │   ├── Logger         → 初始化日志 SQLite 传输
  │   └── LLMConfig      → 加载 Provider 配置
  │
  └─ Step 4: 应用进入就绪状态
```

### 6.2 命名规范

#### 6.2.1 表名规范

| 规则 | 说明 | 示例 |
|------|------|------|
| 全小写 + 下划线分隔 | snake_case | `memory_meta`, `task_deps` |
| 单数形式 | 避免复数 | `logs`（例外，约定俗成） |
| 模块前缀 | 按模块分组 | `memory_*`, `task_*`, `map_*`, `llm_*`, `qq_*` |
| 最长 32 字符 | 保持简洁 | — |

#### 6.2.2 列名规范

| 规则 | 说明 | 示例 |
|------|------|------|
| 全小写 + 下划线分隔 | snake_case | `workspace_id`, `created_at` |
| 不使用缩写 | 除非广泛接受 | `embedding_id`（接受），`mem_id`（拒绝） |
| 布尔字段用 `is_` / `has_` 前缀 | 明确语义 | `is_online`, `has_error` |
| JSON 字段用 `_json` 后缀 | 标明存储格式 | `content_json`, `result_json` |
| 时间戳字段用 `_at` 后缀 | 明确类型 | `created_at`, `updated_at`, `expires_at` |
| 外键字段与主表主键同名 | 便于关联 | `memory_meta.id` → `memory_tags.memory_id` |

#### 6.2.3 索引命名规范

```
idx_{表名}_{列名1}_{列名2}
```

示例：`idx_memory_meta_workspace`、`idx_task_meta_status`

复合索引：`idx_map_features_coords`（dimension, x, z）

#### 6.2.4 约束命名规范

| 约束类型 | 命名规则 | 示例 |
|----------|----------|------|
| CHECK | `ck_{表名}_{列名}` | `ck_logs_level` |
| UNIQUE | `uq_{表名}_{列名}` | `uq_task_schedule_task_id` |

### 6.3 完整表结构定义

> 所有表统一使用 `CREATE TABLE IF NOT EXISTS`，确保幂等初始化。
> 时间戳字段统一使用 `INTEGER` 存储 Unix 毫秒时间戳。
> UUID 字段统一使用 `TEXT` 存储 UUID v4 字符串。

### 6.4 数据库管理器设计

#### 6.4.1 DatabaseManager 接口

统一管理数据库连接、Schema 初始化、迁移、备份。

```typescript
interface IDatabaseManager {
  /** 初始化数据库连接 */
  init(dbPath: string): Promise<void>;

  /** 获取数据库实例 */
  getDb(): Database.Database;

  /** 执行 Schema 迁移 */
  migrate(targetVersion?: number): Promise<void>;

  /** 获取当前 Schema 版本 */
  getSchemaVersion(): Promise<number>;

  /** 备份数据库 */
  backup(backupPath: string): Promise<void>;

  /** 关闭数据库连接 */
  close(): void;
}
```

#### 6.4.2 初始化流程

```
DatabaseManager.init()
  ├─ 1. 打开 SQLite 连接（WAL + foreign_keys）
  ├─ 2. 创建 schema_version 表（如果不存在）
  ├─ 3. 读取当前版本号
  ├─ 4. 按版本顺序执行待迁移脚本
  ├─ 5. 更新 schema_version 表
  └─ 6. 返回就绪状态
```

#### 6.4.3 模块注册机制

每个模块向 DatabaseManager 注册自己的表创建/迁移脚本：

```typescript
interface ModuleSchema {
  module: string;         // 模块名，如 'memory', 'task'
  version: number;        // 模块 schema 版本号
  dependencies: string[]; // 依赖的其他模块
  scripts: {              // 按版本号排序的迁移脚本
    [version: number]: string;  // version → DDL SQL
  };
}
```

### 6.5 迁移策略

#### 6.5.1 Schema 版本管理

- 使用 `schema_version` 表记录每次迁移
- 每个模块独立版本号，全局递增版本号兼做兼容性标记
- 迁移脚本必须是幂等的（`CREATE TABLE IF NOT EXISTS` + 增量 `ALTER TABLE`）

#### 6.5.2 迁移规则

| 规则 | 说明 |
|------|------|
| 只增不删 | 不删除列，只新增列或表 |
| 向后兼容 | 旧版代码读取新版数据库不应崩溃 |
| 增量迁移 | 每个版本一个迁移脚本，按顺序执行 |
| 回滚策略 | 通过备份恢复（`backup()` 方法） |

#### 6.5.3 迁移示例

```sql
-- V1 → V2 迁移：新增 workspace_meta 表
CREATE TABLE IF NOT EXISTS workspace_meta (...);

-- V2 → V3 迁移：新增 llm_call_records 表
CREATE TABLE IF NOT EXISTS llm_call_records (...);

-- V3 → V4 迁移：为 task_meta 新增列
ALTER TABLE task_meta ADD COLUMN IF NOT EXISTS tags TEXT NOT NULL DEFAULT '[]';
```

## 第7章 V9 详细需求 — 后端调试模块（工具调用记录器 + 日志系统）

> 版本：V9（第 11 周）
> 对应需求：AC-UI-06、AC-LOG-01、AC-LOG-02、AC-LOG-03、AC-LOG-04

### 7.1 模块定位

V9 是 Agent Core **后端调试基础设施**的关键版本。本模块**不接入前端 UI**，纯后端运行，为开发者提供调试 Agent 执行过程的能力。它在 V4 Function Calling Pipeline 的事件系统基础上，引入**工具调用记录器**和**全功能日志系统**，所有数据通过文件、控制台、SQLite 三种渠道输出，供开发者离线或实时分析。

| 职责 | 说明 |
|------|------|
| **工具调用记录器** | 监听 Pipeline 事件，将每次工具调用的参数、结果、耗时、依赖关系持久化到 SQLite，并输出到日志文件和控制台 |
| **分级日志系统** | 基于级别的日志记录（debug/info/warning/error），支持多路输出（文件 + 控制台 + SQLite）和文件轮转 |
| **结构化日志存储** | 所有日志写入 SQLite logs 表，支持按级别/模块/时间范围检索 |
| **调试数据持久化** | 工具调用记录和日志均持久化到磁盘，支持事后回溯分析，不依赖前端渲染 |

**关键设计原则**：
- 纯后端运行，不依赖 Electron 渲染进程
- 工具调用数据通过 Pipeline 事件系统自动收集，零侵入
- 日志和工具调用记录均结构化存储，支持程序化查询
- 所有输出渠道（文件/控制台/SQLite）独立可配置

### 7.2 详细功能需求

#### 7.2.1 AC-UI-06 工具调用记录器（后端调试模块）
