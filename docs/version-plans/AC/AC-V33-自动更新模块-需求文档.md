## 第1章 项目背景

### 1.1 现状

Alice Mod 的 Agent Core（AC）是一个 Electron 桌面应用，当前没有内置的自动更新机制。用户需要手动从 GitHub Releases 页面下载新版本安装包，重新安装后才能升级。

随着项目进入运营阶段，AC 需要实现以下能力：

1. **自动检测更新**：启动时自动检查是否有新版本
2. **多通道分发**：基于订阅 tier 分发不同通道（stable/beta）的版本
3. **静默下载与安装**：后台下载更新，安装时提示重启
4. **强制升级**：服务端可设置强制升级窗口，过期后阻断使用
5. **差分更新**：同 minor 版本仅下载增量，减少带宽消耗

### 1.2 与服务端的关系

AC 自动更新模块是 [SERVER 自动更新与 3 档订阅系统](../../../SERVER/docs/01-需求文档.md) 的**客户端侧实现**。服务端（Cloudflare Workers + R2 + Supabase）负责：

- 用户鉴权（token 校验）
- Release Policy 计算（tier → channel → allowed version）
- 安装包存储与签名 URL 分发（R2）
- Feature Flag 下发

AC 客户端负责：

- 本地 token 管理（加密存储）
- 启动时拉取 Release Policy
- 版本对比与更新决策
- 下载与安装更新
- 应用 Feature Flag

## 第2章 用户场景

### 2.1 标准版用户（免费）

**场景**：用户下载安装 AC 后，无需任何激活操作即可使用所有当前功能。

- 启动时直接拉取 `stable` 通道的 Release Policy
- 检测到新版本时弹窗提示更新
- 14 天内未升级则弹出强制升级对话框
- 无 token 文件，无 tier 信息

### 2.2 Pro 用户（¥19.9/季）

**场景**：用户在爱发电购买 Pro 订阅，收到激活码后在 AC 中激活。

- 激活后 token 加密存储在本地
- 启动时携带 token 拉取 `beta` 通道的 Release Policy
- 可升级到领先 stable 一个 major 的版本
- 7 天内未升级则弹出强制升级对话框
- 订阅到期后自动降级为 standard（清空 token）

### 2.3 Max 用户（¥79.9/年）

**场景**：用户在爱发电购买 Max 订阅，激活后享有未来高级功能。

- 激活后 token 加密存储在本地
- 启动时携带 token 拉取 `stable` 通道的 Release Policy（与 standard 相同）
- 不享受 Pro 的提前更新机制
- 14 天内未升级则弹出强制升级对话框
- 未来高级功能开关在 Feature Flag 中下发

### 2.4 离线用户

**场景**：用户处于无网络环境。

- 启动时跳过 Release Policy 拉取
- 使用上次缓存的 policy（如果有）
- 不检查更新
- 所有 Feature Flag 使用默认值

## 第3章 功能需求

### 3.1 许可证管理

#### FR-LIC-001 Token 加密存储
- **描述**：激活后获得的 token 必须安全存储在本地
- **实现**：使用 `electron.safeStorage`（OS 级 keychain）加密，写入 `userData/license.dat`
- **接口**：
  - `save(token: string): Promise<void>`
  - `load(): Promise<string | null>`
  - `clear(): Promise<void>`

#### FR-LIC-002 标准版零配置
- **描述**：标准版用户无需创建任何许可证文件
- **行为**：`load()` 返回 `null`，直接使用默认策略（stable 通道）

#### FR-LIC-003 Token 失效处理
- **描述**：服务端返回 401 或 active=false 时清空本地 token
- **行为**：清空后下次启动按标准版处理

#### FR-LIC-004 Reissue 入口
- **描述**：设置页面提供"重新激活"入口
- **行为**：用户输入邮箱 + 订单号 → 调用 `POST /v1/auth/reissue` → 收到新激活码 → 重新激活

### 3.2 策略拉取

#### FR-POL-001 启动时拉取
- **描述**：AC 启动时自动调用 `GET /v1/release?token=xxx&app_version=x.x.x`
- **时序**：在 `app.whenReady()` 之后、窗口创建之前执行
- **超时**：5 秒超时，超时后使用上次缓存的 policy 或默认值

#### FR-POL-002 重试机制
- **描述**：网络错误时自动重试
- **策略**：3 次重试，指数退避（1s → 2s → 4s）

#### FR-POL-003 Policy 缓存
- **描述**：成功拉取后缓存到内存，供本次会话使用
- **持久化**：可选缓存到本地文件，用于离线启动

#### FR-POL-004 静默失败
- **描述**：拉取失败不影响 AC 正常启动
- **行为**：使用默认策略（stable 通道，无强制升级），打日志告警

### 3.3 版本对比与更新决策

#### FR-UPD-001 版本对比
- **描述**：将当前版本与 `allowed_major.allowed_minor` 比较
- **规则**：
  - `current_major < allowed_major` → 有更新
  - `current_major === allowed_major && current_minor < allowed_minor` → 有更新
  - 否则 → 已是最新

#### FR-UPD-002 静默模式
- **描述**：当前版本落后但未超过强升宽限期时，后台下载更新
- **行为**：下载完成后 toast 提示"更新已就绪，重启生效"

#### FR-UPD-003 强制升级
- **描述**：超过强升宽限期后，弹出阻断式对话框
- **行为**：
  - 对话框无关闭按钮
  - 显示"请升级到最新版本以继续使用"
  - 提供"立即升级"按钮 → 下载更新
  - 下载完成后自动重启安装
  - 用户无法绕过（对话框阻塞主进程）

#### FR-UPD-004 差分更新
- **描述**：同 minor 升级使用 electron-updater 的 blockmap 差分
- **效果**：从 v2.3.0 到 v2.4.0 仅下载增量（通常 < 5MB）
- **跨 major**：全量下载

#### FR-UPD-005 更新进度 UI
- **描述**：下载过程中显示进度条
- **位置**：主窗口底部状态栏，或在系统托盘

### 3.4 Feature Flag 应用

#### FR-FLAG-001 极简 Flag 模型
- **描述**：AC 侧维护一个 `FeatureFlags` 单例
- **Flag 列表**：
  | Key | 类型 | 默认值 | 用途 |
  |---|---|---|---|
  | `active` | boolean | true | 订阅是否有效 |
  | `max_premium_features` | boolean | false | Max 未来高级功能 |
  | `update_channel` | 'stable' \| 'beta' | 'stable' | 更新通道 |
  | `force_upgrade_days` | number | 14 | 强升宽限天数 |

#### FR-FLAG-002 客户端零 tier 判断
- **约束**：AC 主进程代码中不允许出现 `tier === 'pro'` 等硬编码判断
- **例外**：`updater/` 模块内部允许读取 tier 用于日志和调试
- **审计**：CI 中 lint 检查禁止 `tier` 关键字出现在非 updater 模块

#### FR-FLAG-003 Flag 消费点
- **当前**：仅 `update_channel` 有消费方（影响 electron-updater 的更新源）
- **未来**：`max_premium_features` 预留，当前无消费方

### 3.5 UI 交互

#### FR-UI-001 设置页激活入口
- **未激活用户**：显示"升级 Pro/Max"按钮，点击跳转爱发电
- **已激活用户**：显示当前 tier、到期时间、续费链接
- **即将过期用户**：显示警告横幅（到期前 7 天）

#### FR-UI-002 更新通知
- **常规更新**：右下角 toast 通知
- **强制升级**：模态对话框，无关闭按钮
- **更新就绪**：提示"重启以应用更新"

#### FR-UI-003 版本信息展示
- **位置**：设置页 → 关于
- **内容**：当前版本、更新通道、最新版本（如有）

## 第4章 非功能需求

### 4.1 性能

| 指标 | 目标 |
|---|---|
| 启动检查 P50 延迟 | < 500ms（网络正常时） |
| 启动检查 P99 延迟 | < 2000ms（含重试） |
| 更新下载速度 | 取决于用户带宽（R2 全球 CDN） |
| 更新包大小（全量） | < 150MB（压缩后） |
| 更新包大小（差分） | < 10MB（同 minor） |

### 4.2 安全

| 维度 | 要求 |
|---|---|
| Token 存储 | `electron.safeStorage`（OS keychain） |
| 传输加密 | HTTPS |
| 更新包校验 | electron-updater 内置签名校验 |
| 策略接口 | 仅发起 HTTPS 请求，不暴露敏感信息 |

### 4.3 兼容性

| 平台 | 支持 |
|---|---|
| Windows 10+ | ✅ （NSIS 安装包） |
| macOS 12+ (Intel) | ✅ （DMG） |
| macOS 12+ (Apple Silicon) | ✅ （ARM64 DMG） |
| Ubuntu 20.04+ | ✅ （AppImage） |

### 4.4 可用性

| 场景 | 行为 |
|---|---|
| 网络不可用 | 跳过更新检查，使用缓存策略 |
| 服务端不可用 | 跳过更新检查，使用默认策略 |
| 更新下载失败 | 下次启动重试 |
| 安装失败 | 回滚到上一版本（electron-updater 内置） |

## 第5章 约束与边界

### 5.1 必须遵守

1. **客户端零 tier 判断**：所有授权逻辑在服务端，AC 只读 Feature Flag
2. **与服务端契约一致**：Release Policy 接口格式必须与 SERVER 文档一致
3. **不阻塞启动**：更新检查失败不影响 AC 正常使用
4. **最小改动原则**：尽量不修改现有 V1-V32 业务模块代码

### 5.2 不在范围

- ❌ 服务端实现（属 SERVER 模块职责）
- ❌ 爱发电支付集成
- ❌ 邮件发送
- ❌ GitHub Actions CI 配置（属 [工作流方案](../../../SERVER/docs/04-GitHub工作流方案.md) 职责）
- ❌ R2 存储配置

### 5.3 依赖

| 依赖项 | 版本 | 用途 | 来源 |
|---|---|---|---|
| `electron-updater` | ^6.x | 更新下载与安装 | npm |
| `electron.safeStorage` | 内置 | Token 加密存储 | Electron API |
| `undici` / `fetch` | 内置 | HTTP 请求 | Node.js |

## 第6章 验收标准

### 6.1 最小闭环

- [ ] 标准版用户启动时能检测到新版本（stable 通道）
- [ ] 更新下载完成后提示重启
- [ ] 重启后新版本生效
- [ ] 设置页显示当前版本号

### 6.2 完整功能

- [ ] Pro 用户激活后能检测到 beta 通道更新
- [ ] Max 用户激活后仍使用 stable 通道
- [ ] 强制升级对话框按 tier 设置的宽限天数弹出
- [ ] 差分更新生效（同 minor 升级下载量 < 10MB）
- [ ] token 加密存储，卸载重装后丢失
- [ ] 服务端返回 401 时本地 token 自动清空
- [ ] 无网络时跳过更新检查，不报错
- [ ] 客户端代码无 `tier ===` 硬编码

### 6.3 灰度上线

- [ ] 5 个测试用户（2 标准 / 2 Pro / 1 Max）跑通完整流程
- [ ] 升级/降级/续费场景验证通过
- [ ] 7 天无 P0 故障

## 第7章 术语表

| 术语 | 含义 |
|---|---|
| AC | Agent Core，Electron 桌面客户端 |
| Release Policy | 服务端下发的发布策略，包含 tier、channel、allowed version 等 |
| Feature Flag | 功能开关，服务端动态计算后下发 |
| Channel | 更新通道（stable / beta） |
| Token | 用户长期凭据，用于服务端鉴权 |
| license.dat | 本地 token 加密存储文件 |
| electron-updater | Electron 更新框架，支持差分更新和签名校验 |
| 强升 | 服务端强制要求客户端升级到指定版本 |
