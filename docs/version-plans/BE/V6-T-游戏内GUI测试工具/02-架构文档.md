## 第1章 总体架构

### 1.1 设计原则

V6-T 采用**纯组合、零侵入**原则：

1. **不修改现有工具代码**：仅通过 `ToolManager.executeTool()` 调用已注册工具。
2. **复用现有 GUI 基础设施**：复用 `/mcagent test` 命令入口与 LLSE Form API。
3. **声明式表单生成**：基于 `ToolMetadata.input_schema` 运行时生成参数表单，新增工具无需改测试器。
4. **会话隔离**：每个玩家的菜单状态、目标假人、测试报告独立存储。

## 第5章 关键技术选型

| 技术点 | 选型 | 原因 |
|--------|------|------|
| 表单 API | LLSE `mc.newSimpleForm()` / `mc.newCustomForm()` | 无需额外依赖 |
| 表单包装 | 复用 `lib/LLSE-FakePlayer/Gui/BetterForms.js` | 已封装回调路由，减少样板代码 |
| 工具调用 | `ToolManager.executeTool()` | 统一超时、错误处理 |
| 上下文构造 | `ToolContextImpl` | 与 AC 调用路径一致，保证测试真实性 |
| 坐标默认值 | `player.getBlockFromViewVector()` / `player.pos` | 减少玩家手动输入 |
| 报告存储 | 内存数组（非持久化） | 临时测试工具，重启后清空 |
| 权限检查 | `player.isOP()` + 配置白名单 | 简单可靠 |

## 第6章 表单生成策略

### 6.1 Schema → 控件映射

```typescript
// ToolParamFormBuilder 核心逻辑示意

function buildFromSchema(metadata: ToolMetadata): BetterCustomForm {
  const form = new BetterCustomForm(metadata.name);
  const schema = metadata.input_schema;

  for (const [key, prop] of Object.entries(schema.properties || {})) {
    const title = `${key}${isRequired(key) ? ' *' : ''}`;

    if (prop.enum) {
      form.addDropdown(key, title, prop.enum, defaultIndex(prop));
    } else if (prop.type === 'boolean') {
      form.addSwitch(key, title, prop.default ?? false);
    } else if (prop.type === 'number') {
      const def = isCoordinate(key) ? getDefaultCoordinate(key) : (prop.default ?? 0);
      form.addInput(key, title, '', String(def));
    } else if (prop.type === 'string') {
      const def = getSmartDefault(key, prop.default);
      form.addInput(key, title, '', def);
    } else if (prop.type === 'array') {
      form.addInput(key, title, '逗号分隔', '');
    } else if (prop.type === 'object' && prop.properties?.x) {
      // 坐标对象：拆成 x/y/z 三个输入
      form.addInput(`${key}.x`, `${title}.x`, '', String(getDefaultCoordinate('x')));
      form.addInput(`${key}.y`, `${title}.y`, '', String(getDefaultCoordinate('y')));
      form.addInput(`${key}.z`, `${title}.z`, '', String(getDefaultCoordinate('z')));
    }
  }

  return form;
}
```

### 6.2 智能默认值

| 字段名模式 | 默认策略 |
|------------|----------|
| `x` / `y` / `z` | 玩家准心目标方块坐标，无准心则玩家脚下方块 |
| `target.x` / `target.y` / `target.z` | 同上 |
| `from.x` / `from.y` / `from.z` | 玩家脚下方块 |
| `to.x` / `to.y` / `to.z` | 玩家脚下方块 + 2 |
| `block_name` | `stone`（若背包有则优先背包内方块） |
| `item_name` | 玩家主手物品名，若空则 `apple` |
| `food_name` | 空（表示自动选择） |
| `mode` / `action` | schema enum 第一项 |

## 第7章 与周边模块集成

### 7.1 与 ToolManager / ToolRegistry 集成

- 启动时通过 `toolManager.getRegistry().getAll()` 获取已注册工具列表。
- 按 `metadata.category` 过滤出 `movement` / `inventory` / `survival` / `block` 四类。
- 执行时调用 `toolManager.executeTool(name, params, ctx)`，完全复用现有超时和错误处理。

### 7.2 与 BotManager 集成

- 通过 `BotManager.list()` / `BotManager.getAll()` 获取假人信息。
- 工具执行前检查目标假人 `isOnline()`，不在线则提示。
- 通过 `ToolContextImpl({ activeBotName })` 将目标假人注入工具上下文。

### 7.3 与配置模块集成

- 从 `config.json` 读取 `gui_test.*` 配置。
- `enabled=false` 时，`/mcagent test` 提示测试工具已关闭。
- `require_op=true` 时，非 OP 玩家无法打开菜单。

### 7.4 与现有 BotTestSuite 的关系

| 模块 | 用途 | 处理方式 |
|------|------|----------|
| `BotTestSuite` | 旧版假人生命周期/传送/攻击测试 | 保留，但将 `/mcagent test` 主入口切换为 `InGameToolTester`；旧套件可通过 `/mcagent legacytest` 或子菜单保留入口 |
| `InGameToolTester` | 新版基于 ToolManager 的工具测试 | 成为 `/mcagent test` 默认实现 |

> 为最小化对现有测试习惯的影响，可在主菜单底部保留「旧版测试套件」按钮。

## 第8章 错误处理设计

### 8.1 错误分层

| 层级 | 错误类型 | 处理方式 |
|------|----------|----------|
| 权限层 | 无权限 / 测试关闭 | 聊天文本提示，不打开菜单 |
| 参数层 | 必填缺失 / 类型转换失败 | CustomForm 关闭后重新弹出并标红 |
| 前置层 | 无假人 / 假人不在线 | ModalForm 提示，返回主菜单 |
| 执行层 | 工具返回 `success=false` | ModalForm 展示 `error` 字段 |
| 异常层 | 工具抛异常 / 超时 | catch 后展示异常消息或 TIMEOUT |

### 8.2 超时处理

- 测试器不自行设置超时，复用 `ToolManager.executeTool()` 内基于 `metadata.execution.timeout_max_ms` 的超时机制。
- 超时结果以 `TOOL_TIMEOUT` 错误码展示，并计入报告。

### 8.3 并发控制

```typescript
// InGameToolTester 内部
private sessions = new Map<string, PlayerSession>();

async executeTool(player: Player, toolName: string, params: Record<string, unknown>): Promise<void> {
  const session = this.getSession(player);
  if (session.pendingExecution) {
    player.sendText('§c已有工具正在执行，请等待完成');
    return;
  }

  session.pendingExecution = this.doExecute(toolName, params, player);
  await session.pendingExecution;
  session.pendingExecution = null;
}
```

## 第9章 安全设计

### 9.1 权限模型

```
if (config.gui_test.require_op && !player.isOP()) {
  player.sendText('§c需要 OP 权限才能使用测试工具');
  return;
}

if (config.gui_test.allowed_players.length > 0 &&
    !config.gui_test.allowed_players.includes(player.xuid)) {
  player.sendText('§c你不在测试白名单中');
  return;
}
```

### 9.2 危险操作提示

对于 `area_operation`、`mine_block` 等会修改世界的工具，在表单顶部增加红色提示：

```
§c注意：此工具会修改世界方块，请在测试区域使用。
```

### 9.3 配置隔离

- `gui_test` 默认只在 `config.json` 中显式启用。
- 正式发布前建议将 `enabled` 设为 `false`，避免测试入口暴露给普通玩家。

## 第10章 扩展性设计

### 10.1 新增工具自动可见

只要新工具按 `IToolModule` 注册并带有 `metadata.category`，`ToolMenuGui` 会自动将其加入对应分类菜单，无需修改 V6-T 代码。

### 10.2 新增分类

若后续新增 `combat`、`entity` 等分类，只需在 `ToolMenuGui` 的分类名称映射表中追加条目：

```typescript
const CATEGORY_LABELS: Record<string, string> = {
  movement: '移动工具',
  inventory: '背包工具',
  survival: '生存工具',
  block: '方块工具',
  entity: '生物交互',   // 新增
  combat: '战斗工具',   // 新增
};
```

### 10.3 自定义冒烟用例

`SmokeTestCase` 可通过 `config.json` 的 `gui_test.smoke_cases` 覆盖，方便针对特定地图调整默认参数。
