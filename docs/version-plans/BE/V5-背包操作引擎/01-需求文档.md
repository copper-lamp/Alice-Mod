## 第1章 需求

### 1.1 模块定位

V5 背包操作引擎是 Adapter Core BE 执行 AI 引擎（C5）的**第三个子模块**，负责把 Agent Core 下发的物品管理意图转换为游戏内可执行操作。LLM 只需决定“丢什么、拿什么、放什么、装备什么”，引擎自动完成物品查找、槽位选择、容器交互、移动靠近、耐久与堆叠校验等细节。

本模块交付 4 个工具：

| 工具 | 说明 | 分类 |
|------|------|:----:|
| `drop_item` | 丢弃背包中指定物品，可指定数量，也可向实体附近丢弃 | inventory |
| `take_from_container` | 从容器（箱子、熔炉、桶等）取出物品到背包 | inventory |
| `put_to_container` | 将背包中的物品放入容器 | inventory |
| `equip_item` | 装备或卸下指定部位的装备/副手物品 | inventory |

### 1.4 接口需求

#### 1.4.1 工具层接口

4 个工具均实现 [IToolModule](file:///d:/McAgent/packages/adapter-bedrock/src/registry/tool-module.types.ts#L116) 接口，通过 `metadata()` 返回 JSON Schema，通过 `execute(params, ctx)` 执行。

工具返回统一格式：

```typescript
interface ToolResult {
  success: boolean;
  data?: Record<string, any>;   // 操作结果：item / count / remaining 等
  error?: string;               // 失败原因
  duration_ms: number;          // 执行耗时
}
```

每个工具返回需包含：

- **成功/失败原因**
- **最终状态**：实际操作的物品、数量、剩余数量
- **时间消耗**：`duration_ms`
- **资源消耗**：如有工具耐久损耗需在 `data` 中返回

#### 1.4.2 引擎层接口

```typescript
interface InventoryEngine {
  // 查询
  list(): ItemStack[];
  find(itemName: string): ItemSlot | null;
  findAll(itemName: string): ItemSlot[];
  getSelectedSlot(): number;
  selectSlot(slot: number): boolean;

  // 丢弃
  drop(itemName?: string, count?: number, targetEntityId?: string): DropResult;
  dropSlot(slot: number, count?: number): DropResult;

  // 装备
  equip(itemName: string, slot: ArmorSlot): EquipResult;
  unequip(slot: ArmorSlot): EquipResult;

  // 容器交互（委托 ContainerAPI）
  getContainerAPI(): ContainerAPI;
}

interface ContainerAPI {
  open(block: Block): Container | null;
  take(container: Container, itemName?: string, count?: number): TransferResult;
  put(container: Container, itemName?: string, count?: number): TransferResult;
  close(): void;
}
```

#### 1.4.3 JSON Schema 定义

详见 [BE-01-需求文档.md](../BE-01-需求文档.md#63-背包工具4-个) 第 6.3 节，本文不再重复。Schema 中 `is_movement` 字段定义如下：

| 工具 | is_movement | 原因 |
|------|:-----------:|------|
| `drop_item` | false | 原地丢弃 |
| `equip_item` | false | 原地装备 |
| `take_from_container` | true | 需先移动到容器旁 |
| `put_to_container` | true | 需先移动到容器旁 |

### 1.5 依赖关系

- **依赖 C3 假人管理**：通过 `BotAccess.getBotPlayer()` 获取 `Player` 对象，操作后调用 `BotManager.saveInventory()` 持久化。
- **依赖 C4 状态上报**：背包变更会反映在下一次 `status_report` 的 `inventory_summary` 中。
- **依赖 C5.2 移动系统**：`take_from_container` / `put_to_container` 在执行前调用 `move_to` 靠近容器。
- **依赖 C2 工具注册**：4 个工具需注册到 `ToolRegistry`，Agent Core 方可调用。
- **被 V6 方块/生存工具依赖**：`mine_block` / `place_block` / `eat` 等需要装备或材料管理。

### 2.3 数据流

```
[Agent Core 工具调用]
        │
        ▼

        │

[IToolModule.execute(params, ctx)]
        │
```

### 2.4 关键技术选型

| 技术点 | 选型 | 原因 |
|--------|------|------|
| 背包访问 | LLSE `Player.getInventory()` / `Player.getArmor()` / `Player.getOffHand()` | 原生 API，直接操作玩家容器 |
| 容器访问 | `Block.getContainer()` | 统一获取箱子、熔炉等容器对象 |
| 物品序列化 | NBT / SNBT（复用 BotInstance） | 保留完整附魔、耐久、自定义 NBT |
| 移动集成 | 调用 `move_to` 工具或 `BotManager.walkToPos()` | 与 V4 移动系统解耦，避免重复实现 |
| 名称匹配 | 先精确匹配，再模糊匹配（忽略命名空间与下划线） | 降低 LLM 对 `minecraft:` 前缀的依赖 |

### 3.4 关键实现细节

#### 3.4.1 丢弃物品流程

```typescript
// InventoryEngine.drop()
const slot = itemName ? this.find(itemName) : { slot: this.getSelectedSlot() };
if (!slot) return { success: false, error: '物品不存在' };

const item = inventory.getItem(slot.slot);
const dropCount = count ?? item.count;
const clone = item.clone();
clone.count = Math.min(dropCount, item.count);

const dropPos = calcPosFromViewDirection(getEntityFeetPos(pl), pl.direction, 2);
if (!mc.spawnItem(clone, dropPos)) {
  return { success: false, error: '生成掉落物失败' };
}

inventory.removeItem(slot.slot, clone.count);
BotManager.saveInventory(this.botName);
```

#### 3.4.2 容器取出流程

```typescript
// take_from_container 工具 execute()
const moveResult = await moveTo({ target: { x, y, z }, target_type: 'block', distance: 2 });
if (!moveResult.success) return moveResult;

const block = ctx.world.getBlock(x, y, z);
const container = block.getContainer();
if (!container) return { success: false, error: '目标方块不是容器' };

const result = containerAPI.take(container, itemName, count);
BotManager.saveInventory(botName);
return result;
```

### 3.5 与 C5 其他模块的协作

- **与 C5.2 移动系统**：容器类工具通过 `move_to` 靠近目标；`drop_item` / `equip_item` 不依赖移动。
- **与 C5.4 方块操作引擎**：`place_block` 需要预先从背包选择方块，`mine_block` 需要自动切换最佳工具，均依赖本模块的查找/选择能力。
- **与 C3 假人管理**：工具执行前通过 `ctx.bot.getBotPlayer()` 获取 `Player`；执行后调用 `BotManager.saveInventory()` 持久化。
- **与 C4 状态上报**：背包变化在下次 2s 周期上报中体现。

## 第4章 风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|:----:|:----:|:----:|----------|
| LLSE `SimulatedPlayer` 的容器交互 API 不稳定 | 中 | 高 | 优先使用 `Block.getContainer()`；对关键操作加 try/catch 与状态回滚 |
| 容器打开后未正确关闭导致物品不同步 | 中 | 中 | `ContainerAPI` 在 take/put 后统一 `close()`；异常时强制关闭 |
| 物品名称匹配歧义（如 `stone` vs `stone_bricks`） | 中 | 中 | 精确匹配优先；模糊匹配时返回候选列表供 LLM 确认 |
| 移动失败导致容器操作超时被误判为物品丢失 | 中 | 中 | `take_from_container` / `put_to_container` 先校验 move_to 结果 |
| 堆叠拆分计算错误导致超堆叠或丢失 | 低 | 高 | 单元测试覆盖 64/16/1 三种堆叠上限；操作前后校验总数量 |
| 装备位索引与 LLSE 版本差异 | 低 | 高 | 将映射关系集中放在 `InventoryEngine` 常量中，升级时统一调整 |
| 高并发容器操作导致 BDS tick 阻塞 | 低 | 中 | 单次容器交互限制处理槽位数；大数量转移分帧执行 |
