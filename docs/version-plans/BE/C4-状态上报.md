## 第1章 需求

### 1.1 模块定位

C4 状态上报模块负责将 Adapter Core BE 侧的游戏内状态周期性推送到 Agent Core，为 LLM 决策提供**基础上下文**。该模块仅上报 Agent Core 必须掌握的最小状态集合，避免过度注入感知细节（符合项目 memory 中 "上下文注入限制在 ~150 tokens" 的约束）。

### 1.2 功能需求

| 编号 | 功能 | 说明 | 优先级 |
|:----:|------|------|:------:|
| C4-FR-01 | 周期性状态上报 | 每 2s 采集并发送一次假人基础状态 | P0 |
| C4-FR-02 | 生命状态采集 | 生命值、最大生命值、饥饿值、饱和度、氧气值 | P0 |
| C4-FR-03 | 位置状态采集 | 坐标、维度、朝向（yaw/pitch） | P0 |
| C4-FR-04 | 装备快照 | 主手、副手、头盔、胸甲、护腿、靴子物品名称 | P0 |
| C4-FR-05 | 背包摘要 | 已用槽位数、总槽位数、物品名称与数量列表 | P0 |
| C4-FR-06 | 连接感知 | TCP 未连接时停止上报，恢复后自动重启 | P0 |
| C4-FR-07 | 事件通知推送 | 死亡、聊天、玩家加入/离开等事件主动推送 | P1 |
| C4-FR-08 | 多假人支持 | 为每个在线假人独立生成状态报文 | P1 |
| C4-FR-09 | 耗时告警 | 单次采集 + 上报耗时超过 100ms 时记录警告 | P2 |
| C4-FR-10 | 上报压缩 | 减少冗余字段，控制单次报文大小 | P2 |

### 1.3 非功能需求

| 编号 | 指标 | 要求 | 测量方式 |
|:----:|------|:----:|----------|
| C4-NFR-01 | 上报周期 | 2000ms（±500ms） | 模拟器统计相邻报文间隔 |
| C4-NFR-02 | 单次采集耗时 | < 100ms | 采集到序列化完成耗时 |
| C4-NFR-03 | 报文大小 | < 4KB | 序列化后字符串长度 |
| C4-NFR-04 | 上报成功率 | ≥ 99%（网络正常时） | 成功发送 / 总尝试次数 |
| C4-NFR-05 | 资源占用 | 不显著影响 TPS | 插件开启前后 TPS 对比 |

### 1.4 接口需求

状态上报以 JSON-RPC 2.0 Notification 形式发送，方法名为 `status_report`，无响应：

```json
{
  "jsonrpc": "2.0",
  "method": "status_report",
  "params": {
    "timestamp": "2026-07-12T10:00:00.000Z",
    "bot_id": "AliceBot",
    "health": { "health": 20, "max_health": 20, "hunger": 18, "saturation": 0, "air": 300 },
    "position": { "x": 100.5, "y": 64, "z": -50.2, "dimension": "overworld", "yaw": 90, "pitch": 0 },
    "equipment": { "hand": "iron_pickaxe", "offhand": null, "helmet": null, "chestplate": "iron_chestplate", "leggings": null, "boots": null },
    "inventory_summary": { "used_slots": 12, "total_slots": 36, "items": [{ "name": "cobblestone", "count": 64 }] }
  }
}
```

事件通知以 JSON-RPC 2.0 Notification 形式发送，方法名为 `event`：

```json
{
  "jsonrpc": "2.0",
  "method": "event",
  "params": {
    "event_type": "death",
    "severity": "warning",
    "timestamp": "2026-07-12T10:00:00.000Z",
    "data": { "player_name": "AliceBot", "source": "zombie" }
  }
}
```

### 1.5 依赖关系

- 依赖 C1 TCP 客户端：通过 `sendNotification` 发送报文
- 依赖 C3 假人管理：获取在线假人列表与玩家对象
- 被 C5 执行 AI 引擎间接依赖：工具执行结果可能触发状态变化，但不直接依赖本模块

### 2.2 核心类图

```
┌─────────────────────────────────────┐
│         StatusReporter              │
│  - sendNotification: function       │
│  - isConnected: function            │
│  - intervalMs: number               │

└─────────────────────────────────────┘
```

### 2.3 数据流

```
[onServerStarted / TCP connected]
       │
       ▼
[StatusReporter.start()] ──> setInterval 2000ms
       │
       ▼
[tick()] ──> 检查 enabled + TCP 连接
       │
       ▼
[collect()] ──> 遍历在线假人（当前为首个在线玩家）
       │
       ▼
[report()] ──> sendNotification('status_report', data)
       │
       ▼
[耗时检查] ──> >100ms 记录 warn
```

事件推送数据流：

```
[游戏事件 onPlayerDie / onChat / onJoin / onLeft]
       │
       ▼
[pushEvent(eventType, data)]
       │
       ▼
[sendNotification('event', payload)]
```

### 2.4 状态报文结构

| 字段 | 类型 | 说明 |
|------|------|------|
| `timestamp` | string | ISO 8601 时间戳 |
| `bot_id` | string | 假人唯一标识（建议新增） |
| `health` | object | 生命、饥饿、饱和度、氧气 |
| `position` | object | 坐标、维度、朝向 |
| `equipment` | object | 各装备位物品名称或 null |
| `inventory_summary` | object | 背包摘要 |

### 2.5 事件类型映射

| 游戏事件 | event_type | severity | 触发条件 |
|----------|:----------:|:--------:|----------|
| onPlayerDie | death | warning | 假人死亡 |
| onChat | player_chat | info | 任意玩家发送聊天 |
| onJoin | player_join | info | 玩家加入服务器 |
| onLeft | player_leave | info | 玩家离开服务器 |
| 自定义 | entity_attack | warning | 假人被攻击（V14 完善） |
| 自定义 | health_low | warning | 生命值低于阈值（V14 完善） |
| 自定义 | hunger_low | warning | 饥饿值低于阈值（V14 完善） |

### 2.6 关键技术选型

| 技术点 | 选型 | 原因 |
|--------|------|------|
| 定时器 | `setInterval` | 简单、满足 2s 周期需求 |
| 报文协议 | JSON-RPC 2.0 Notification | 无需响应，降低阻塞 |
| 玩家获取 | `mc.getOnlinePlayers()` / `BotManager.getAll()` | 当前实现取首个在线玩家，后续切到 BotManager |
| 时间戳 | `new Date().toISOString()` | 统一 UTC 时间 |

### 3.5 与 C3 假人管理模块的协作

当前 `StatusReporter` 未与 `BotManager` 集成。建议修改为：

```typescript
// 在 StatusReporter 中注入 BotManager
collect(): StatusReport[] {
  const bots = BotManager.getAll().filter(b => b.isOnline());
  if (bots.length === 0) return [];
  return bots.map(bot => {
    const pl = bot.getPlayer();
    return {
      timestamp: new Date().toISOString(),
      bot_id: bot.name,
      health: { ... },
      position: { ... },
      equipment: { ... },
      inventory_summary: this.collectInventorySummary(pl),
    };
  });
}

// report() 中逐条发送
report(): void {
  if (!this.isConnected()) return;
  const reports = this.collect();
  for (const report of reports) {
    this.sendNotification('status_report', report);
  }
}
```

## 第4章 风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|:----:|:----:|:----:|----------|
| `mc.getOnlinePlayers()` 在高并发下返回不稳定 | 中 | 中 | 改用 BotManager 维护的在线假人列表 |
| 多假人同时上报导致报文风暴 | 中 | 中 | 限制最大假人数，必要时合并为数组一次性上报 |
| BDS API 获取玩家属性时抛异常 | 低 | 高 | collect 中捕获异常，返回默认值并记录 error |
| 状态上报定时器未清理导致内存泄漏 | 中 | 中 | stop() 中 clearInterval， disconnect 时自动 stop |
| 事件推送频率过高 | 中 | 低 | 对高频事件（如聊天）做 1s 去抖或批量合并 |
