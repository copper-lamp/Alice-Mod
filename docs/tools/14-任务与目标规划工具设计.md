# 任务与目标规划工具设计（Task & Goal Planning Tools Design）

> 版本：v2.0（重设计）
> 日期：2026-07-16
> 架构：LLM 调用工具 → 任务调度器 → 执行器 → 返回结果
> 协议：遵循 `00-工具协议规范.md` 统一接口契约
> 关联文档：[V13-任务系统.md](../version-plans/AC/AC-V13-任务系统.md)、[14-工具协议规范.md](./00-工具协议规范.md)

---

## 1. 模块概述

### 1.1 目标

让 LLM 能完整地规划、跟踪、控制任务：

- 创建/批量创建任务（简单/复合/循环/条件 4 种类型）
- 分解复杂任务为子任务
- 查询任务、跟踪进度、列出队列
- 控制任务生命周期（暂停/恢复/取消/重试）
- 调整任务配置（优先级/依赖/调度）
- 维护任务系统（统计/清理/导入导出/队列状态）

### 1.2 工具数量与设计原则

| 版本 | 工具数 | 状态 |
| --- | --- | --- |
| v1.0（原始设计） | 7 | 已废弃 |
| v1.5（实现膨胀） | 20 | 工具过细，LLM 选择负担大 |
| **v2.0（本期）** | **6** | 按"动词 + action 枚举"二段命名，工具最简 |

**设计原则**：

1. **动词即工具名**：核心 CRUD 与控制动作各自独立工具（`create/query/update/control/decompose`）
2. **同类操作走 action 枚举**：所有"管理类"动作合并到 `task_manage.action`
3. **必填项最小化**：单条/批量、单查/列表/详情都用同一工具，参数互斥
4. **LLM 不感知错误码以外的低层细节**：调度器、执行器、超时对 LLM 透明

### 1.3 架构

```
┌─────────────────────────────────────────┐
│              LLM 大脑                   │
│  6 个工具：                              │
│  task_create / task_query               │
│  task_update / task_control             │
│  task_decompose / task_manage           │
└──────────────┬──────────────────────────┘
               │ JSON-RPC 工具调用
               ▼
┌─────────────────────────────────────────┐
│           任务管理工具层                 │
│  6 工具 → 统一 dispatch → TaskManager   │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│           任务调度器                     │
│  优先级队列 / 依赖管理 / 并发控制 / 超时 │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│           任务执行器                     │
│  simple / composite / loop / conditional│
└─────────────────────────────────────────┘
```

## 2. 工具清单（6 个）

| # | 工具 | 用途 | 关键入参 | 主要返回 |
| --- | --- | --- | --- | --- |
| 1 | `task_create` | 创建单个/批量任务 | `name` 或 `items[]` | `task_id` / `task_ids[]` |
| 2 | `task_query` | 查任务详情/进度/列表 | `mode` + 过滤 | `task` / `progress` / `tasks[]` |
| 3 | `task_update` | 更新任务属性 | `task_id` + 字段 | void |
| 4 | `task_control` | 暂停/恢复/取消/重试 | `task_id` + `action` | void |
| 5 | `task_decompose` | LLM 分解复杂任务 | `task_description` | `subtasks[]` |
| 6 | `task_manage` | 配置/统计/清理/导入导出/队列 | `action` + 动作参数 | 视 action 而定 |

## 3. 任务数据结构

与 v1.0 保持一致（详见 §3.1）。实现层字段全部继承，本期不破坏既有 Task 类型。

### 3.1 Task 类型（核心字段）

```typescript
interface Task {
  id: string;
  workspaceId: string;
  name: string;
  description?: string;
  type: 'simple' | 'composite' | 'loop' | 'conditional';
  status: 'pending' | 'running' | 'paused' | 'completed' | 'failed' | 'cancelled';
  progress: number;            // 0-100

  // 任务内容
  action?: { toolName: string; parameters: Record<string, any> };
  subtaskIds?: string[];
  loopConfig?: { mode: 'count' | 'interval' | 'condition'; count?: number; interval?: number; condition?: string; maxIterations?: number };
  condition?: { type: 'time' | 'event' | 'state' | 'expression'; value: any; description?: string };

  // 任务控制
  priority: 'critical' | 'high' | 'normal' | 'low';
  dependencies?: string[];
  timeout?: number;
  retryConfig?: { maxRetries: number; retryDelay: number; backoffMultiplier?: number };

  // 结果
  result?: any;
  error?: string;
  retryCount: number;

  // 时间
  createdAt: number;
  startedAt?: number;
  completedAt?: number;
  updatedAt: number;

  // 元数据
  tags?: string[];
  metadata?: Record<string, any>;
}
```

## 4. 工具详细设计

> 所有工具的参数规范遵循 `00-工具协议规范.md §2.2`；返回格式遵循 `§3`；错误码遵循 `§4`。

### 4.1 task_create

**功能**：创建任务，**单条与批量共用一个工具**——`items` 不传时按单条模式处理，传时按批量模式处理。

**ParameterSchema**：

| 参数名 | 类型 | 必填 | 默认 | 描述 |
| --- | --- | --- | --- | --- |
| **单条模式**（`items` 不传） | | | | |
| `name` | string | 是 | — | 任务名称 |
| `type` | string | 是 | — | `'simple'` / `'composite'` / `'loop'` / `'conditional'` |
| `description` | string | 否 | — | 任务描述 |
| `action` | object | 否 | — | 简单/循环/条件任务：`{ toolName, parameters }` |
| `subtask_ids` | array | 否 | — | 复合任务：子任务 ID 列表 |
| `loop_config` | object | 否 | — | 循环：`{ mode, count?, interval?, condition?, maxIterations? }` |
| `condition` | object | 否 | — | 条件：`{ type, value, description? }` |
| `priority` | string | 否 | `'normal'` | `'critical'` / `'high'` / `'normal'` / `'low'` |
| `dependencies` | array | 否 | — | 依赖的任务 ID 列表 |
| `timeout` | number | 否 | `300` | 超时时间（秒） |
| `retry_config` | object | 否 | — | 重试：`{ maxRetries, retryDelay, backoffMultiplier? }` |
| `tags` | array | 否 | — | 标签 |
| `metadata` | object | 否 | — | 自定义元数据 |
| **批量模式**（`items` 传入） | | | | |
| `items` | array | 是 | — | 任务列表，每项含 `name` + `type` + 上述可选字段 |

**ResultEnvelope**：

```typescript
ResultEnvelope<{
  task_id?: string;      // 单条模式
  task_ids?: string[];   // 批量模式
  count: number;         // 创建数量（1 或 items.length）
}>
```

**ErrorCode**：

| 错误码 | 含义 | 触发条件 |
| --- | --- | --- |
| `INVALID_PARAMS` | 参数无效 | `name` 空、批量中任意项缺 `name`/`type`、`items` 同时与单条字段传 |
| `TASK_TYPE_MISMATCH` | 类型不匹配 | 简单任务无 `action`、复合任务无 `subtask_ids` 等 |
| `DEPENDENCY_CYCLE` | 依赖环 | 批量创建导致循环依赖 |
| `INTERNAL_ERROR` | 内部错误 | 未预期的运行时错误 |

**示例**：

```jsonc
// 单条
task_create({
  name: "挖 10 个钻石",
  type: "simple",
  action: { toolName: "mine_block", parameters: { target: "diamond_ore", count: 10 } },
  priority: "high",
  timeout: 600,
  retry_config: { maxRetries: 3, retryDelay: 30 },
  tags: ["挖矿", "钻石"]
})

// 批量
task_create({
  items: [
    { name: "收集木材", type: "simple", action: { toolName: "collect", parameters: { target: "wood", count: 64 } } },
    { name: "建造墙壁", type: "simple", action: { toolName: "build", parameters: { blueprint: "wall" } }, dependencies: ["收集木材"] }
  ]
})
```

### 4.2 task_query

**功能**：查询任务。通过 `mode` 参数复用同一工具：取详情 / 进度 / 列表。

**ParameterSchema**：

| 参数名 | 类型 | 必填 | 默认 | 描述 |
| --- | --- | --- | --- | --- |
| `mode` | string | 否 | `'list'` | `'detail'` / `'progress'` / `'list'` |
| **detail 模式** | | | | |
| `task_id` | string | 是 | — | 任务 ID |
| **progress 模式** | | | | |
| `task_id` | string | 是 | — | 任务 ID |
| **list 模式** | | | | |
| `filter` | object | 否 | — | 过滤条件（见下） |
| `limit` | number | 否 | `50` | 返回上限 |
| `offset` | number | 否 | `0` | 偏移量 |
| `sort_by` | string | 否 | `'created_at'` | `'priority'` / `'created_at'` / `'updated_at'` / `'progress'` |
| `sort_dir` | string | 否 | `'desc'` | `'asc'` / `'desc'` |
| `filter.id` | string | 否 | — | 按 ID 精确查找 |
| `filter.status` | string \| array | 否 | — | 状态过滤 |
| `filter.type` | string \| array | 否 | — | 类型过滤 |
| `filter.priority` | string \| array | 否 | — | 优先级过滤 |
| `filter.tags` | array | 否 | — | 标签过滤（任意匹配） |
| `filter.workspace_id` | string | 否 | — | 工作区过滤 |

**ResultEnvelope**：

```typescript
// detail 模式
ResultEnvelope<{ task: Task }>
// progress 模式
ResultEnvelope<{ task_id: string; progress: number; status: TaskStatus; startedAt?: number; completedAt?: number; eta?: number }>
// list 模式
ResultEnvelope<{ tasks: Task[]; total: number; limit: number; offset: number }>
```

**ErrorCode**：

| 错误码 | 含义 | 触发条件 |
| --- | --- | --- |
| `TASK_NOT_FOUND` | 任务不存在 | `task_id` 在 detail/progress 模式下未找到 |
| `INVALID_PARAMS` | 参数无效 | `mode` 非法、过滤条件格式错误 |
| `INTERNAL_ERROR` | 内部错误 | 未预期的运行时错误 |

**示例**：

```jsonc
// 详情
task_query({ mode: "detail", task_id: "task_001" })

// 进度
task_query({ mode: "progress", task_id: "task_001" })
// → { progress: 50, status: "running", startedAt: 1719820000, eta: 1719820600 }

// 列表
task_query({ mode: "list", filter: { status: ["pending", "running"], priority: "high" }, limit: 10, sort_by: "priority" })
```

### 4.3 task_update

**功能**：更新任务元数据。**不动** status / progress（这两个由 task_control 或执行器维护）。

**ParameterSchema**：

| 参数名 | 类型 | 必填 | 默认 | 描述 |
| --- | --- | --- | --- | --- |
| `task_id` | string | 是 | — | 任务 ID |
| `name` | string | 否 | — | 新名称 |
| `description` | string | 否 | — | 新描述 |
| `priority` | string | 否 | — | 新优先级（也可用 `task_manage.action='priority'`） |
| `timeout` | number | 否 | — | 新超时时间（秒） |
| `retry_config` | object | 否 | — | 新重试配置 |
| `tags` | array | 否 | — | 新标签（整体替换） |
| `metadata` | object | 否 | — | 自定义元数据（整体替换；深度合并可选） |

**ResultEnvelope**：

```typescript
ResultEnvelope<{ updated_fields: string[] }>
```

**ErrorCode**：

| 错误码 | 含义 | 触发条件 |
| --- | --- | --- |
| `TASK_NOT_FOUND` | 任务不存在 | `task_id` 未找到 |
| `INVALID_PARAMS` | 参数无效 | 字段值不合法、必填字段未传 |
| `TASK_RUNNING_IMMUTABLE` | 任务运行中 | 运行时不允许改 `timeout`（其它字段允许） |
| `INTERNAL_ERROR` | 内部错误 | 未预期的运行时错误 |

**示例**：

```jsonc
task_update({
  task_id: "task_001",
  priority: "high",
  tags: ["挖矿", "钻石", "夜间"],
  metadata: { current_location: { x: 128, y: 20, z: -256 } }
})
// → { updated_fields: ["priority", "tags", "metadata"] }
```

### 4.4 task_control

**功能**：控制任务生命周期。

**ParameterSchema**：

| 参数名 | 类型 | 必填 | 默认 | 描述 |
| --- | --- | --- | --- | --- |
| `task_id` | string | 是 | — | 任务 ID |
| `action` | string | 是 | — | `'pause'` / `'resume'` / `'cancel'` / `'retry'` |
| `reason` | string | 否 | — | 原因说明（cancel/pause 建议传） |
| `force` | boolean | 否 | `false` | 强制操作（仅 retry：忽略依赖失败检查） |

**ResultEnvelope**：

```typescript
ResultEnvelope<{ task_id: string; action: string; prev_status: TaskStatus; new_status: TaskStatus }>
```

**ErrorCode**：

| 错误码 | 含义 | 触发条件 |
| --- | --- | --- |
| `TASK_NOT_FOUND` | 任务不存在 | `task_id` 未找到 |
| `INVALID_STATE_TRANSITION` | 状态非法跳转 | 暂停已暂停的任务、恢复未暂停的任务、重试成功的任务等 |
| `DEPENDENCY_FAILED` | 依赖未满足 | retry 时上游依赖仍处于非完成态（`force=true` 跳过） |
| `INTERNAL_ERROR` | 内部错误 | 未预期的运行时错误 |

**示例**：

```jsonc
task_control({ task_id: "task_001", action: "pause" })
task_control({ task_id: "task_002", action: "cancel", reason: "资源不足" })
task_control({ task_id: "task_003", action: "retry", force: true })
```

### 4.5 task_decompose

**功能**：让 LLM 自主把复杂任务分解为可执行的子任务。

**ParameterSchema**：

| 参数名 | 类型 | 必填 | 默认 | 描述 |
| --- | --- | --- | --- | --- |
| `task_description` | string | 是 | — | 任务描述（要说清目标） |
| `context` | object | 否 | — | 上下文：位置、资源、约束等 |
| `max_subtasks` | number | 否 | `10` | 最多分解多少个子任务（防止 LLM 过度拆分） |
| `strategy` | string | 否 | `'sequential'` | `'sequential'` / `'parallel'` / `'mixed'` |

**ResultEnvelope**：

```typescript
ResultEnvelope<{
  subtasks: Array<{
    name: string;
    description: string;
    type: 'simple' | 'composite' | 'loop' | 'conditional';
    action?: { toolName: string; parameters: Record<string, any> };
    dependencies?: string[];        // 引用其它 subtask.name
    estimated_duration?: number;    // 秒
  }>;
  summary: string;                  // 总体思路摘要（≤ 200 字）
}>
```

**ErrorCode**：

| 错误码 | 含义 | 触发条件 |
| --- | --- | --- |
| `INVALID_PARAMS` | 参数无效 | `task_description` 为空 |
| `DECOMPOSE_TOO_COMPLEX` | 太复杂 | LLM 返回的子任务数 > `max_subtasks` |
| `INTERNAL_ERROR` | 内部错误 | 分解失败 |

**示例**：

```jsonc
task_decompose({
  task_description: "建造一个村民交易所",
  context: { location: { x: 100, y: 64, z: -200 }, resources: ["wood", "stone"], budget: 1000 },
  strategy: "mixed"
})
// → { subtasks: [...4 个子任务], summary: "收集 → 框架 → 捕获 → 交易站" }
```

### 4.6 task_manage

**功能**：所有"管理类"动作的合并入口。**9 种 action 收敛到一个工具**，降低 LLM 工具选择负担。

**ParameterSchema**：

| 参数名 | 类型 | 必填 | 默认 | 描述 |
| --- | --- | --- | --- | --- |
| `action` | string | 是 | — | 见下表 9 种 action |
| `task_id` | string | 视 action | — | 任务 ID（单任务类 action 必填） |
| `priority` | string | 视 action | — | 优先级（仅 `action='priority'`） |
| `depends_on_id` | string | 视 action | — | 依赖任务 ID（仅 `add_dep` / `remove_dep`） |
| `mode` | string | 视 action | — | 调度模式（仅 `action='schedule'`） |
| `delay` | number | 视 action | — | 延迟秒数（仅 `mode='delayed'`） |
| `cron` | string | 视 action | — | cron 表达式（仅 `mode='cron'`） |
| `event` | string | 视 action | — | 事件名（仅 `mode='event'`） |
| `keep_recent` | number | 视 action | `100` | 保留最近 N 条（仅 `action='cleanup'`） |
| `older_than` | number | 视 action | `604800` | 清理早于该秒数（仅 `action='cleanup'`） |
| `status_filter` | array | 视 action | — | 状态过滤（仅 `cleanup`） |
| `format` | string | 视 action | `'json'` | `'json'` / `'csv'`（仅 `action='export'`） |
| `data` | string | 视 action | — | JSON 字符串（仅 `action='import'`） |
| `export_filter` | object | 视 action | — | 导出过滤（仅 `action='export'`） |

**Action 列表**：

| action | 必填附加参数 | 用途 | 返回 |
| --- | --- | --- | --- |
| `stats` | — | 统计：总数/各状态分布/完成率/平均耗时 | `{ total, byStatus, byType, byPriority, completionRate, avgDuration }` |
| `cleanup` | `keep_recent`/`older_than` | 清理已完成/失败/取消的任务 | `{ cleaned: number }` |
| `export` | `format`/`export_filter` | 导出任务为 JSON/CSV | `{ data: string, count: number }` |
| `import` | `data` | 从 JSON 导入 | `{ imported: number, skipped: number }` |
| `priority` | `task_id`/`priority` | 修改优先级（也可走 task_update） | void |
| `add_dep` | `task_id`/`depends_on_id` | 添加依赖 | void |
| `remove_dep` | `task_id`/`depends_on_id` | 移除依赖 | void |
| `schedule` | `task_id`/`mode`/... | 调度任务 | void |
| `queue_status` | — | 查看调度队列当前状态 | `{ pending, running, completed, failed, concurrency, oldest_pending_age }` |

**ResultEnvelope**：

```typescript
ResultEnvelope<{ result?: any }>     // 视 action 不同
```

**ErrorCode**：

| 错误码 | 含义 | 触发条件 |
| --- | --- | --- |
| `INVALID_PARAMS` | 参数无效 | `action` 非法、必填附加参数缺失 |
| `TASK_NOT_FOUND` | 任务不存在 | `task_id` 在需要任务 ID 的 action 下未找到 |
| `DEPENDENCY_CYCLE` | 依赖环 | add_dep 时形成环 |
| `IMPORT_FORMAT_INVALID` | 导入格式错 | import 时 JSON 解析失败或 schema 不符 |
| `INTERNAL_ERROR` | 内部错误 | 未预期的运行时错误 |

**示例**：

```jsonc
// 统计
task_manage({ action: "stats" })
// → { total: 10, byStatus: { pending: 3, running: 2, completed: 4, failed: 1 }, completionRate: 0.4, avgDuration: 4500 }

// 清理
task_manage({ action: "cleanup", keep_recent: 100, older_than: 604800, status_filter: ["completed", "cancelled"] })
// → { cleaned: 3 }

// 修改优先级
task_manage({ action: "priority", task_id: "task_002", priority: "high" })

// 添加依赖
task_manage({ action: "add_dep", task_id: "task_003", depends_on_id: "task_001" })

// 调度
task_manage({ action: "schedule", task_id: "task_004", mode: "delayed", delay: 300 })
task_manage({ action: "schedule", task_id: "task_005", mode: "cron", cron: "0 */5 * * *" })
task_manage({ action: "schedule", task_id: "task_006", mode: "event", event: "night_fall" })

// 队列状态
task_manage({ action: "queue_status" })
// → { pending: 3, running: 2, completed: 4, failed: 1, concurrency: 3, oldest_pending_age: 120000 }

// 导出
task_manage({ action: "export", format: "json", export_filter: { status: "completed" } })
// → { data: "[{...}, ...]", count: 4 }

// 导入
task_manage({ action: "import", data: "[{...}]" })
// → { imported: 4, skipped: 0 }
```

## 5. 任务调度器实现

### 5.1 优先级队列

```typescript
const PRIORITY_ORDER = { critical: 0, high: 1, normal: 2, low: 3 } as const;

class TaskScheduler {
  private taskQueue: PriorityQueue<Task>;
  private runningTasks = new Map<string, Task>();
  private maxConcurrent = 3;  // 默认并发上限

  constructor() {
    this.taskQueue = new PriorityQueue<Task>((a, b) => {
      const diff = PRIORITY_ORDER[a.priority] - PRIORITY_ORDER[b.priority];
      return diff !== 0 ? diff : a.createdAt - b.createdAt;  // 相同优先级按创建时间
    });
  }

  async enqueue(task: Task): Promise<void> {
    if (task.dependencies?.length) {
      const unmet = await this.checkDependencies(task.dependencies);
      if (unmet.length > 0) {
        task.metadata = { ...task.metadata, waiting_for: unmet };
      }
    }
    this.taskQueue.enqueue(task);
  }

  async schedule(): Promise<void> {
    while (true) {
      if (this.runningTasks.size < this.maxConcurrent && !this.taskQueue.isEmpty()) {
        const task = this.taskQueue.peek();
        const unmet = await this.checkDependencies(task.dependencies ?? []);
        if (unmet.length > 0) {
          this.taskQueue.dequeue();
          this.taskQueue.enqueue(task);  // 放回队尾
          continue;
        }
        this.taskQueue.dequeue();
        this.executeTask(task);
      }
      await sleep(1000);  // 1s 轮询
    }
  }

  private async executeTask(task: Task): Promise<void> {
    task.status = 'running';
    task.startedAt = Date.now();
    this.runningTasks.set(task.id, task);

    try {
      const result = await this.dispatchToExecutor(task);
      task.status = 'completed';
      task.progress = 100;
      task.result = result;
      task.completedAt = Date.now();
      await this.triggerDependentTasks(task.id);
    } catch (err) {
      task.error = (err as Error).message;
      if (task.retryConfig && task.retryCount < task.retryConfig.maxRetries) {
        task.retryCount++;
        task.status = 'pending';
        const delay = task.retryConfig.retryDelay *
          (task.retryConfig.backoffMultiplier ?? 1) ** (task.retryCount - 1);
        setTimeout(() => this.enqueue(task), delay * 1000);
      } else {
        task.status = 'failed';
        task.completedAt = Date.now();
      }
    } finally {
      this.runningTasks.delete(task.id);
    }
  }

  private dispatchToExecutor(task: Task): Promise<any> {
    switch (task.type) {
      case 'simple':      return simpleExecutor.run(task);
      case 'composite':   return compositeExecutor.run(task);
      case 'loop':        return loopExecutor.run(task);
      case 'conditional': return conditionalExecutor.run(task);
    }
  }
}
```

### 5.2 4 种执行器

| 执行器 | 职责 | 关键点 |
| --- | --- | --- |
| `simple-executor` | 调用 action.toolName | 直接走 V20 的 BatchToolDispatcher 通道 |
| `composite-executor` | 顺序/并行执行子任务 | 按 `subtaskIds` 依赖关系调度；维护父任务进度 |
| `loop-executor` | 按 loop_config 循环 | 严格遵守 `maxIterations` 防止死循环 |
| `conditional-executor` | 等待 condition 满足 | 周期检查 condition；条件 type=state 时调 `task_query` 查状态 |

## 6. 使用场景

### 场景 1：复杂任务分解

```
LLM 接收: "建造一个村民交易所"
  ↓
LLM 调用: task_decompose({ task_description: "建造村民交易所" })
  ↓
返回 4 个子任务设计（不创建实体任务）
  ↓
LLM 调用: task_create({ name: "建造村民交易所", type: "composite", subtask_ids: [...] })
  ↓
LLM 再调用: task_create({ items: [...4个子任务的描述...] })
  ↓
调度器按依赖关系自动执行
  ↓
LLM 调: task_query({ mode: "progress", task_id: "..." }) 跟踪进度
```

### 场景 2：循环巡逻

```
LLM 调用: task_create({
  name: "巡逻基地",
  type: "loop",
  action: { toolName: "move_to", parameters: { path: "patrol_route" } },
  loop_config: { mode: "interval", interval: 300, maxIterations: 100 },
  priority: "low"
})
  ↓
调度器每 5 分钟执行一次巡逻
```

### 场景 3：条件触发

```
LLM 调用: task_create({
  name: "饥饿时吃东西",
  type: "conditional",
  action: { toolName: "eat", parameters: {} },
  condition: { type: "state", value: "hunger < 10" }
})
  ↓
调度器持续监控饥饿值
```

### 场景 4：批量与依赖

```
LLM 调用: task_create({ items: [
  { name: "收集木材", type: "simple", action: {...} },
  { name: "建造墙壁", type: "simple", dependencies: ["收集木材"] },
  { name: "建造屋顶", type: "simple", dependencies: ["建造墙壁"] }
]})
  ↓
返回 { task_ids: ["task_a", "task_b", "task_c"] }
  ↓
LLM 调: task_manage({ action: "queue_status" }) 确认调度状态
  ↓
过程中需要中止某个: task_control({ task_id: "task_b", action: "cancel", reason: "..." })
```

### 场景 5：维护

```
LLM 调: task_manage({ action: "stats" })          → 看完成率
LLM 调: task_manage({ action: "cleanup", older_than: 604800 })  → 清理一周前的已完成
LLM 调: task_manage({ action: "export", format: "json" })  → 备份
```

## 8. 错误码体系（补 v1.0 缺口）

| 错误码 | 含义 | 触发场景 |
| --- | --- | --- |
| `INVALID_PARAMS` | 参数无效 | 所有工具的入参校验失败 |
| `TASK_NOT_FOUND` | 任务不存在 | task_id 引用未找到 |
| `TASK_TYPE_MISMATCH` | 类型不匹配 | 必填字段与 type 不一致（如 simple 缺 action） |
| `INVALID_STATE_TRANSITION` | 状态非法跳转 | task_control 触发非法状态机转换 |
| `TASK_RUNNING_IMMUTABLE` | 任务运行中 | 修改运行时不可变字段 |
| `DEPENDENCY_FAILED` | 依赖未满足 | retry 时上游未完成（force=true 跳过） |
| `DEPENDENCY_CYCLE` | 依赖环 | 添加/创建导致循环依赖 |
| `DECOMPOSE_TOO_COMPLEX` | 分解过深 | decompose 返回子任务 > max_subtasks |
| `IMPORT_FORMAT_INVALID` | 导入格式错 | import 时 JSON 解析失败 |
| `INTERNAL_ERROR` | 内部错误 | 未预期的运行时错误 |

> 错误码定义需在 `00-工具协议规范.md §4` 注册；本期新增 8 个（v1.0 仅 3 个）。
