## 第1章 概述

### 1.1 QQ 机器人的角色

QQ 机器人是 McAgent Agent Core 的**远程交互入口**，允许用户通过 QQ 群聊或私聊与 Minecraft 智能体进行交互。它在系统中承担三个核心角色：

| 角色 | 说明 |
|------|------|
| **远程控制** | 通过 QQ 消息向智能体下发指令，如移动、挖掘、建造、战斗等操作，无需直接操作游戏 |
| **状态查询** | 实时查询智能体的状态信息，包括位置、血量、饥饿度、背包内容、当前任务进度等 |
| **任务下发** | 以自然语言描述任务目标，由 LLM 抽象层自动解析并分解为可执行的任务序列 |

### 1.2 体系位置

QQ 机器人模块位于 Agent Core 内部，作为 LLM 抽象层的外部输入源之一。消息处理路径如下：

```
┌──────────┐    群聊/私聊消息    ┌──────────────────────────────────────┐
│  QQ 用户  │ ──────────────────▶ │           Agent Core                 │
└──────────┘                     │                                      │
                                 │  ┌────────────┐  ┌───────────────┐   │
┌──────────┐                     │  │ QQ Bot     │  │ LLM 抽象层     │   │
│ QQ 群聊   │ ──────────────────▶ │  │ 模块        │─▶│ (多 Provider)  │   │
└──────────┘                     │  │ (OneBot)   │  │ 提示词组装     │   │
                                 │  └──────┬─────┘  │ Function      │   │
                                 │         │         │ Calling 解析   │   │
                                 │         │ 消息     └───────┬───────┘   │
                                 │         │ 桥接              │           │
                                 │         ▼                   ▼           │
                                 │  ┌─────────────────────────────────┐   │
                                 │  │      TCP JSON-RPC 桥接层        │   │
                                 │  └──────────────┬──────────────────┘   │
                                 └─────────────────┼──────────────────────┘
                                                   │
                                          TCP 长连接 │ JSON-RPC 2.0
                                                   │
                                          ┌────────▼──────────┐
                                          │   Adapter Core    │
                                          │  (游戏内模组/插件) │
                                          │  假人 · 工具集    │
                                          └───────────────────┘
```

**消息处理流程说明：**

1. 用户在 QQ 中发送消息（群聊或私聊）
2. QQ Bot 模块通过 OneBot WebSocket 协议接收到消息
3. 模块对消息进行鉴权（检查用户权限、群白名单）
4. 消息分两条路径处理：
   - **指令消息**（以 `/` 开头）：由内置命令解析器直接处理，快速响应
   - **自然语言消息**（非指令）：通过消息桥接层转发给 LLM 抽象层
5. LLM 抽象层处理后的结果（或内置命令的执行结果）通过 QQ Bot 模块回复给用户

### 1.3 支持的交互方式

| 交互方式 | 说明 | 适用场景 |
|----------|------|----------|
| **群聊** | 在允许的 QQ 群中 @机器人 或发送命令 | 团队协作、多人查看智能体状态 |
| **私聊** | 直接与机器人账号对话，无需 @ | 个人远程控制、管理员管理操作 |
| **指令** | 以 `/` 为前缀的特殊命令，不经过 LLM | 快速查询状态、取消任务、获取帮助 |

## 第2章 前置准备

### 2.1 QQ 机器人平台

McAgent QQ 机器人模块基于 **OneBot 协议**（v11）实现，目前支持的标准化方案为 **LLOneBot**（基于 NapCat）。

| 项目 | 说明 |
|------|------|
| **协议标准** | OneBot v11（正向 WebSocket） |
| **推荐方案** | NapCat v2.x |
| **备选方案** | 任何兼容 OneBot v11 正向 WebSocket 的框架（如 Lagrange、go-cqhttp） |

**NapCat 简介：**

NapCat 是一个轻量级的 QQ 机器人协议实现，基于 Modern Web 技术构建。它将 QQ 账号包装为标准 OneBot 服务，通过 WebSocket 对外暴露消息收发接口。相比传统方案，NapCat 具有以下优势：

- 无需手机挂载，纯服务端运行
- 支持 Windows / Linux / Docker 多平台部署
- WebUI 管理界面，配置可视化
- 性能开销低，内存占用小

**安装方式：**

| 方式 | 适用场景 | 推荐度 |
|------|----------|--------|
| Windows 直接运行 | 开发和测试环境 | ⭐⭐⭐ |
| Docker 部署 | 生产环境 / Linux 服务器 | ⭐⭐⭐⭐⭐ |
| Linux 直接部署 | 无 Docker 环境 | ⭐⭐⭐ |

> **注意：** NapCat 使用 QQ 协议模拟登录，存在被腾讯风控的风险。建议使用**新注册的 QQ 小号**作为机器人账号，不要使用常用主号。

### 2.2 获取必要信息

在开始配置前，请准备好以下信息：

| 信息 | 说明 | 获取方式 |
|------|------|----------|
| **QQ 机器人账号** | 用于作为机器人的 QQ 号（建议新注册小号） | 注册新 QQ 号或使用闲置账号 |
| **管理员 QQ 号列表** | 拥有完整管理权限的 QQ 号 | 记录你本人的 QQ 号 |
| **允许的群组列表** | 允许机器人发言和响应的 QQ 群号 | 从 QQ 群设置中获取群号 |
| **访问令牌**（可选） | WebSocket 连接的鉴权 Token | 自定义字符串，建议使用 UUID |

## 第3章 NapCat 部署

### 3.1 Windows 部署

**步骤 1：下载 NapCat**

从 NapCat 官方仓库（[https://github.com/NapNeko/NapCatQQ](https://github.com/NapNeko/NapCatQQ)）下载最新发布的 Windows 版本安装包，或使用一键安装脚本：

```powershell
# 使用 PowerShell 运行一键安装脚本
iwr -Uri "https://nclatest.znin.net/NapNeko/NapCat-Installer/main/scripts/install.ps1" -OutFile install.ps1
.\install.ps1
```

**步骤 2：运行安装向导**

按照安装向导的提示完成以下配置：

1. 选择安装目录（建议：`C:\NapCat` 或 `D:\NapCat`）
2. 选择 Shell 模式（推荐：WebSocket）
3. 配置 WebSocket 端口（默认：`3001`）
4. 设置访问令牌（可选，但建议设置以防止未授权访问）

**步骤 3：登录 QQ 账号**

启动 NapCat 后，在终端中会弹出二维码登录界面：

1. 使用 QQ 手机客户端扫描二维码
2. 确认登录
3. 登录成功后 NapCat 会自动保存会话 Token，后续重启无需重复扫码

**步骤 4：确认 WebSocket 服务运行**

NapCat 启动后，确认以下服务状态：

- WebSocket 服务已监听在配置的端口（默认 `3001`）
- 日志中无连接错误
- 可通过 WebUI 访问 NapCat 管理界面（默认地址：`http://127.0.0.1:6099`）

### 3.2 Docker 部署（推荐）

Docker 部署更加稳定且易于管理，推荐在生产环境中使用。

**docker-compose.yml：**

```yaml
version: '3.8'

services:
  napcat:
    image: mlikiowa/napcat-docker:latest
    container_name: mcagent-napcat
    restart: unless-stopped
    environment:
      - ACCOUNT=<你的机器人QQ号>     # 必填：机器人 QQ 号
      - WS_PORT=3001                  # WebSocket 端口
      - WEBUI_PORT=6099               # WebUI 管理端口
      - ACCESS_TOKEN=<访问令牌>       # 可选：WebSocket 鉴权 Token
    ports:
      - "3001:3001"                   # WebSocket 端口映射
      - "6099:6099"                   # WebUI 端口映射
    volumes:
      - ./napcat/config:/usr/src/app/napcat/config  # 配置文件持久化
      - ./napcat/data:/usr/src/app/napcat/data      # 数据持久化
```

**启动 NapCat 容器：**

```bash
docker-compose up -d
```

**首次登录：**

1. 查看容器日志，获取登录二维码
2. 使用 QQ 手机客户端扫码登录
3. 登录成功后容器会自动保存会话

```bash
# 查看容器日志，扫描二维码登录
docker logs -f mcagent-napcat
```

**重要端口说明：**

| 端口 | 用途 | 说明 |
|------|------|------|
| `3001` | WebSocket 端口 | Agent Core QQ 模块通过此端口连接 |
| `6099` | WebUI 端口 | NapCat 管理后台，浏览器访问 |
| `3000` | HTTP API 端口 | OneBot HTTP 接口（当前未使用） |

### 3.3 验证 NapCat 运行

NapCat 部署完成后，通过以下方式验证运行状态：

**方法 1：访问 WebUI**

在浏览器中打开 `http://<部署IP>:6099`：

- 登录 WebUI 管理界面（默认无密码，建议在面板中设置管理密码）
- 查看「连接状态」页面，确认 WebSocket 服务处于「运行中」状态
- 查看「日志」页面，确认无报错信息

**方法 2：检查 WebSocket 连接**

NapCat WebSocket 端点地址为：
```
ws://<部署IP>:3001
```

可以使用 WebSocket 测试工具（如 `wscat`、浏览器 WebSocket 插件）验证连通性：

```bash
# 使用 wscat 测试 WebSocket 连接
wscat -c ws://127.0.0.1:3001
```

如果配置了 `access_token`，连接时需要添加鉴权头：

```json
{
  "headers": {
    "Authorization": "Bearer <你的访问令牌>"
  }
}
```

**方法 3：检查日志输出**

NapCat 日志中应包含以下关键信息：

```
[NapCat] WebSocket 服务已启动，端口: 3001
[NapCat] QQ 账号 <账号> 已登录
[NapCat] 正向 WebSocket 服务已就绪
```

## 第4章 Agent Core QQ 模块配置

### 4.1 配置文件

QQ 模块的配置文件位于 Agent Core 的 `config.json` 中，整体配置结构如下：

```json
{
  "qq_bot": {
    "enabled": true,
    "platform": "napcat",
    "napcat": {
      "ws_host": "127.0.0.1",
      "ws_port": 3001,
      "ws_protocol": "ws",
      "access_token": ""
    },
    "authorization": {
      "admins": ["123456789"],
      "allowed_groups": ["987654321"],
      "allow_private": true
    },
    "behavior": {
      "reply_prefix": "[McAgent] ",
      "max_history": 20,
      "cooldown_seconds": 3,
      "auto_mention": true
    }
  }
}
```

**配置字段说明：**

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `qq_bot.enabled` | boolean | `false` | 是否启用 QQ 机器人模块 |
| `qq_bot.platform` | string | `"napcat"` | QQ 机器人平台类型（当前仅支持 napcat） |
| `qq_bot.napcat.ws_host` | string | `"127.0.0.1"` | NapCat WebSocket 服务地址 |
| `qq_bot.napcat.ws_port` | number | `3001` | NapCat WebSocket 服务端口 |
| `qq_bot.napcat.ws_protocol` | string | `"ws"` | WebSocket 协议（ws 或 wss） |
| `qq_bot.napcat.access_token` | string | `""` | WebSocket 鉴权 Token（与 NapCat 配置一致） |
| `qq_bot.authorization.admins` | string[] | `[]` | 管理员 QQ 号列表 |
| `qq_bot.authorization.allowed_groups` | string[] | `[]` | 允许机器人响应的群组列表 |
| `qq_bot.authorization.allow_private` | boolean | `true` | 是否允许私聊 |
| `qq_bot.behavior.reply_prefix` | string | `"[McAgent] "` | 回复消息前缀 |
| `qq_bot.behavior.max_history` | number | `20` | 每个会话保留的最大消息历史数 |
| `qq_bot.behavior.cooldown_seconds` | number | `3` | 消息响应冷却时间（秒） |
| `qq_bot.behavior.auto_mention` | boolean | `true` | 是否自动 @ 回复者 |

### 4.2 权限配置

QQ 模块的权限系统采用三级权限模型：

| 权限等级 | 角色 | 允许操作 |
|----------|------|----------|
| **管理员** | `admins` 列表中的 QQ 号 | 所有操作，包括执行敏感命令、管理机器人、修改配置 |
| **群成员** | 白名单群组中的普通成员 | 受限操作（如 `/status`、`/position` 等查询类命令） |
| **无权限** | 非白名单用户或群组 | 不响应任何消息 |

**配置示例：**

```json
{
  "qq_bot": {
    "authorization": {
      "admins": ["100001", "100002"],
      "allowed_groups": ["200001", "200002"],
      "allow_private": true
    }
  }
}
```

**关于 `allowed_groups` 和 `allow_private` 的交互行为：**

| `allowed_groups` | `allow_private` | 群聊行为 | 私聊行为 |
|------------------|-----------------|----------|----------|
| 有值 | `true` | 仅允许指定群 | 所有私聊用户均可 |
| 有值 | `false` | 仅允许指定群 | 拒绝所有私聊 |
| 空数组 | `true` | 不响应任何群聊 | 所有私聊用户均可 |
| 空数组 | `false` | 不响应任何群聊 | 拒绝所有私聊 |

**黑白名单模式：**

当前权限系统基于**白名单模式**（默认拒绝，仅允许列表中的用户/群组）。如需切换到黑名单模式，可将 `allowed_groups` 设置为一个通配符值（如 `["*"]`），并在应用层增加黑名单过滤列表（此功能将在后续版本中内置）。

### 4.3 安全配置

**1. access_token 鉴权**

`access_token` 用于 Agent Core 与 NapCat 之间的 WebSocket 连接鉴权：

- 在 NapCat 中配置相同的 `access_token`
- Agent Core 连接 WebSocket 时自动在请求头中携带 `Authorization: Bearer <token>`
- 建议使用随机生成的 UUID 作为 Token，长度不低于 16 位

**2. 频率限制**

`cooldown_seconds` 参数控制单用户的消息响应频率：

```json
{
  "qq_bot": {
    "behavior": {
      "cooldown_seconds": 3
    }
  }
}
```

- 同一用户两次消息之间至少间隔 `cooldown_seconds` 秒
- 冷却期间的消息会被静默丢弃，不会积压
- 管理员不受频率限制约束
- 建议值：群聊 `3` 秒，高频查询场景可设为 `1` 秒

**3. 敏感操作确认**

对于涉及游戏内重大变更的操作（如 `/task cancel`、`/say`），QQ 模块要求二次确认：

- 用户发送敏感指令后，机器人回复确认消息
- 用户需在 30 秒内发送"确认执行"或"是"来完成操作
- 超时未确认则自动取消操作
- 管理员可以配置绕过确认的指令白名单

**确认流程示例：**

```
用户: /task cancel abc-123
机器人: [McAgent] 你确定要取消任务 "abc-123" 吗？（当前进度：60%）
        请在 30 秒内回复"确认执行"以确认操作。
用户: 确认执行
机器人: [McAgent] 任务 "abc-123" 已取消。
```

**4. 命令前缀配置**

默认命令前缀为 `/`，用户可以在 `behavior` 中自定义，但建议保持默认以降低用户学习成本。

## 第5章 功能与命令

### 5.1 内置命令

QQ 机器人内置了一系列快捷命令，这些命令**不经过 LLM 处理**，由模块直接解析并快速响应。

**状态查询类：**

| 命令 | 参数 | 说明 | 权限 |
|------|------|------|------|
| `/status` | 无 | 查看智能体当前状态（健康、位置、任务） | 所有用户 |
| `/inventory` | 无 | 查看智能体背包内容摘要 | 所有用户 |
| `/position` | 无 | 查看智能体当前坐标和所在维度 | 所有用户 |

**任务管理类：**

| 命令 | 参数 | 说明 | 权限 |
|------|------|------|------|
| `/task list` | 无 | 查看当前任务列表及进度 | 所有用户 |
| `/task cancel <id>` | 任务 ID | 取消指定任务（需确认） | 管理员 |

**智能体控制类：**

| 命令 | 参数 | 说明 | 权限 |
|------|------|------|------|
| `/say <message>` | 消息内容 | 让智能体在游戏中发送聊天消息 | 管理员 |

**系统类：**

| 命令 | 参数 | 说明 | 权限 |
|------|------|------|------|
| `/help` | 无 | 查看所有可用命令及说明 | 所有用户 |

**命令响应示例：**

```
用户: /status
机器人: [McAgent] 📊 智能体状态
  ├─ 血量: 18/20 ❤️
  ├─ 饥饿: 16/20 🍖
  ├─ 位置: (120, 64, -45) @ overworld
  ├─ 维度: 主世界
  ├─ 生物群系: 森林
  ├─ 当前任务: "挖掘10个铁矿石" (进行中 60%)
  └─ 智能体: 在线 ✅

用户: /task list
机器人: [McAgent] 📋 当前任务列表
  ├─ #1: "挖掘10个铁矿石" — 进行中 (6/10)
  ├─ #2: "返回基地" — 等待中
  └─ #3: "熔炼铁锭" — 等待中
```

### 5.2 自然语言交互

非命令消息会通过消息桥接层转发给 LLM 抽象层处理，由 LLM 理解用户意图并生成响应。

**处理流程：**

```
用户发送: "帮我挖点钻石"
  ↓
QQ 模块接收消息
  ↓
权限检查通过
  ↓
消息桥接层将消息格式化后发给 LLM 调度层
  ↓
LLM 抽象层：
  1. 解析用户意图 → "获取钻石矿石"
  2. 查询当前状态 → 位置、背包工具、是否有铁镐
  3. 制定计划 → 前往矿区 → 挖掘钻石 → 收集掉落
  4. 生成回复 → "好的，我先检查一下工具和位置"
  ↓
回复发送到 QQ
  ↓
LLM 开始通过 Function Calling 执行任务
  ↓
任务执行完成后，主动通知用户
```

**交互示例：**

```
用户: 帮我挖点钻石
机器人: [McAgent] 好的，我先确认一下装备和位置...
        （经过几秒）
机器人: [McAgent] 已到达 Y=11 的矿区，开始挖掘钻石矿。
        当前背包有铁镐，可以挖掘。
        （任务执行中）
机器人: [McAgent] ✅ 挖掘完成！共获得 4 个钻石矿石。
        已放入背包。

用户: 现在几点了
机器人: [McAgent] 游戏内时间是白天（第 124 天），
        你可以放心外出活动。

用户: 附近有什么危险吗
机器人: [McAgent] 我检测到周围有 3 只僵尸和 1 只骷髅，
        建议先清理它们再继续工作。需要我处理吗？
```

### 5.3 主动通知

QQ 机器人支持主动推送通知，让用户及时了解智能体的重要状态变化。

**通知类型：**

| 通知类型 | 触发条件 | 示例消息 |
|----------|----------|----------|
| **任务完成通知** | LLM 完成一个任务目标 | `[McAgent] ✅ 任务"挖掘 10 个铁矿石"已完成！` |
| **智能体死亡通知** | 智能体在游戏中死亡 | `[McAgent] 💀 警告：智能体已死亡！坐标：(-15, 57, 82)` |
| **资源不足通知** | 工具耐久度低、背包满、饥饿值低 | `[McAgent] ⚠️ 饥饿值低于 6，建议尽快进食` |
| **异常状态通知** | 被攻击、中毒、掉落伤害等 | `[McAgent] ⚠️ 智能体正在被僵尸攻击！位置：(-15, 64, 80)` |
| **连接状态通知** | 与游戏服务器的连接断开/恢复 | `[McAgent] 🔌 与游戏服务器的连接已断开，正在重连...` |

主动通知仅在配置了 `allowed_groups` 或开启私聊时有效。通知会发送到**所有白名单群组**以及**所有在线管理员**。

**通知配置：**

主动通知目前是默认启用的，不支持单独关闭（这是为防止用户错过重要事件）。如需关闭，可在后续版本中的 `behavior` 配置中增加 `notifications` 开关。

## 第7章 常见问题

### Q1: NapCat 连接失败

**现象：** Agent Core 日志显示 "WebSocket 连接失败" 或 "连接超时"。

**可能原因及解决方案：**

| 原因 | 排查方法 | 解决方案 |
|------|----------|----------|
| NapCat 未启动 | 检查 NapCat 进程/容器是否运行 | 启动 NapCat |
| 端口不匹配 | 确认 NapCat 配置的 WS_PORT 与 Agent Core 配置的 ws_port 一致 | 统一端口配置 |
| 防火墙拦截 | 在 NapCat 机器上测试 `telnet 127.0.0.1 3001` 是否通 | 添加防火墙规则 |
| 主机地址错误 | NapCat 和 Agent Core 在不同机器时使用了 127.0.0.1 | 使用实际 IP 地址 |

### Q2: WebSocket 握手失败

**现象：** Agent Core 日志显示 "WebSocket 握手失败" 或 "401 Unauthorized"。

**原因及解决方案：**

```json
// 检查两端 access_token 是否一致
// NapCat 配置：
{
  "accessToken": "your-access-token"
}

// Agent Core 配置：
{
  "qq_bot": {
    "napcat": {
      "access_token": "your-access-token"  // 必须与 NapCat 完全一致
    }
  }
}
```

### Q3: QQ 账号被封风险

**风险说明：** 使用协议模拟登录 QQ 存在一定风控风险，腾讯可能会：

- 要求滑块验证
- 限制账号功能（加群、发消息）
- 临时封禁（通常 24 小时）
- 长期封禁（较少见）

**降低风险的措施：**

1. **使用专用小号**：不要使用主号或常用号
2. **模拟正常行为**：避免短时间内大量发消息
3. **配置频率限制**：设置合理的 `cooldown_seconds`
4. **保持登录稳定**：避免频繁重连和登出
5. **使用 Docker 部署**：NapCat Docker 版本的登录态保持更稳定

### Q4: 消息频率限制

**现象：** 群聊中机器人回复延迟，或部分消息未处理。

**原因：** QQ 平台本身对机器人消息有频率限制（通常约 20 条/分钟）。

**解决方案：**

1. 降低 `cooldown_seconds` 到合理值（如 1-2 秒）
2. 避免连续发送大量消息
3. 合并多条回复为一条消息发送
4. 使用消息队列缓存待发送消息

### Q5: 权限验证失败

**现象：** 消息日志显示 "权限不足" 或 "拒绝访问"。

**排查步骤：**

1. 确认用户 QQ 号是否在 `admins` 列表中
2. 确认群号是否在 `allowed_groups` 列表中
3. 检查 `allow_private` 是否为 `true`（如果是私聊场景）
4. 检查配置文件中 QQ 号是否使用字符串格式（推荐用字符串以避免数字精度丢失）

### Q6: 消息格式错误

**现象：** 机器人回复的内容在 QQ 中显示为乱码或无法解析的格式。

**原因及解决方案：**

| 原因 | 解决方案 |
|------|----------|
| CQ 码格式错误 | 检查图片、@等 CQ 码格式是否正确 |
| 编码问题 | 确保 NapCat 和 Agent Core 的字符编码均为 UTF-8 |
| 消息超长 | QQ 消息上限约 4500 字符，超长消息自动截断 |
| 特殊字符 | 消息中的 `[`、`]` 等特殊字符需要转义 |

## 第8章 安全建议

### 8.1 使用专用 QQ 小号

- **务必使用新注册的 QQ 小号**作为机器人账号
- 不要使用日常聊天的主号或常用号
- 小号不要加入过多群聊，降低风控概率
- 建议对小号开启设备锁和登录保护

### 8.2 配置严格的权限白名单

- 精确填写 `allowed_groups`，只包含有需求的群
- 管理员列表 `admins` 严格控制到最小范围
- 如无需私聊功能，将 `allow_private` 设为 `false`
- 定期审查权限配置，清理不再使用的群组和管理员

**最小权限原则示例：**

```json
{
  "qq_bot": {
    "authorization": {
      "admins": ["100001"],
      "allowed_groups": ["200001"],
      "allow_private": false
    }
  }
}
```

### 8.3 启用操作确认机制

- 保持默认的敏感操作确认机制
- 涉及游戏内重大变更的操作必须经过二次确认
- 管理员操作建议同样保持确认机制，防止误操作

### 8.4 定期检查日志

- 定期查看 Agent Core 的 QQ 模块日志
- 关注异常登录尝试、频繁权限验证失败等安全事件
- 记录并分析所有命令执行历史

**关键日志事件：**

| 日志事件 | 说明 | 建议操作 |
|----------|------|----------|
| `WebSocket 连接成功` | 正常连接 | 无需操作 |
| `权限验证失败: QQ=xxxxx` | 未授权用户尝试访问 | 确认是否为正常用户遗漏配置 |
| `WebSocket 连接断开` | 网络异常或 NapCat 重启 | 检查 NapCat 状态 |
| `消息频率限制生效` | 用户消息超频 | 检查是否存在刷屏行为 |

### 8.5 不在公网暴露 NapCat 端口

- NapCat 的 WebSocket 端口（3001）和管理端口（6099）**不要暴露到公网**
- 如果 Agent Core 与 NapCat 不在同一台机器，使用 VPN 或 SSH 隧道加密通信
- 不要直接通过公网 IP 访问 NapCat WebUI
- 如果在云服务器部署，使用安全组限制仅允许 Agent Core 的 IP 访问

**安全拓扑建议：**

```
推荐架构（VPC 内网通信）
┌──────────┐    内网    ┌──────────┐
│ NapCat   │ ◄───────── │ Agent    │
│ (内网IP)  │  WS:3001   │ Core     │
├──────────┤            ├──────────┤
│ WebUI    │  仅限本地    │ 桌面应用  │
│ :6099    │  localhost  │          │
└──────────┘            └──────────┘

避免的架构（公网暴露）
✗ NapCat 端口映射到公网
✗ WebUI 公网可访问
✗ 无 access_token 鉴权
```

**SSH 隧道示例（跨机器场景）：**

```bash
# 在 Agent Core 机器上建立 SSH 隧道
ssh -L 3001:127.0.0.1:3001 user@napcat-server -N
# 之后 Agent Core 通过 127.0.0.1:3001 连接 NapCat
```
