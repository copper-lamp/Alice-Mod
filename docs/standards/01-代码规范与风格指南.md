## 第1章 概述

### 1.1 设计目标

本规范旨在为 McAgent 项目确立统一的代码风格和质量标准，确保全体贡献者产出的代码具有以下特性：

| 目标 | 说明 |
|------|------|
| **一致性** | 所有模块和子项目的代码风格统一，降低认知负担 |
| **可读性** | 代码自解释，减少非必要的注释依赖，提升协作效率 |
| **可维护性** | 约束代码复杂度，通过工具链自动检查，防止风格退化 |

### 1.2 适用范围

本规范覆盖 McAgent 项目中使用到的所有编程语言和标记格式：

| 语言/格式 | 适用模块 | 备注 |
|-----------|----------|------|
| TypeScript | Agent Core、Adapter Core BE（LeviLamina 插件） | 主力开发语言 |
| Java | Adapter Core JE（Fabric 模组） | Java 版接入核心 |
| SQL | 记忆系统（SQLite 数据库操作） | 内联 SQL 及迁移脚本 |
| JSON / YAML | 配置文件、工具 Schema、数据集 | 配置与数据交换格式 |

### 1.3 工具链

McAgent 项目通过自动化工具确保规范落地：

| 语言 | 工具 | 用途 |
|------|------|------|
| TypeScript | **ESLint** + **Prettier** | 静态检查 + 自动格式化 |
| Java | **Spotless** + **Checkstyle** | 代码格式检查 + 风格校验 |
| SQL | 人工 Review（暂无自动化工具） | 以本规范为准 |
| JSON/YAML | Prettier | 自动格式化 |
| Git | husky + lint-staged | 提交前自动 lint |

工具配置文件统一放置于项目根目录：

```
mcagent/
├── .eslintrc.json          # ESLint 配置
├── .prettierrc.json        # Prettier 配置（TS/JSON/YAML）
├── buildSrc/               # Spotless + Checkstyle 配置（Java 子项目）
└── .husky/
    └── pre-commit          # 提交前 lint hook
```

## 第2章 通用规范

本章规定所有语言和文件类型均适用的基础规则。

### 2.1 文件编码

所有源文件必须使用 **UTF-8 without BOM** 编码。

### 2.2 行尾（End of Line）

使用 **Unix LF（Line Feed，`\n`）** 作为换行符。禁止使用 CRLF（`\r\n`）或 CR（`\r`）。

Git 配置参考：

```bash
# 在项目根目录 .gitattributes 中设置
* text=auto eol=lf
```

### 2.3 缩进

| 语言 | 缩进规则 |
|------|----------|
| TypeScript | 2 空格 |
| Java | 2 空格 |
| JSON / YAML | 2 空格 |
| SQL | 大写关键字，缩进 2 空格 |

禁止使用 Tab 缩进。编辑器应配置为 Tab → 空格。

### 2.4 最大行宽

| 语言 | 最大行宽 |
|------|----------|
| TypeScript | 100 字符 |
| Java | 100 字符 |
| SQL | 120 字符 |
| JSON / YAML | 无限制（推荐自动换行） |

当一行超过最大行宽时，应在运算符或逗号处换行，保持可读性。

### 2.5 文件末尾空行

所有文件末尾必须保留一个空行。Git 差异提示 `No newline at end of file` 时需修复。

### 2.6 尾随空格

禁止尾随空格（trailing whitespace）。编辑器应启用"自动删除尾随空格"功能。

## 第3章 TypeScript 规范（Agent Core & Adapter Core BE）

本章适用于 Agent Core（Electron + TypeScript/Node.js 桌面应用）和 Adapter Core BE（LeviLamina Node.js 插件）的所有 TypeScript 代码。

### 3.1 命名规范

| 标识符类型 | 命名风格 | 示例 |
|-----------|---------|------|
| 类（Class） | PascalCase | `LLMProvider`、`ToolRegistry`、`WorkspaceManager` |
| 接口（Interface） | PascalCase（不加 I 前缀） | `LLMProvider`（✅）、`ILLMProvider`（❌） |
| 类型别名（Type） | PascalCase | `ToolMetadata`、`ChatMessage` |
| 枚举（Enum） | PascalCase | `MemoryType`、`ToolCategory` |
| 枚举值（Enum Member） | UPPER_SNAKE_CASE | `WORKSPACE_CREATED`、`CONNECTION_LOST` |
| 常量（const） | UPPER_SNAKE_CASE | `DEFAULT_TIMEOUT_MS`、`MAX_RETRY_COUNT` |
| 变量（let/var） | camelCase | `workspaceId`、`toolList`、`isConnected` |
| 函数 / 方法 | camelCase | `registerTool()`、`sendMessage()`、`getState()` |
| 参数 | camelCase | `request: ToolCallRequest` |
| 私有属性 | camelCase（加 `#`） | `#client`、`#state`、`#handleMessage()` |
| 文件/目录名 | kebab-case | `llm-provider.ts`、`tool-registry/` |

> 接口名不加 `I` 前缀是 McAgent 的强制约束。例如使用 `LLMProvider` 而非 `ILLMProvider`，使用 `ToolConfig` 而非 `IToolConfig`。

### 3.2 文件组织结构

### 3.3 导入规范

#### 3.3.1 import type

类型导入必须使用 `import type` 语法，避免运行时依赖：

```typescript
// ✅ 正确
import type { LLMProviderConfig, ChatMessage } from './provider.types';
import { ToolRegistry } from './registry';

// ❌ 错误
import { LLMProviderConfig, ToolRegistry } from './registry';
```

1. Node.js 内置模块
2. 第三方依赖

### 3.4 语言特性约束

#### 3.4.1 装饰器（Decorator）

McAgent 不使用 Stage 3 以下的 ECMAScript 提案特性。装饰器（目前 Stage 3）暂不使用。工具注册通过函数式 API 完成，而非装饰器：

```typescript
// ✅ 正确：函数式注册
export const moveToTool: Tool = {
  name: 'move_to',
  description: '移动到指定坐标',
  handler: async (params: MoveToParams) => { ... }
};

// ❌ 错误：装饰器（不使用）
@Tool()
class MoveToTool { ... }
```

#### 3.4.2 async / await

始终优先使用 `async/await` 而非裸 `Promise.then()` 链式调用：

```typescript
// ✅ 正确
export async function getWorkspaceState(id: string): Promise<WorkspaceState> {
  const raw = await database.query(`SELECT * FROM workspaces WHERE id = ?`, [id]);
  return parseState(raw);
}

// ❌ 错误
export function getWorkspaceState(id: string): Promise<WorkspaceState> {
  return database.query(`SELECT * FROM workspaces WHERE id = ?`, [id])
    .then(raw => parseState(raw));
}
```

#### 3.4.3 类型声明

禁止使用 `object` 类型。使用 `Record<string, any>` 或具体接口：

```typescript
// ✅ 正确
function processConfig(config: Record<string, any>): void;

// ❌ 错误
function processConfig(config: object): void;
```

#### 3.4.4 枚举

枚举优先使用 `const enum` 或联合类型（union type），减少运行时开销：

```typescript
// ✅ 正确：const enum
export const enum MemoryType {
  PLAYER_HABIT = 'player_habit',
  MAP_POINT = 'map_point',
  TASK_EXPERIENCE = 'task_experience',
  SOCIAL_RELATION = 'social_relation',
  SKILL = 'skill',
}

// ✅ 正确：联合类型（适合简单场景）
export type ConnectionState = 'disconnected' | 'connecting' | 'connected' | 'error';

// ❌ 错误：普通 enum（生成额外运行时代码）
export enum MemoryType {
  PLAYER_HABIT,
  MAP_POINT,
}
```

#### 3.4.5 工具文件元数据导出

导出一个 class 的工具文件必须同时导出一个 `ToolMetadata` 实例：

```typescript
// tools/movement/move_to.ts
export class MoveToTool {
  async execute(params: MoveToParams): Promise<MoveToResult> {
    // ...
  }
}

// ✅ 必须导出 ToolMetadata 实例
export const toolMetadata: ToolMetadata = {
  name: 'move_to',
  description: '移动智能体到指定坐标位置',
  category: 'movement',
  parameters: {
    x: { type: 'number', description: '目标 X 坐标' },
    y: { type: 'number', description: '目标 Y 坐标' },
    z: { type: 'number', description: '目标 Z 坐标' },
  },
};
```

#### 3.4.6 函数参数解构

函数参数使用解构赋值（destructuring）提升可读性：

```typescript
// ✅ 正确
export async function moveTo({
  x,
  y,
  z,
  speed = 1.0,
  timeout = 5000,
}: MoveToParams): Promise<MoveToResult> {
  // ...
}

// ❌ 错误
export async function moveTo(
  x: number,
  y: number,
  z: number,
  speed?: number,
  timeout?: number,

  // ...
}
```

### 3.5 错误处理

所有自定义错误使用 `McAgentError` 及其子类体系：

```typescript
// error.types.ts
export class McAgentError extends Error {
  constructor(
    message: string,
    public readonly code: string,
    public readonly context?: Record<string, any>,
  ) {
    super(message);
    this.name = 'McAgentError';
  }
}

export class ConnectionError extends McAgentError {
  constructor(message: string, context?: Record<string, any>) {
    super(message, 'CONNECTION_ERROR', context);
    this.name = 'ConnectionError';
  }
}

export class ToolExecutionError extends McAgentError {
  constructor(message: string, context?: Record<string, any>) {
    super(message, 'TOOL_EXECUTION_ERROR', context);
    this.name = 'ToolExecutionError';
  }
}
```

使用示例：

```typescript
import { ConnectionError, ToolExecutionError } from '../errors';

export async function executeTool(request: ToolCallRequest): Promise<ToolResult> {
  if (!this.#client.connected) {
    throw new ConnectionError('Adapter Core 未连接', { workspaceId: this.#workspaceId });
  }
  const result = await this.#client.call(request);
  if (result.error) {
    throw new ToolExecutionError(result.error.message, { toolName: request.tool });
  }
  return result;
}
```

### 3.6 日志规范

禁止提交 `console.log` / `console.error` 等裸控制台输出。必须使用统一的 `logger` 模块：

```typescript
import { logger } from '../logger';

// ✅ 正确
logger.info('工具调用开始', { toolName: 'move_to', params: { x, y, z } });
logger.warn('连接超时，正在重试...', { retryCount: 2 });
logger.error('工具执行失败', { toolName: 'move_to', error: err.message });

// ❌ 错误
console.log('工具调用开始');
console.error('工具执行失败', err);
```

`logger` 模块支持以下日志级别（按严重程度递增）：

```
debug → info → warn → error
```

### 3.7 注释规范

代码注释使用 **JSDoc** 格式，且**仅对 public API 进行注释**。内部实现不写注释（通过自解释的命名和结构表达意图）：

```typescript
/**
 * 注册一个工具到工具注册表中。
 *
 * @param tool - 工具定义，包含名称、描述、参数 schema 和执行函数
 * @throws {ToolRegistrationError} 当工具名称已存在或参数 schema 校验失败时抛出
 */
export function registerTool(tool: Tool): void {
  // 内部实现不需要注释——代码本身就说明意图
  if (registry.has(tool.name)) {
    throw new ToolRegistrationError(`工具 "${tool.name}" 已注册`);
  }
  registry.set(tool.name, validateAndFreeze(tool));
}
```

JSDoc 必须包含 `@param` 和 `@throws`（如果适用）。单行 public API 可使用行内 JSDoc：

```typescript
/** 获取当前已注册的工具列表（只读快照） */
export function getRegisteredTools(): ReadonlyArray<Tool> {
  return Object.freeze([...registry.values()]);
}
```

### 4.5 Lombok 限制

Lombok 的使用严格限制为以下两个注解：

| 注解 | 用途 | 说明 |
|------|------|------|
| `@Data` | 数据类（POJO/DTO/VO） | 自动生成 getter/setter/equals/hashCode/toString |
| `@Builder` | 构建器模式 | 配合 `@Data` 使用 |

```java
import lombok.Builder;
import lombok.Data;

@Data
@Builder
public class MoveToParams {
    private double x;
    private double y;
    private double z;
    @Builder.Default
    private double speed = 1.0;
    @Builder.Default
    private int timeout = 5000;
}
```

**禁止使用**以下 Lombok 特性：`@Setter`、`@Getter`、`@AllArgsConstructor`、`@NoArgsConstructor`、`@ToString`、`@EqualsAndHashCode`、`@Slf4j`、`@RequiredArgsConstructor`、`@Value`、`@With` 等。

### 4.6 Optional 约束

禁止将 `Optional` 用作字段类型或方法参数类型。`Optional` 仅用于方法返回值：

```java
// ✅ 正确：Optional 用于方法返回值
public Optional<Tool> findTool(String name) {
    return Optional.ofNullable(toolMap.get(name));
}

// ❌ 错误：Optional 用作字段
public class ToolConfig {
    private Optional<String> description; // ❌
}

// ❌ 错误：Optional 用作方法参数
public void setConfig(Optional<String> value) { // ❌
}
```

### 4.7 Stream API 约束

Stream API 限于单行操作，复杂逻辑应使用传统 for 循环：

```typescript
// ✅ 正确：单行 Stream 操作
List<String> toolNames = tools.stream()
    .map(Tool::getName)
    .collect(Collectors.toList());

// ✅ 正确：简单过滤
List<Tool> movementTools = tools.stream()
    .filter(t -> "movement".equals(t.getCategory()))
    .collect(Collectors.toList());

// ❌ 错误：复杂逻辑使用 Stream
List<ToolResult> results = tools.stream()
    .filter(t -> t.isEnabled())
    .map(t -> {
        // 多行复杂逻辑
        ToolParams params = buildParams(t);
        ToolResult result = t.execute(params);
        if (result.isSuccess()) {
            // ... 更多逻辑
        }
        return result;
    })
    .collect(Collectors.toList());

// ✅ 正确：复杂逻辑使用 for 循环
List<ToolResult> results = new ArrayList<>();
for (Tool tool : tools) {
    if (!tool.isEnabled()) continue;
    ToolParams params = buildParams(t);
    ToolResult result = tool.execute(params);
    if (result.isSuccess()) {
        // 复杂逻辑自由展开
    }
    results.add(result);
}
```

## 第5章 SQL 规范

本章适用于 McAgent 项目中所有 SQL 语句，包括应用代码中的内联 SQL 和数据库迁移脚本。

### 5.1 关键字大写

SQL 保留关键字必须大写：

```sql
-- ✅ 正确
SELECT id, name, state FROM workspace WHERE is_active = TRUE;

-- ❌ 错误
select id, name, state from workspace where is_active = true;
```

常用关键字清单：`SELECT`、`FROM`、`WHERE`、`INSERT`、`INTO`、`VALUES`、`UPDATE`、`SET`、`DELETE`、`CREATE`、`TABLE`、`ALTER`、`ADD`、`COLUMN`、`DROP`、`INDEX`、`PRIMARY KEY`、`FOREIGN KEY`、`REFERENCES`、`ON`、`JOIN`、`LEFT`、`RIGHT`、`INNER`、`OUTER`、`ON`、`AS`、`AND`、`OR`、`NOT`、`IN`、`EXISTS`、`BETWEEN`、`LIKE`、`ORDER BY`、`GROUP BY`、`HAVING`、`LIMIT`、`OFFSET`、`DISTINCT`、`CASE`、`WHEN`、`THEN`、`ELSE`、`END`、`NULL`、`IS`、`TRUE`、`FALSE`。

### 5.2 命名规则

| 对象 | 规则 | 示例 |
|------|------|------|
| 表名 | **单数** + snake_case | `task_meta`（✅）、`tasks`（❌） |
| 字段名 | 小写 snake_case | `created_at`、`workspace_id` |
| 主键 | `id TEXT PRIMARY KEY` | `id TEXT PRIMARY KEY` |
| 外键 | 关联表名 + `_id` | `workspace_id` |
| 索引 | `idx_` + 表名 + `_` + 字段 | `idx_task_meta_status` |

> 表名使用单数形式的理由：表结构对应一个实体的定义，`task_meta` 读作"任务元数据表"而非"多个任务表"，语义更清晰。

### 5.3 建表示例

```sql
CREATE TABLE workspace (
    id            TEXT    PRIMARY KEY,
    name          TEXT    NOT NULL,
    host          TEXT    NOT NULL,
    port          INTEGER NOT NULL DEFAULT 25565,
    status        TEXT    NOT NULL DEFAULT 'disconnected',
    config_json   TEXT,
    created_at    TEXT    NOT NULL DEFAULT (datetime('now')),
    updated_at    TEXT    NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE task_meta (
    id            TEXT    PRIMARY KEY,
    workspace_id  TEXT    NOT NULL REFERENCES workspace(id) ON DELETE CASCADE,
    title         TEXT    NOT NULL,
    description   TEXT,
    priority      INTEGER NOT NULL DEFAULT 0,
    status        TEXT    NOT NULL DEFAULT 'pending',
    parent_id     TEXT    REFERENCES task_meta(id) ON DELETE SET NULL,
    result_json   TEXT,
    created_at    TEXT    NOT NULL DEFAULT (datetime('now')),
    updated_at    TEXT    NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_task_meta_status ON task_meta(status);
CREATE INDEX idx_task_meta_workspace ON task_meta(workspace_id);
```

### 5.4 字段类型约定

| SQL 类型 | McAgent 用途 | 说明 |
|----------|-------------|------|
| `TEXT` | 字符串、ID、时间戳 | SQLite 没有专用时间类型，使用 TEXT 存储 ISO 8601 |
| `INTEGER` | 整数、布尔值（0/1） | 布尔值用 0/1 表示 |
| `REAL` | 浮点数 | 坐标、速度等 |
| `TEXT`（JSON） | 复杂结构 | 字段名后缀 `_json` |

### 5.5 时间戳字段

所有表必须包含 `created_at` 和 `updated_at` 字段（除非有明确理由排除）：

```sql
created_at    TEXT    NOT NULL DEFAULT (datetime('now')),
updated_at    TEXT    NOT NULL DEFAULT (datetime('now'))
```

### 5.6 JSON 字段

存储 JSON 数据的字段必须以 `_json` 结尾：

```sql
config_json   TEXT,    -- 工作区配置 JSON
result_json   TEXT,    -- 工具执行结果 JSON
metadata_json TEXT     -- 扩展元数据 JSON
```

### 5.7 外键约束

外键约束必须显式声明 `ON DELETE CASCADE` 或 `ON DELETE SET NULL`：

```sql
-- 子记录随父记录删除而级联删除
workspace_id  TEXT  NOT NULL REFERENCES workspace(id) ON DELETE CASCADE

-- 父记录删除后，子记录的外键置为 NULL
parent_id     TEXT  REFERENCES task_meta(id) ON DELETE SET NULL
```

### 7.2 提交信息格式

提交信息使用 **Conventional Commits** 规范（详细要求参见《Git 工作流规范》文档）：

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

**type（必填）**：

| 类型 | 含义 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 Bug |
| `refactor` | 重构（不新增功能也不修复 Bug） |
| `perf` | 性能优化 |
| `style` | 代码风格变更（不影响功能） |
| `docs` | 文档变更 |
| `test` | 测试变更 |
| `chore` | 构建/工具/依赖变更 |
| `ci` | CI 配置变更 |

**scope（可选）**：变更影响的范围，如 `agent-core`、`adapter-be`、`adapter-je`、`bridge`、`memory`、`docs`。

**description（必填）**：简要描述变更，使用中文或英文均可，首字母小写，末尾不加句号。

**示例**：

```
feat(agent-core): 添加 LLM Provider 热切换功能

fix(adapter-be): 修复假人寻路超时未重置状态的问题

refactor(bridge): 重构 JSON-RPC 批量调用处理逻辑

docs: 更新 TCP 通信协议规范中的心跳机制说明

chore: 升级 ESLint 至 v9 并更新相关规则配置
```
