## 第1章 概述

### 1.1 测试目标

McAgent 项目的测试体系围绕以下三个核心目标构建：

| 目标 | 说明 |
|------|------|
| **保证质量** | 验证各模块（LLM Provider、TaskManager、MemoryManager、工具系统等）的功能正确性，确保符合接口规范 |
| **防止回归** | 在迭代开发过程中，确保已有功能不被新代码破坏，特别是跨模块联调场景（Agent Core ↔ Adapter Core） |
| **文档行为** | 测试用例作为可执行的文档，清晰描述模块的预期行为和边界条件 |

### 1.2 测试金字塔

McAgent 采用经典测试金字塔策略，按层次分配测试投入比例：

```
                    ╱╲
                   ╱  ╲
                  ╱ E2E╲            ← 端到端测试（占比 ~10%）
                 ╱ 测试  ╲
                ╱──────────╲
               ╱            ╲
              ╱  集成测试    ╲      ← 集成测试（占比 ~20%）
             ╱              ╲
            ╱────────────────╲
           ╱                  ╲
          ╱    单元测试        ╲    ← 单元测试（占比 ~70%）
         ╱                    ╲
        ╱──────────────────────╲
```

- **单元测试（70%）**：测试独立模块/类的功能，隔离外部依赖
- **集成测试（20%）**：测试模块间协作，如 Provider + Model Router、TaskManager + TaskScheduler、MemoryManager + SQLite
- **E2E 测试（10%）**：测试完整链路，从 Agent Core 到 Adapter Core 的全流程

### 1.3 测试原则：FIRST

所有测试用例必须遵循 FIRST 原则：

| 原则 | 要求 | 反例 |
|------|------|------|
| **F**ast（快速） | 单元测试毫秒级完成，集成测试秒级完成 | 测试中发起真实 HTTP 请求 |
| **I**ndependent（独立） | 测试可独立运行，不依赖执行顺序 | 测试 A 依赖测试 B 设置的数据 |
| **R**epeatable（可重复） | 任意环境运行结果一致 | 测试依赖系统时间或随机端口 |
| **S**elf-validating（自验证） | 测试自动判断通过/失败 | 需要人工查看日志确认结果 |
| **T**imely（及时） | 测试与生产代码同时编写 | 功能完成后补写测试 |

## 第2章 TypeScript 测试规范（Agent Core + Adapter BE）

### 2.1 测试框架

#### 2.1.1 主框架：Vitest

Agent Core 和 Adapter Core BE 统一使用 **Vitest** 作为主要测试框架。

**配置文件**：`vitest.config.ts`

```typescript
// vitest.config.ts
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    // 全局配置
    globals: true,
    environment: 'node',
    root: 'src',

    // 包含/排除模式
    include: ['**/*.test.ts'],
    exclude: ['node_modules', 'dist'],

    // 超时设置
    testTimeout: 10_000,
    hookTimeout: 10_000,

    // 覆盖率
    coverage: {
      provider: 'v8',
      reporter: ['text', 'json', 'html', 'lcov'],
      reportsDirectory: '../coverage',
      include: ['src/**/*.ts'],
      exclude: [
        'src/**/*.test.ts',
        'src/**/*.d.ts',
        'src/types/**',
      ],
    },

    // 全局 setup 文件
    setupFiles: ['./test/setup.ts'],
  },
});
```

#### 2.1.2 辅助框架：Node 测试运行器

对于无需 Vitest 特性的轻量级场景，可使用 Node.js 内置测试运行器（Node >= 18）：

```typescript
// 仅用于极简脚本级验证
import { describe, it } from 'node:test';
```

**选择建议**：
- 绝大多数测试使用 **Vitest**（Mock、覆盖率、watch 模式支持更好）
- Node 测试运行器仅在无外部依赖的纯函数测试中使用

#### 2.2.2 测试结构

使用 `describe → it → expect` 三级结构：

```typescript
import { describe, it, expect } from 'vitest';
import { OpenAIProvider } from './OpenAIProvider';

describe('OpenAIProvider', () => {
  describe('generate', () => {
    it('should return response text when API call succeeds', async () => {
      // Arrange
      const provider = new OpenAIProvider({ apiKey: 'test-key' });

      // Act
      const result = await provider.generate('Hello');

      // Assert
      expect(result).toBeTypeOf('string');
      expect(result.length).toBeGreaterThan(0);
    });

    it('should throw ConfigError when apiKey is empty', () => {
      // Arrange
      const provider = new OpenAIProvider({ apiKey: '' });

      // Act & Assert
      expect(() => provider.generate('Hello')).rejects.toThrow('API key is required');
    });
  });
});
```

#### 2.2.3 Mock 策略

使用 `vi.mock()` 模拟外部依赖，禁止手动编写 mock 文件：

```typescript
// Mock 文件系统操作
vi.mock('fs/promises', () => ({
  readFile: vi.fn().mockResolvedValue(JSON.stringify({ key: 'value' })),
  writeFile: vi.fn().mockResolvedValue(undefined),

```

```typescript
import { OpenAIProvider } from './OpenAIProvider';

// Mock OpenAI SDK
vi.mock('openai', () => {
  const mockCreate = vi.fn();
  return {
    default: vi.fn().mockImplementation(() => ({
      chat: { completions: { create: mockCreate } },
    })),
    mockCreate, // 导出引用以便测试中访问
  };
});

describe('OpenAIProvider', () => {
  let provider: OpenAIProvider;

  beforeEach(() => {
    vi.clearAllMocks();
    provider = new OpenAIProvider({
      apiKey: 'sk-test',
      model: 'gpt-4o',
    });
  });

  describe('generate', () => {
    it('should return streamed response correctly', async () => {
      const result = await provider.generate('Hello');

      expect(result).toContain('mocked');
    });

    it('should handle API timeout gracefully', async () => {
      // 模拟超时
      const { mockCreate } = await import('openai');
      mockCreate.mockRejectedValueOnce(new Error('timeout'));

      await expect(provider.generate('Hello')).rejects.toThrow('Request timeout');
    });
  });
});
```

```typescript
// src/task/__integration__/TaskManagerScheduler.integration.test.ts
import { describe, it, expect, beforeEach } from 'vitest';
import { TaskManager } from '../TaskManager';
import { TaskScheduler } from '../TaskScheduler';

describe('TaskManager + TaskScheduler Integration', () => {
  let taskManager: TaskManager;
  let scheduler: TaskScheduler;

  beforeEach(() => {
    scheduler = new TaskScheduler();
    taskManager = new TaskManager(scheduler);
  });

  it('should create and schedule a task in sequence', async () => {
    const task = await taskManager.createTask('mine diamond', 'high');
    await scheduler.schedule(task.id);

    expect(status).toBe('scheduled');
  });

  it('should cancel a scheduled task and update its status', async () => {
    const task = await taskManager.createTask('build house', 'medium');
    await scheduler.schedule(task.id);
    await scheduler.cancel(task.id);

    const status = await taskManager.getTaskStatus(task.id);
    expect(status).toBe('cancelled');
  });
});
```

```typescript
describe('MemoryManager + SQLite Integration', () => {
```

```typescript
// e2e/toolChain.e2e.test.ts
import { describe, it, expect, beforeAll, afterAll } from 'vitest';
import { TestContainer } from 'testcontainers';
import { MockAdapterClient } from './mock/MockAdapterClient';

describe('E2E: Tool Call Chain', () => {
  let container;
  let client: MockAdapterClient;

  beforeAll(async () => {
    container = await new TestContainer('agent-core:test')
      .withExposedPorts(12345)
      .start();

    client = new MockAdapterClient({
      host: 'localhost',
      port: container.getMappedPort(12345),
    });
    await client.connect();
  }, 60_000);

  afterAll(async () => {
    await client.disconnect();
    await container.stop();
  });

  it('should execute full tool call: look_at -> move_to -> mine_block', async () => {
    // Agent Core 发送 look_at 指令
    const lookResult = await client.sendToolCall('look_at', { x: 10, y: 64, z: 20 });
    expect(lookResult.success).toBe(true);

    // Agent Core 发送 move_to 指令
    const moveResult = await client.sendToolCall('move_to', { x: 10, z: 20 });
    expect(moveResult.success).toBe(true);

    // Agent Core 发送 mine_block 指令
    const mineResult = await client.sendToolCall('mine_block', {
      position: { x: 10, y: 64, z: 20 },
    });
    expect(mineResult.success).toBe(true);
    expect(mineResult.result).toHaveProperty('drops');
  });

  it('should handle reconnection after TCP disconnect', async () => {
    await client.simulateDisconnect();
    const reconnected = await client.waitForReconnect(5000);
    expect(reconnected).toBe(true);

    const result = await client.sendToolCall('get_status', {});
    expect(result.success).toBe(true);
  });
});
```

```java
// src/test/java/com/mcagent/je/tool/MoveToolTest.java
package com.mcagent.je.tool;

import com.mcagent.je.tool.MoveTool;
import net.minecraft.server.network.ServerPlayerEntity;
import net.minecraft.server.world.ServerWorld;
import net.minecraft.util.math.Vec3d;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.extension.ExtendWith;
import org.mockito.Mock;
import org.mockito.junit.jupiter.MockitoExtension;

import static org.junit.jupiter.api.Assertions.*;
import static org.mockito.Mockito.*;

@ExtendWith(MockitoExtension.class)
class MoveToolTest {

    @Mock
    private ServerPlayerEntity player;

    @Mock
    private ServerWorld world;

    private MoveTool moveTool;

    @BeforeEach
    void setUp() {
        moveTool = new MoveTool();
    }

    @Test
    void should_move_player_to_target_position_when_path_is_clear() {
        // Arrange
        when(player.getWorld()).thenReturn(world);
        when(player.getPos()).thenReturn(new Vec3d(0, 64, 0));
        when(world.isSpaceEmpty(any())).thenReturn(true);

        // Act
        ToolResult result = moveTool.execute(player, new Vec3d(10, 64, 20));

        // Assert
        assertTrue(result.isSuccess());
        verify(player).teleport(10, 64, 20);
    }

    @Test
    void should_return_failure_when_target_is_obstructed() {
        // Arrange
        when(player.getWorld()).thenReturn(world);
        when(world.isSpaceEmpty(any())).thenReturn(false);

        // Act
        ToolResult result = moveTool.execute(player, new Vec3d(10, 64, 20));

        // Assert
        assertFalse(result.isSuccess());
        assertEquals("OBSTRUCTED", result.getErrorCode());
        verify(player, never()).teleport(anyDouble(), anyDouble(), anyDouble());
    }
}
```

### 3.3 集成测试

#### 3.3.1 Fabric GameTest 框架

对于需要真实 Minecraft 环境的测试，使用 Fabric 提供的 GameTest 框架：

```java
// src/test/java/com/mcagent/je/gametest/MoveToolGameTest.java
package com.mcagent.je.gametest;

import net.minecraft.block.Blocks;
import net.minecraft.test.GameTest;
import net.minecraft.test.TestContext;
import net.minecraft.util.math.BlockPos;

public class MoveToolGameTest {

    /**
     * 测试在平坦地面上移动到目标位置
     * 要求：玩家成功移动到目标位置，无碰撞
     */
    @GameTest(tickLimit = 100)
    public void testMoveToClearPosition(TestContext context) {
        // 设置测试环境——平坦草地
        context.setBlockState(0, 0, 0, Blocks.GRASS_BLOCK);

        // 在目标位置放置标记
        context.setBlockState(10, 0, 20, Blocks.LIGHT_BLUE_WOOL);

        // 执行移动并断言
        context.assertTrue(
            context.getPlayer().getBlockPos().equals(new BlockPos(10, 1, 20)),
            "Player should reach the target position"
        );
    }

    /**
     * 测试移动到被阻挡的位置
     * 期望：工具返回失败状态，玩家位置不变
     */
    @GameTest(tickLimit = 100)
    public void testMoveToObstructedPosition(TestContext context) {
        // 在目标位置放置障碍物
        context.setBlockState(10, 1, 20, Blocks.STONE);

        // 断言玩家未移动到障碍物位置
        context.assertFalse(
            context.getPlayer().getBlockPos().equals(new BlockPos(10, 1, 20)),
            "Player should not move into obstructed position"
        );
    }
}
```

#### 3.3.2 @GameTest 注解使用规范

| 属性 | 说明 | 示例 |
|------|------|------|
| `tickLimit` | 最大 tick 数，超时视为失败 | `@GameTest(tickLimit = 100)` |
| `batchId` | 同一批次测试共享环境 | `@GameTest(batchId = "mining_tests")` |
| `required` | 是否为必要条件 | `@GameTest(required = false)` |
| `templateName` | 自定义测试模板结构 | `@GameTest(templateName = "mcagent:test_flat")` |

## 第4章 测试覆盖率要求

### 4.1 覆盖率目标

| 模块 | 行覆盖率目标 | 说明 |
|------|-------------|------|
| **LLM Provider**（OpenAI, Claude, Gemini 等） | ≥ 80% | 核心 AI 交互逻辑，需充分覆盖 |
| **TaskManager** | ≥ 80% | 任务生命周期管理，状态机全覆盖 |
| **MemoryManager** | ≥ 80% | 增删改查及向量搜索 |
| **工具模块**（Move, Inventory, Block 等） | ≥ 60% | 工具逻辑相对简单，覆盖主流程 |
| **网络层**（TCP Client/Server） | ≥ 60% | 协议编解码和连接管理 |
| **UI 组件** | ≥ 50% | React 组件主要覆盖交互逻辑 |

### 4.2 集成测试覆盖率

- 集成测试不设具体的覆盖率数值目标
- **每个公开接口至少有一个集成测试用例**，验证模块间协作正确性
- 核心业务流程（任务创建→调度→执行→完成）必须有完整的集成测试覆盖

### 4.3 覆盖率检查

- PR 提交时，CI 自动运行覆盖率检查
- 覆盖率下降超过 5% 的 PR **禁止合并**
- 每季度进行覆盖率审计，识别低覆盖区域并制定改进计划

## 第6章 Mock 与 Stub 规范

### 6.1 TypeScript Mock 规范

| 场景 | 方式 | 禁止 |
|------|------|------|
| 外部 NPM 模块 | `vi.mock('module-name')` | 手动创建 `__mocks__/` 目录 |
| 内部模块 | `vi.mock('./local-module')` | 修改模块内部状态 |
| 时间相关 | `vi.useFakeTimers()` | `setTimeout` 真实等待 |
| 网络请求 | `vi.mock('node:http')` 或 mock SDK | 发起真实 HTTP 调用 |

```typescript
// ✅ 正确做法
vi.mock('openai', () => ({
  default: vi.fn().mockImplementation(() => ({
    chat: { completions: { create: vi.fn() } },
  })),
}));

// ❌ 禁止做法
// 手动创建 __mocks__/openai.ts 文件
// 或使用 vi.mock('openai', () => { ... }, { spy: true })
```

### 6.2 Java Mock 规范

| 场景 | 方式 | 禁止 |
|------|------|------|
| Minecraft 服务端 API | `@Mock ServerPlayerEntity` | PowerMock |
| 数据库操作 | `@Mock DatabaseManager` | 真实数据库连接 |
| 网络通信 | `@Mock TcpClient` | 真实 TCP 连接 |

```java
// ✅ 正确做法
@ExtendWith(MockitoExtension.class)
class BlockToolTest {
    @Mock
    private ServerPlayerEntity player;

    @Mock
    private ServerWorld world;

    @InjectMocks
    private BlockTool blockTool;
}

// ❌ 禁止做法
// @PrepareForTest(BlockTool.class)  ← PowerMock
// 或使用 new BlockTool() 并手动注入真实依赖
```

### 6.3 必须 Mock 的外部服务

以下外部服务在测试中**必须 Mock**，禁止发起真实调用：

| 服务 | Mock 方式 | 原因 |
|------|----------|------|
| OpenAI API / Claude API / Gemini API | `vi.mock()` + mock 响应 | 付费 API，且网络不稳定 |
| ChromaDB（向量数据库） | `vi.mock('chromadb')` | 需要独立服务进程，不适合测试 |
| Minecraft 服务端 | Mockito `@Mock` | 无法在单元测试中启动真实服务端 |
| 文件系统（部分场景） | `vi.mock('fs/promises')` | 避免测试产生脏数据 |

### 6.4 SQLite 测试规范

使用 `:memory:` 数据库进行测试，无需真实文件：

```typescript
// ✅ 正确
const db = new Database(':memory:');

// ❌ 禁止
const db = new Database('./test-data.db');  // 产生脏文件
```

### 6.5 Mock 清理

- **TypeScript**：在每个 `describe` 或 `beforeEach` 中使用 `vi.clearAllMocks()` 或 `vi.resetAllMocks()`
- **Java**：`@MockitoExtension` 自动在每测试方法执行后重置 mock
- 禁止在测试之间共享 mock 状态

```typescript
// TypeScript：每个测试文件必须包含
beforeEach(() => {
  vi.clearAllMocks();
});
```
