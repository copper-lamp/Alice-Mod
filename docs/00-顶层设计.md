> 版本：v2.0（合并重构）

## 第1章 项目概述

### 1.1 背景

Minecraft 是世界上最受欢迎的沙盒游戏之一，但现有的 AI 自动化方案存在明显不足：

- **传统脚本**（如红石机器、命令方块）只能执行预设逻辑，无法应对复杂、动态的任务
- **学术方案**（如 Voyager）基于 Java 版，有独立运行环境，难以集成到现有服务器生态
- **商业模组**偏重单一功能（如自动农场），缺乏通用决策能力

### 1.2 目标

构建一个通用的 Minecraft AI Agent 框架，让大语言模型（LLM）像真人玩家一样思考、规划和执行，在 Minecraft 世界中自主完成从挖矿建造到全物品备货的复杂任务。

### 1.3 核心数据

| 指标      | 数值                                          |
| ------- | ------------------------------------------- |
| AI 工具数量 | 108+（待简化）                                   |
| 支持游戏版本  | 基岩版（LeviLamina）+ Java 版（Fabic）              |
| 核心模块    | 3 个（Agent Core / TCP Bridge / Adapter Core） |
| 扩展能力    | 协议级解耦，支持多实例、多服协作                            |

### 1.4 设计原则

| 原则            | 说明                              |
| ------------- | ------------------------------- |
| **LLM 不处理细节** | 移动、跳跃、视角等由执行AI自动处理              |
| **上下文精简**     | 只注入基础状态（\~150 tokens），环境感知靠工具调用 |
| **工具粒度适中**    | 一个工具 = 一个"动作"，不是"脚本"            |
| **有反馈**       | 工具返回执行结果（成功/失败/详情/耗时/消耗）        |
| **可失败**       | 工具可能失败（寻路失败、物品不足等），LLM 需处理      |
| **有成本**       | 操作消耗时间、饥饿、耐久等资源                 |

### 1.5 关键约束

- LLM 不参与路径规划和视角控制细节，这些由执行AI 自动处理
- LLM 上下文仅注入基础状态，环境感知需通过工具主动获取
- 工具命名使用 `action_prefix` 格式（如 `move_to`、`organize_inventory`）
- 工具反馈包含成功/失败、最终状态、耗时、资源消耗

## 第3章 通信协议

### 3.1 协议概述

Agent Core 与 Adapter Core 之间通过 **TCP 长连接** 使用 **JSON-RPC 2.0** 协议通信，核心约束：

| 维度   | 规范                              |
| ---- | ------------------------------- |
| 传输层  | TCP 长连接                         |
| 应用层  | JSON-RPC 2.0                    |
| 序列化  | JSON（UTF-8）                     |
| 消息边界 | 换行符 `\n` 分隔                     |
| 批量调用 | JSON-RPC Batch 扩展               |
| 目标端口 | Agent Core 监听，Adapter Core 主动连接 |

### 3.2 消息格式

#### 标准请求

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tool_call",
  "params": {
    "tool_name": "move_to",
    "parameters": {
      "x": 100,
      "y": 64,
      "z": 200
    }
  }
}
```

#### 标准响应

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "success": true,
    "data": {
      "position": { "x": 100, "y": 64, "z": 200 },
      "duration_ms": 3450,
      "hunger_cost": 2
    }
  }
}
```

#### 错误响应

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32000,
    "message": "Tool execution failed",
    "data": {
      "reason": "path_not_found",
      "detail": "目标位置不可达：被岩浆湖阻挡",
      "position": { "x": 50, "y": 64, "z": 100 }
    }
  }
}
```

### 3.3 批量调用（Batch）

LLM 一次发出多个工具调用时使用 Batch 格式，Adapter Core 分析依赖关系后决定并行/顺序执行。

```json
[
  {"jsonrpc": "2.0", "id": 1, "method": "tool_call", "params": {"tool_name": "move_to", ...}},
  {"jsonrpc": "2.0", "id": 2, "method": "tool_call", "params": {"tool_name": "mine_block", ...}},
  {"jsonrpc": "2.0", "id": 3, "method": "tool_call", "params": {"tool_name": "place_block", ...}}
]
```

执行规则：

- **无依赖**：并行执行所有工具
- **有依赖**：按依赖关系顺序执行（依赖字段由工具本身语义决定）
- **统一返回**：所有工具执行完毕后统一返回结果数组

### 3.4 工具注册协议

Adapter Core 连接建立后，首先发送注册消息上报其实现的工具列表（JSON Schema 格式）：

```json
{
  "jsonrpc": "2.0",
  "method": "register_tools",
  "params": {
    "tools": [
      {
        "name": "move_to",
        "description": "走到指定坐标",
        "input_schema": {
          "type": "object",
          "properties": {
            "x": {"type": "number", "description": "目标 X 坐标"},
            "y": {"type": "number", "description": "目标 Y 坐标"},
            "z": {"type": "number", "description": "目标 Z 坐标"},
            "dimension": {"type": "string", "enum": ["overworld", "nether", "end"], "default": "overworld"}
          },
          "required": ["x", "y", "z"]
        }
      }
    ]
  }
}
```

Agent Core 收到后动态注册工具，无需硬编码。支持运行时热更新（重新调用注册消息即可）。

### 3.5 心跳与重连

| 机制     | 参数                        |
| ------ | ------------------------- |
| 心跳间隔   | 10 秒                      |
| 超时阈值   | 30 秒无响应视为断开               |
| 最大重连次数 | 5 次                       |
| 重连策略   | 指数退避（1s, 2s, 4s, 8s, 16s） |
| 状态恢复   | 重连成功后自动重注册工具列表            |

### 3.6 数据流向

1. **初始化**：Adapter Core 启动 → 连接 Agent Core → 注册工具列表（JSON Schema）+ 上报基础状态
2. **任务轮**：收到用户输入 → 组装提示词（含工具列表）→ 发送给 LLM → LLM 返回思考 + 工具调用（Batch）→ Agent Core 通过 TCP 分发 → Adapter Core 执行 → 返回结果 → LLM 继续决策
3. **事件上报**：Adapter Core 主动上报状态变化（血量低、被攻击、聊天消息等）→ Agent Core 决定是否中断当前任务

## 第4章 Agent Core

### 4.1 LLM 抽象层

LLM 抽象层提供统一的多模型接入接口，屏蔽不同 API 差异。

#### Provider 接口

```typescript
interface LLMProvider {
  // 发送消息并获取响应
  chat(messages: Message[], tools?: ToolDefinition[]): Promise<LLMResponse>;
  
  // 流式接口
  chatStream(messages: Message[], tools?: ToolDefinition[]): AsyncIterable<LLMChunk>;
}
```

#### 支持的 Provider

| Provider | 协议            | 备注                         |
| -------- | ------------- | -------------------------- |
| OpenAI   | OpenAI API 格式 | 兼容 Azure OpenAI、DeepSeek 等 |
| Claude   | Anthropic API | Messages API               |
| Gemini   | Google AI API | Gemini 系列模型                |
| Ollama   | Ollama API    | 本地部署模型                     |

#### Model Router

运行时根据配置切换模型，支持：

- 按工作区指定模型
- 按任务类型选择模型（复杂任务用更强模型，简单任务用轻量模型）

### 4.2 提示词系统

每轮对话前自动组装提示词，三部分组成：

```
1. 系统提示词
   - 角色设定（Agent 身份、行为规范）
   - 交互规则（输出格式、工具使用约束）

2. 基础状态注入（~150 tokens）
   【你的状态】
   生命: 18/20  饥饿: 15/20  饱和度: 3.2
   防御: 护甲 12  韧性 2
   位置: (128, 64, -256) 主世界  平原  光照 12
   装备: 主手=铁剑  头盔=铁头盔  胸甲=铁胸甲  ...
   背包: 23/36 (空位 13) - 苹果 x12, 圆石 x45, ...
   状态: 无特殊状况

3. 可用工具列表（动态组装，按工作区）
   - 从工作区管理器获取当前已注册的工具
   - 转换为 LLM 可识别的 Function Calling 格式
```

#### 上下文窗口管理

- 限制基础上下文长度（\~2048 tokens）
- 超出时自动截断历史消息
- 支持滑动窗口（保留最近 N 轮对话）

### 4.3 工作区管理器

| 能力   | 说明                   |
| ---- | -------------------- |
| 自动发现 | TCP 端口监听，新连接自动创建工作区  |
| 多实例  | 同时管理多个 Adapter Core  |
| 独立隔离 | 每个工作区拥有独立的工具列表、会话、记忆 |
| 动态切换 | UI 界面支持工作区切换         |
| 持久化  | 工作区状态可保存/恢复          |

### 4.4 Function Calling 解析

LLM 返回的 Function Calling 结果按以下步骤处理：

1. 解析 LLM 返回的 JSON，提取所有工具调用参数
2. 构造 JSON-RPC Batch 请求
3. 通过 TCP 发送给对应 Adapter Core
4. 收集执行结果
5. 将结果注入下一轮 LLM 上下文

### 4.5 UI 界面

Agent Core 提供 Electron 桌面界面，包含以下面板：

| 面板       | 功能                  |
| -------- | ------------------- |
| 主控制面板    | 连接状态、工作区切换、启动/停止    |
| LLM 对话界面 | 消息流历史、思考过程可视化       |
| 工具调用可视化  | 调用链、参数、结果实时显示       |
| 游戏状态面板   | 血量/饥饿/位置/背包摘要       |
| 配置管理     | LLM 模型、TCP 端口、QQ 设置 |
| 记忆浏览器    | 记忆类型、检索、查看、编辑       |

## 第6章 Adapter Core

### 6.1 架构

Adapter Core 是实现层，负责将 LLM 的工具调用转化为真实的游戏操作。核心组件为 **执行AI 引擎**。

```
┌─────────────────────────────────────────┐
│           Adapter Core                   │
│                                         │
│  ┌─────────────┐    ┌─────────────────┐  │
│  │ TCP Client  │    │  工具注册模块    │  │
│  │ (连接Agent  │    │  (JSON Schema)  │  │
│  │  Core)      │    └─────────────────┘  │
│  └──────┬──────┘                         │
│         │ 工具调用                        │
│         ▼                                │
│  ┌─────────────────────────────────────┐ │
│  │          执行AI 引擎                │ │
│  │                                     │ │
│  │  ┌──────────┐  ┌──────────────┐    │ │
│  │  │ 寻路系统  │  │  移动系统    │    │ │
│  │  │ (A* 3D)  │  │ (坐标/视角)  │    │ │
│  │  └──────────┘  └──────────────┘    │ │
```

### 6.3 版本实现

#### 基岩版（LeviLamina 插件）

- 技术栈：LeviLamina Node.js 插件
- 实现方式：通过 BDS API 操作游戏世界
- 假人管理：支持创建/销毁多个假人实例

#### Java 版（Fabric 模组）

- 技术栈：Fabric 模组
- 实现方式：通过 Minecraft Java API 操作游戏世界
- 工具接口与基岩版保持一致，实现细节适配 Java 版机制

### 6.4 状态上报

Adapter Core 周期性上报基础状态，并主动上报异常事件：

- 基础状态（\~150 tokens）：生命、饥饿、位置、装备、背包摘要
- 事件上报：血量过低、被攻击、聊天消息、环境变化

## 第7章 外部集成

### 7.1 QQ 机器人

通过 OneBot 协议对接 QQ 平台，实现游戏与社交平台的互通。

#### 消息桥接

```
QQ 群/私聊消息 ──→ 桥接 ──→ 游戏内聊天
游戏内聊天 ──→ 桥接 ──→ QQ 群/私聊
```

#### 权限控制

| 等级      | 说明   | 能力          |
| ------- | ---- | ----------- |
| NONE    | 无权限  | 只能接收消息      |
| BASIC   | 基础权限 | 发送聊天、查询状态   |
| COMMAND | 指令权限 | 执行预设指令      |
| ADMIN   | 管理员  | 全部权限（含配置修改） |

#### 工具列表（13个）

| 工具名                                   | 功能     |
| ------------------------------------- | ------ |
| `qq_send_group_msg`                   | 发送群消息  |
| `qq_send_private_msg`                 | 发送私聊   |
| `qq_send_image`                       | 发送图片   |
| `qq_send_file`                        | 发送文件   |
| `qq_get_messages`                     | 获取消息列表 |
| `qq_get_unread`                       | 获取未读消息 |
| `qq_mark_read`                        | 标记已读   |
| `qq_bridge_enable` / `bridge_disable` | 桥接开关   |
| `qq_bridge_status`                    | 查询桥接状态 |
| `qq_get_group_info`                   | 获取群信息  |
| `qq_get_group_members`                | 获取群成员  |
| `qq_get_user_info`                    | 获取用户信息 |

### 7.2 事件触发器

支持四类触发源，支持离线运行：

| 触发源       | 说明                              |
| --------- | ------------------------------- |
| **定时任务**  | LLM 可自主创建定时计划，到达时间自动触发执行        |
| **游戏聊天**  | 玩家在线发送对话消息时，自动捕获作为输入触发 Agent 应答 |
| **插件消息**  | 接收 AI 本体动态加载插件输出的消息通知           |
| **QQ 渠道** | QQ 机器人被 @用户后，自动捕获消息触发对话交互流程     |

## 第8章 安全与扩展

### 8.1 认证与安全

- **TCP 连接认证**：Adapter Core 连接 Agent Core 时需提供认证令牌
- **QQ 权限分级**：NONE / BASIC / COMMAND / ADMIN 四级控制
- **工具执行权限**：可按工作区配置可用工具子集
- **日志审计**：所有工具调用记录日志，支持回溯

### 8.2 扩展性

- **热插拔工具**：Adapter Core 可在运行时重新注册工具列表，Agent Core 无需重启
- **多版本支持**：基岩版 / Java 版通过相同协议接入
- **协议级扩展**：任何支持 JSON-RPC 2.0 的游戏模组均可接入
- **插件化记忆**：新增记忆类型无需修改核心代码

### 8.3 版本演进策略

| 阶段     | 内容                                         |
| ------ | ------------------------------------------ |
| Alpha  | Agent Core + TCP Bridge + 基岩版 Adapter Core |
| Beta   | Java 版 Adapter Core + QQ 机器人               |
| Stable | 记忆系统 + 任务规划 + 事件触发器                        |
| 扩展     | 社区插件接口 + 更多游戏适配                            |

## 附录C：架构决策记录

| 决策       | 选项                            | 选择                  | 理由                                 |
| -------- | ----------------------------- | ------------------- | ---------------------------------- |
| 通信协议     | REST / WebSocket / TCP        | **TCP + JSON-RPC**  | 长连接低延迟，支持双向推送，不依赖 HTTP 基础设施        |
| 消息格式     | Protobuf / MessagePack / JSON | **JSON**            | 人类可读，无需代码生成，调试成本低                  |
| 批量调用     | 自定义 / JSON-RPC Batch          | **JSON-RPC Batch**  | 标准协议原生支持，可保证调用顺序                   |
| 记忆存储     | PostgreSQL / MongoDB / SQLite | **SQLite + Chroma** | 单机部署无依赖，SQLite 结构化 + Chroma 向量检索互补 |
| Java 版方案 | Mineflayer / Fabric           | **Fabric 模组**       | 原生集成更高性能，Mineflayer 依赖控制台重定向       |
| 桌面应用     | Electron / Tauri              | **Electron**        | 生态成熟，Node.js 原生集成，团队熟悉 TypeScript  |
